1. 什么是闭包

闭包是指函数在创建时保留了对其定义作用域的引用，即使函数执行在其词法作用域之外，也能访问该作用域中的变量。

闭包在 JavaScript 中的常见表现形式是：函数嵌套函数，内部函数访问外部函数的变量。由于 JavaScript 的函数是“第一类公民”，可以作为值返回、传递或保存，因此在外部函数返回后，闭包依然保留对外部变量的访问权限。

2. 闭包导致的内存泄露场景

在 JavaScript 中，闭包有时会导致内存泄露，这是因为闭包在访问外部作用域的变量时会让这些变量无法被垃圾回收，从而导致不必要的内存占用。

常见的内存泄露场景

  • 未清理的事件监听：如果事件监听器引用了外部作用域中的变量，且在不需要时未移除，则会导致闭包一直存在，无法释放内存。
  • 定时器未清理：在定时器的回调函数中使用了闭包，但在不再需要时未清除定时器，导致回调函数及其引用的外部变量无法被回收。

3. 循环引用导致内存泄露

循环引用是指两个或多个对象相互引用，从而形成一个循环结构，导致垃圾回收器无法回收这些对象。

为什么循环引用会导致内存泄露？

JavaScript 的垃圾回收机制使用**标记清除（mark-and-sweep）**算法，垃圾回收器会从根对象（如全局对象）出发，查找所有可达对象。若对象形成了循环引用，且不再被根对象访问，则垃圾回zhe收器无法将其清除，这会导致这些对象长期保留在内存中，形成内存泄露。

4. 如何检测循环引用

  • 手动检测：在代码中通过逻辑分析或使用console.log输出检查对象的相互引用关系。
  • 使用开发者工具检测：现代浏览器的开发者工具提供了内存快照和堆分析，可以捕获内存快照来分析内存的使用情况，帮助发现循环引用和内存泄露。
  • 在 Chrome 开发者工具中，可以通过Memory（内存）面板，使用Heap Snapshot（堆快照）来查看对象的引用关系，并检查是否有意外的循环引用。
  • JSON.stringify 检测：尝试使用JSON.stringify序列化对象，如果对象中存在循环引用，JSON.stringify会抛出TypeError异常，可以用这种方式简单检测循环引用（注意这种方法只能用于检测较简单的循环引用，复杂场景需结合其他方法）。
  • WeakMap 弱引用：使用WeakMap结构管理对象引用。由于WeakMap的键是弱引用，不会影响对象的垃圾回收，可以通过WeakMap追踪对象引用关系，并避免循环引用导致的内存泄露。

5. 如何避免循环引用导致的内存泄露

  • 避免对象互相引用：在设计数据结构时，尽量避免互相引用，尤其是大的复杂对象。
  • 使用WeakMap或WeakSet：在 JavaScript 中，WeakMap和WeakSet是弱引用结构，存储在WeakMap或WeakSet中的对象不会被阻止垃圾回收。可以使用WeakMap和WeakSet来存储对象之间的引用关系，避免循环引用导致的内存泄露。
  • 在不需要时手动断开引用：当对象不再使用时，可以手动将引用设为null或undefined，确保垃圾回收器能够正常回收它们。

总结

  • 闭包是函数对其定义作用域的变量的引用，虽然便于保存状态，但可能导致内存泄露。
  • 循环引用是内存泄露的常见原因之一，互相引用的对象会阻止垃圾回收器清理。
  • 使用WeakMap和开发者工具等方法可以帮助检测和避免循环引用的内存泄露。

【困难】ES5 有几种实现继承的方式？怎么演变的？为什么会有这种演变？

在ES5中，有几种常见的实现继承的方式，主要是通过构造函数、原型链、和混合继承。这些方式都是通过不同的技巧来模拟类继承的特性。

  • 原型链继承：原型链继承是通过让子类的原型对象指向父类的实例，实现继承父类原型上的方法。但这种方式会导致子类共享父类原型，可能出现意外的修改。


  • 构造函数继承：构造函数继承是通过在子类构造函数中调用父类构造函数，从而继承父类的属性。这种方式的缺点是无法继承父类原型上的方法，只能继承属性。


  • 混合继承：混合继承是结合构造函数继承和原型链继承的方式，通过调用父类构造函数继承属性，同时将子类原型指向一个新的对象，继承父类原型的方法。


这种继承方式的演变主要是为了解决继承过程中的一些问题，比如属性共享、无法继承父类原型上的方法等。构造函数继承解决了属性继承的问题，原型链继承解决了方法继承的问题，而混合继承结合了两者的优点，解决了继承过程中的多个问题。

这种演变是为了更好地模拟类继承，使代码更易于理解和维护。然而，需要注意的是，ES5中的继承方式仍然有一些限制和问题，例如不支持多继承，而ES6引入了class语法来更直观地实现类继承和面向对象编程。因此，在ES6及以后的版本中，推荐使用class来实现继承。

【简单】箭头函数与普通函数（function）的区别是什么？构造函数（function）可以使用 new 生成实例，那么箭头函数可以吗？为什么？

箭头函数和普通函数（使用function关键字声明的函数）之间有几个重要的区别：

  • 语法：
  • 箭头函数使用箭头（=>）来定义函数，通常是(参数) => 表达式的形式。
  • 普通函数使用function关键字来定义函数，通常是function 函数名(参数)的形式。
  • this绑定：
  • 箭头函数具有词法作用域的this绑定，意味着它继承了父级作用域的this值。箭头函数内部的this与外部的this是相同的。
  • 普通函数的this值在运行时动态绑定，它依赖于调用方式（方法、构造函数等），因此在不同情况下this的值可能会不同。
  • arguments对象：
  • 箭头函数没有自己的arguments对象，它会继承外部函数的arguments。
  • 普通函数有自己的arguments对象，它包含了所有传递给函数的参数。
  • 构造函数：
  • 普通函数可以用new关键字来创建实例，因此可以用作构造函数。
  • 箭头函数不能用new来创建实例，因为它们没有自己的this，无法设置实例属性。尝试使用new创建箭头函数实例会引发错误。

【中等】说说你是如何理解JS中封装、继承这两个特点的？

在JavaScript中，封装和继承是面向对象编程（OOP）的两个重要概念，它们有助于组织和抽象代码，使代码更具可维护性和可扩展性。

  • 封装：封装是一种将数据（属性）和操作数据的方法（函数）组合在一个单元内的编程概念。这有助于隐藏数据的内部细节，只暴露必要的接口，从而提高代码的可维护性和降低了代码的复杂度。
  • 封装数据：通过将数据存储在对象的属性中，可以封装数据，然后提供公共方法来访问和修改这些数据。这有助于保护数据的完整性和隐藏内部实现。
  • 封装方法：封装方法是将一组相关的操作放在一个对象内。这有助于组织代码，减少全局作用域中的函数数量，并提供一致的接口。

  • 类继承：ES6引入了类继承，通过extends关键字，可以更容易地创建基于类的继承关系。

继承和封装是面向对象编程的核心概念，它们允许你创建可重用和抽象的代码单元，并建立对象之间的关系。理解这两个特点有助于你更好地组织和设计JavaScript代码。


【中等】谈谈 JS 垃圾回收机制的方式和内存管理

JavaScript的垃圾回收机制和内存管理是非常重要的，因为它们直接影响到JavaScript应用程序的性能和稳定性。以下是关于JavaScript垃圾回收机制和内存管理的一些重要概念：

垃圾回收机制的方式：

  • 标记-清除（Mark and Sweep）：这是JavaScript最常见的垃圾回收方式。它通过标记不再需要的对象，然后清除（或回收）这些对象来释放内存。垃圾回收器会遍历对象之间的引用关系，从根对象开始，标记所有可以访问到的对象，然后清除未标记的对象。
  • 引用计数（Reference Counting）：这是一种简单的垃圾回收方式，它通过跟踪每个对象的引用次数来判断对象是否可回收。当引用计数变为零时，对象就可以被回收。但这种方式存在很多问题，如循环引用问题，不再使用。

内存管理：

  • 变量作用域：合理控制变量的作用域，尽早释放不再需要的变量，以便垃圾回收器可以回收相关的内存。
  • 手动释放引用：当你不再需要一个对象时，手动将对象的引用设置为null，以明确告诉垃圾回收器该对象可以被回收。
  • 避免循环引用：确保不会出现循环引用，这会阻止垃圾回收器回收被引用的对象。在使用DOM元素、事件监听器或闭包时，特别要注意潜在的循环引用问题。
  • 使用对象池：对于频繁创建和销毁的对象，可以考虑使用对象池，以减少内存分配和回收的开销。
  • 避免内存泄漏：谨慎使用全局变量，定期检查你的代码，确保不会出现内存泄漏，特别是在长时间运行的应用中。
  • 性能优化：优化算法和数据结构，避免不必要的内存分配，减少对象的生命周期，以提高性能和内存利用率。

总之，JavaScript的垃圾回收机制和内存管理对于编写高性能和稳定的应用程序至关重要。了解这些概念以及如何使用工具来分析和解决内存问题是JavaScript开发人员的重要技能。


【中等】JS 中哪些操作可能会导致内存泄露

在JavaScript中，内存泄漏通常是由不正确的内存管理引起的。以下是一些常见的操作和情况，可能会导致内存泄漏：

  • 未释放事件监听器：当你添加事件监听器（如addEventListener）却忘记在不再需要时移除它们，这将导致内存泄漏。事件监听器会保留对DOM元素的引用，使DOM元素无法被垃圾回收。

解决方案：确保在不再需要事件监听器时，使用removeEventListener来移除它们。


  • 循环引用：当两个或多个对象相互引用，但没有外部引用时，它们不会被垃圾回收。

解决方案：确保不会出现循环引用，或者手动断开引用关系。


  • 全局变量：将数据存储在全局变量中，它们在整个应用程序生命周期内都存在，容易导致内存泄漏。

解决方案：使用适当的作用域管理变量，避免滥用全局变量。


  • 未释放定时器和间隔器：如果你使用setTimeout或setInterval创建定时器，但不清除它们，它们会持续运行并引用函数，导致内存泄漏。

解决方案：确保清除不再需要的定时器和间隔器，使用clearTimeout或clearInterval。


  • DOM元素未删除：如果你通过JavaScript创建了DOM元素，但没有从DOM中删除它们，它们将一直存在，占用内存。

解决方案：在不再需要DOM元素时，确保将它们从DOM中删除。


内存泄漏可能会导致应用程序性能下降，并最终导致崩溃。因此，定期检查你的代码并确保正确的内存管理是很重要的。使用开发者工具的内存分析工具和代码审查可以帮助你发现和解决潜在的内存泄漏问题。


【简单】说一下 JavaScript 的闭包，以及闭包可能会产生什么样的问题？如何避免这些问题？

JavaScript中的闭包是一个重要的概念，它发生在一个函数内部可以访问其外部函数的变量和作用域的情况。闭包的工作原理是，内部函数可以引用外部函数的变量，即使外部函数已经执行完毕，这些变量仍然可以被内部函数访问。

以下是对JavaScript闭包的解释以及可能产生的问题和如何避免这些问题的说明：

JavaScript闭包的基本概念：

  • 闭包的产生：闭包通常发生在内部函数（嵌套函数）引用外部函数的变量时。当内部函数被返回或传递到其他地方并在外部函数执行完成后仍然存在，闭包就形成了。
  • 作用域链：内部函数能够访问其包含的外部函数的变量，因为JavaScript引擎维护了一个作用域链，包含了所有外部作用域的变量。

闭包可能产生的问题：

  • 内存泄漏：闭包会导致外部函数中的变量不被垃圾回收，因此可能会导致内存泄漏。如果闭包持续存在并引用大量内存，这将对应用程序的性能和内存占用产生负面影响。
  • 变量覆盖：多个闭包引用相同的外部变量时，可能会导致变量覆盖问题，其中一个闭包的操作影响了其他闭包的行为。

如何避免闭包问题：

  • 及时释放资源：当不再需要闭包时，确保及时解除对外部变量的引用，从而允许垃圾回收器回收内存。
  • 使用模块模式：将相关的变量封装在一个函数内部，只暴露必要的接口，以减小闭包的作用范围。
  • 小心使用全局变量：避免在全局范围内创建闭包，因为它们会长时间存在。限制全局变量的使用，并使用适当的作用域。
  • 定期审查代码：定期检查代码以查找潜在的内存泄漏和变量覆盖问题。使用工具和浏览器的开发者工具来分析内存占用。

总之，JavaScript闭包是一种有用的编程技巧，但也需要小心使用，以避免潜在的内存泄漏和变量覆盖问题。了解闭包的工作原理以及如何正确使用和释放它们对于编写高质量的JavaScript代码至关重要。


【中等】介绍一下浏览器的事件循环机制（Event Loop）？

先说直观理解：浏览器其实就像一个“工厂流水线”，主线程在那儿不停地处理“任务”。

这些任务分两大类：

  • 宏任务（Macro Task）：像setTimeout、setInterval、DOM 事件回调、I/O。
  • 微任务（Micro Task）：像Promise.then、MutationObserver、queueMicrotask。

浏览器执行顺序是这样的：


  • 先执行主线程上的同步代码（这是最优先的）。
  • 然后把本轮产生的所有微任务执行完。
  • 执行完微任务后，浏览器会有机会去渲染一次页面（不一定每轮都渲染，要看是否需要）。
  • 接着再从宏任务队列里取一个任务出来执行。
  • 重复上面的过程。

举个例子：

console.log('1')

setTimeout(() => {
  console.log('2')
}, 0)

Promise.resolve().then(() => {
  console.log('3')
})

console.log('4')

执行顺序是：

  • console.log('1')→ 输出1
  • 注册setTimeout回调，丢进宏任务队列
  • Promise.then回调丢进微任务队列
  • console.log('4')→ 输出4
  • 主线程同步任务完毕，开始执行微任务 → 输出3
  • 微任务清空后，取出宏任务 → 输出2

最终结果：1, 4, 3, 2。

【中等】JS 中生成器和迭代器的区别

生成器和迭代器是 JavaScript 中用于处理集合数据的重要概念，它们有一些区别：


  • 迭代器（Iterator）：迭代器是一个对象，它提供了一个统一的方式来遍历集合中的元素，比如数组、集合或者自定义数据结构。迭代器通常包含一个next()方法，每次调用该方法都会返回一个包含value和done属性的对象，value表示当前迭代的值，done是一个布尔值，表示迭代是否已经结束。示例：


  • 生成器（Generator）：生成器是一种特殊类型的函数，它使用function*声明，可以通过yield关键字多次暂停和恢复函数的执行。每次调用生成器函数时，都会返回一个迭代器对象，可以通过迭代器的next()方法控制生成器函数的执行，并获取生成器函数yield语句返回的值。示例：


主要区别在于：

  • 语法：生成器使用function*声明，而迭代器是一个对象。
  • 控制执行：生成器函数内部通过yield关键字可以暂停执行并返回值，而迭代器对象通过next()方法控制函数的执行。
  • 多次执行：生成器函数可以在调用时生成多个迭代器，每个迭代器都会从生成器函数的当前状态开始执行，而迭代器对象是一个单独的迭代序列。

总的来说，生成器是一种函数，用于创建可暂停和恢复执行的迭代器。通过yield控制函数执行，而迭代器是一个对象，用于管理数据集合的迭代访问，通过next()方法控制迭代过程。

5. Promise 和 XMLHttpRequest 的差异

  • 语法简洁：Promise 的链式调用比嵌套回调更简洁，代码更易读。
  • 支持链式调用：Promise 可以通过.then()和.catch()实现异步任务的顺序控制，而 XMLHttpRequest 的嵌套回调较复杂。
  • 错误处理：Promise 提供.catch()来集中处理错误，XMLHttpRequest 需要在每个请求中手动处理错误。
  • Promise 与fetch更兼容：通常情况下，Promise 是与fetchAPI 配合使用，而不是直接与 XMLHttpRequest 结合。
  • XMLHttpRequest 缺少返回 Promise 的特性：XMLHttpRequest 本身没有返回 Promise，需要手动封装，无法天然实现 Promise 链式。
