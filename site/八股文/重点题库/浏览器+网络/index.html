【中等】什么是回流（Reflow）和重绘（Repaint）？它们的触发场景是什么？

这个问题基本属于浏览器渲染原理的必考点

先给大家一段代码，这段代码展示了 重绘 和 回流 的场景

<div id="box"></div>

<style>
  #box {
    width: 100px;
    height: 100px;
    background-color: red;
  }
</style>

<script>
  const box = document.getElementById('box');

  // 改宽度：会触发回流 + 重绘
  setTimeout(() => {
    box.style.width = "200px";
  }, 1000);

  // 改颜色：只触发重绘
  setTimeout(() => {
    box.style.backgroundColor = "blue";
  }, 2000);
</script>

先说回流（Reflow）：回流就是浏览器需要重新计算页面元素的几何结构和布局。比如位置、大小、盒子模型这些东西一旦变了，浏览器得重新排版。常见触发场景有：

  • 改变元素的大小（width、height、padding、margin、border）。
  • 改变元素的显示状态（比如display: none→block）。
  • DOM 结构的增删，或者位置变化。
  • 获取一些需要最新布局信息的属性，比如offsetHeight、getBoundingClientRect()，这些都会强制触发回流。

再说重绘（Repaint）：重绘是浏览器已经知道元素的位置和大小没变，但需要重新画一遍样式，比如颜色、背景、阴影。它不用重新计算布局，只是重新绘制像素。常见触发场景有：

  • 改变颜色（color、background-color）。
  • 改变可见性（visibility: hidden）。
  • 改变阴影（box-shadow）。

一个比较核心的面试区别点是：

  • 回流一定会引起重绘，但重绘不一定回流。
  • 回流开销更大，因为它会触发布局计算，性能影响比单纯的重绘要重。


【困难】浏览器的事件轮训机制是什么样子？

浏览器的事件轮询机制是什么？

浏览器的事件轮询机制是一个协调 JavaScript 执行、异步任务处理和 UI 渲染的机制，它保证了浏览器在单线程环境下有序执行任务。

JavaScript 是单线程的，事件轮询机制通过调用栈、任务队列和事件循环这三部分协同工作，管理同步任务与异步任务的执行顺序。

事件轮询的核心组成

  • 调用栈（Call Stack）
  • 存储当前执行的函数调用。
  • 同步任务（如普通函数调用）会直接进入调用栈执行，执行完后从栈中弹出。
  • Web APIs
  • 异步任务（如setTimeout、DOM 事件、网络请求）由浏览器的 Web APIs 处理，不阻塞主线程。
  • 任务队列（Task Queue）
  • 存储待执行的异步任务的回调。
  • 分为：
  • 宏任务（Macro Task）：setTimeout、setInterval、DOM 事件。
  • 微任务（Micro Task）：Promise.then、MutationObserver。
  • 事件循环（Event Loop）
  • 不断检查调用栈是否为空。
  • 如果调用栈为空，就从任务队列中取出任务放入调用栈执行。
  • 执行优先级：微任务优先于宏任务。

【困难】NodeJS 和 浏览器的轮询机制有什么区别？

1. 事件轮询的相似点

  • 单线程模型：无论是浏览器还是 Node.js，JavaScript 都是单线程执行，通过事件轮询机制协调同步任务和异步任务。
  • 异步任务的处理方式：两者都将异步任务分为宏任务（Macro Task）和微任务（Micro Task），并按照优先级调度执行：
  • 微任务优先执行。
  • 宏任务按队列顺序执行。

2. Node.js 和浏览器事件轮询机制的区别

2.1 设计目的不同

  • 浏览器：事件轮询机制主要用于处理用户交互、DOM 操作和页面渲染，因此关注点是高效响应用户事件和保持流畅的 UI 渲染。
  • Node.js：Node.js 的事件轮询机制基于libuv实现，设计目的是支持高性能的 I/O 操作，适合后端的高并发场景。

2.2 执行上下文

  • 浏览器：包含以下常见任务来源：
  • 宏任务：setTimeout、setInterval、setImmediate、DOM 事件。
  • 微任务：Promise.then、MutationObserver。
  • 渲染相关任务：在每帧（约 16ms）触发一次。
  • Node.js：Node.js 的事件轮询包含六个阶段（详见 2.4）。这些阶段主要用于调度 I/O 任务、定时器和其他系统级任务。

2.3 微任务的执行时机

  • 浏览器：微任务会在每个宏任务结束后立即执行。
  • Node.js：微任务会在当前阶段执行完所有回调后立即执行，无论是哪个阶段。

【中等】什么场景下会有跨域？怎么解决？

跨域是前端绕不过去的一个问题，特别是在校招的面试中，也是比较常见的。

很多同学的回答会停留在“请求不同源就跨域了，用 CORS 解决”这种很表面的层面，这对于大厂面试肯定的不够的

那么什么场景下会出现跨域呢？

只要不符合同源协议（同协议、同域名、同端口），那么就会跨域。

举几个常见的实际场景：

1. 开发环境前后端分离（最常见）

本地前端起了一个http://localhost:3000，后端接口是http://localhost:8080，虽然都是localhost，但端口不一样，也算跨域。

2. 正式环境多子域名

比如页面在https://www.example.com，接口却是https://api.example.com，虽然主域相同，但子域不同，也触发跨域。

3. 静态资源 CDN 加载

页面在https://app.my.com，但图片、JS、CSS 等从https://cdn.my.com加载，如果没有配置 CORS，也可能遇到限制。

4. iframe 嵌套外部页面

主页面是a.com，iframe 嵌套的是b.com，这时候访问iframe里面的内容，就会遇到跨域限制。

5. 跨域写 cookie（很容易忽略）

很多项目想让多个子域共享登录态（写 cookie），但不设置document.domain或cookie的 domain 字段，也会受限。

那么怎么解决跨域问题？呢？

不同场景下的解决方式也不太一样，下面是主流几种方式：

1. CORS（最常用）

CORS（跨域资源共享）是服务器设置 HTTP 响应头，告诉浏览器“这不是非法访问，放心用”。

关键响应头有几个：

  • Access-Control-Allow-Origin：允许的来源，比如设置为*或具体的http://localhost:3000
  • Access-Control-Allow-Credentials：是否允许携带 cookie
  • Access-Control-Allow-Methods：允许的请求方法
  • Access-Control-Allow-Headers：允许携带的请求头字段

注意：前端设置withCredentials = true时，服务端必须明确写具体域名，不能是*，并且开启Access-Control-Allow-Credentials。

2. 反向代理（前端开发常用）

比如本地用 Vite、Webpack 起服务时，可以设置 proxy，把/api的请求代理到http://localhost:8080，浏览器以为是同源请求，其实后面被代理走了。

这种方式适合本地开发或某些中转服务器，比如用 Nginx 做一层反向代理。


3. JSONP（只能 GET，已过时了）

早期常用的解决方案，通过<script>标签绕过同源限制（因为<script>标签不受同源限制），通过回调函数传值，但只能支持 GET 请求，且存在安全隐患。

现在基本被淘汰了。

4. postMessage（跨域 iframe 通信）

当你必须用iframe嵌套外部页面，又想传值时，可以用window.postMessage实现跨域通信。

5. 设置 document.domain（只适用于同主域的子域通信）

比如a.example.com和b.example.com，都设置document.domain = 'example.com'，可以实现部分跨域通信，但这种方式要求主域一致，且页面可控。

【中等】用户打开 URL 到浏览器完全渲染的全过程

用户打开一个 URL 并在浏览器中完全渲染页面的全过程可以分为多个阶段，从用户输入 URL 开始，到页面完全显示出来结束。

整个流程可以简单分为以下几个步骤：

工作流程

  • 用户请求：用户向某个网站发出请求（如打开一个网页或视频）。
  • CDN 调度：CDN 根据用户的 IP 地址判断最近的 CDN 节点，并将请求定向到该节点。
  • 缓存检查：CDN 节点检查是否有该内容的缓存。
  • 有缓存：直接返回缓存内容给用户。
  • 无缓存：向原始服务器请求内容并缓存，然后返回给用户。
  • 返回内容：用户接收到响应内容。

完整流程

完整流程共分为 8 个核心步骤：

  • 用户输入 URL
  • DNS 解析 → 获取 IP 地址
  • 建立 TCP 连接（三次握手）
  • 发送 HTTP 请求
  • 服务器处理请求并返回响应
  • 浏览器接收响应 → 构建 DOM 和 CSSOM
  • 构建渲染树 → 布局 → 绘制
  • 执行 JS → 完成页面渲染


以下是详细解释：

1.用户输入 URL

  • 当用户在浏览器地址栏中输入 URL 并按下回车，浏览器会开始处理这个请求。

2.DNS 解析

  • 浏览器首先通过 DNS（域名系统）解析 URL 中的域名，将其转换为 IP 地址。这个过程可能涉及以下步骤：
  • 浏览器缓存：浏览器先查找本地缓存中是否已经有该域名的 IP 地址。
  • 操作系统缓存：如果浏览器没有缓存，操作系统会查找其缓存。
  • 路由器缓存：如果操作系统没有缓存，查询会继续通过路由器。
  • ISP 的 DNS 服务器：最终，查询可能需要通过互联网服务提供商（ISP）的 DNS 服务器获取 IP 地址。

3.建立 TCP 连接

  • 通过 IP 地址确定了目标服务器后，浏览器与服务器通过TCP 三次握手建立连接。三次握手过程如下：
  • 客户端发送 SYN（同步）请求。
  • 服务器收到请求后，返回 SYN+ACK。
  • 客户端收到 ACK 后，发送 ACK 确认连接。

4.发送 HTTP 请求

  • 一旦 TCP 连接建立，浏览器向服务器发送HTTP 请求，包括请求头和相关参数。请求头中包含以下信息：
  • 请求的资源类型（HTML、CSS、JS 等）
  • 浏览器信息（User-Agent）
  • 可接受的内容类型（Accept）
  • 缓存信息（Cache-Control）等。

5.服务器处理请求

  • 服务器接收到请求后，处理请求并返回相应的资源（HTML 文件、图片、CSS、JavaScript 等）。这一步包括：
  • 服务器可能需要从数据库或文件系统中获取数据。
  • 如果是动态网页，服务器需要运行相关代码生成 HTML 内容（例如 PHP、Node.js 等服务器端语言）。

6.服务器返回 HTTP 响应

  • 服务器返回HTTP 响应，包括状态码（如 200 OK）、响应头（例如内容类型、缓存控制等）和请求的资源数据（例如 HTML、CSS、JS 等）。

7.浏览器接收响应并开始解析

  • 浏览器接收到响应后，开始解析 HTML 文档并逐步构建DOM 树。同时，浏览器会发出进一步的请求以获取页面中引用的资源：
  • 如果 HTML 中有外部的 CSS 文件、JS 文件或图片资源等，浏览器会并行发送 HTTP 请求来获取这些资源。

8.构建 DOM 和 CSSOM

  • DOM 树构建：浏览器从 HTML 构建文档对象模型（DOM），每个 HTML 标签都会转换成 DOM 树的节点。
  • CSSOM 构建：浏览器解析 CSS 文件或内嵌的 CSS 样式，生成 CSSOM（CSS 对象模型）。

9.合并 DOM 和 CSSOM，生成渲染树

  • 当 DOM 和 CSSOM 都构建完成后，浏览器将它们结合在一起，生成渲染树。渲染树包含页面的可见元素以及它们的样式信息。

10.布局（Layout）

  • 浏览器根据渲染树进行布局，计算每个元素的位置和大小。这一阶段决定了页面元素在屏幕上的确切位置。

11.绘制（Painting）

  • 在完成布局后，浏览器开始绘制，将渲染树的每个节点绘制到屏幕上，生成页面的视觉内容。

12.JS 执行与事件绑定

【简单】为什么会产生跨域？什么是同源策略？

跨域（Cross-Origin）是指在Web开发中，由于浏览器的同源策略（Same-Origin Policy）限制，一个网页的文档或脚本尝试获取来自不同源（不同域、不同协议、不同端口）的资源时引发的安全问题。跨域问题的存在是为了保护用户的隐私和安全，防止恶意网站滥用用户的数据。

同源策略（Same-Origin Policy）是一种浏览器安全策略，它定义了浏览器如何限制一个页面中的文档或脚本来自不同源的资源请求。同源策略要求：

  • 协议相同：请求页面和目标资源的协议必须相同，通常是HTTP或HTTPS。
  • 域名相同：请求页面和目标资源的主机名（域名）必须相同。
  • 端口相同：请求页面和目标资源的端口号必须相同。如果未指定端口号，默认为80（HTTP）或443（HTTPS）。

如果以上任何一项不满足，就被视为跨域请求，浏览器会拒绝访问跨域资源，以防止潜在的安全风险。

跨域问题的产生主要有以下原因：

  • 隐私保护：同源策略有助于保护用户的隐私，防止恶意网站获取用户的敏感数据。
  • 安全性：同源策略有助于防止跨站请求伪造（CSRF）等攻击，以及阻止网站在未经授权的情况下访问其他网站的数据。

尽管同源策略在安全性方面非常重要，但在某些情况下，Web应用程序需要进行跨域通信。为了实现跨域通信，可以通过以下方式来处理跨域问题：

  • JSONP：通过动态创建<script>标签来加载跨域的JSON数据。
  • CORS（跨源资源共享）：服务器设置响应头以允许跨域请求。这是一种现代的跨域解决方案。
  • 代理服务器：在同一域中设置一个代理服务器，用于中转跨域请求。
  • 使用WebSocket：WebSocket通信不受同源策略的限制。
  • 跨文档消息通信（PostMessage）：在不同窗口或标签页之间安全地传递信息。

需要根据具体的情况选择适合的跨域解决方案，以确保数据的安全性和可访问性。

【简单】什么是简单请求？什么是复杂请求？

在跨域请求中，根据请求的类型，请求可以分为两种主要类型：简单请求（Simple Request）和复杂请求（Preflighted Request）。

简单请求（Simple Request）：

简单请求是指浏览器直接发出的跨域请求，符合一定的条件，可以不经过预检（Preflight）。

一个请求被视为简单请求，需要满足以下条件：

  • 请求方法是以下之一：GET、HEAD、POST。
  • 请求头（HTTP headers）只包含以下几种字段之一：
  • Accept
  • Accept-Language
  • Content-Language
  • Last-Event-ID
  • Content-Type（只允许部分值，如 application/x-www-form-urlencoded、multipart/form-data、text/plain）

如果一个请求满足上述条件，它就被认为是简单请求，浏览器会直接发出该请求，而不需要进行预检。


复杂请求（Preflighted Request）：

复杂请求是指不满足简单请求条件的跨域请求。当一个请求不满足简单请求的条件时，浏览器会先发送一个预检请求（Preflight Request），以确定是否服务器允许该请求。

预检请求是一个使用 HTTP 方法OPTIONS的请求，它包含一个Origin头字段，用于指示请求的来源，以及一些其他额外的头字段。服务器接收到预检请求后，会检查是否允许实际请求。如果服务器允许，浏览器随后会发送实际请求。

复杂请求的情况包括但不限于以下情况：

  • 使用非简单请求方法，如 PUT、DELETE 等。
  • 发送自定义请求头。
  • 发送请求的Content-Type头字段为非简单值，例如application/json。

需要注意的是，对于复杂请求，服务器必须在响应中包含适当的跨域头，如Access-Control-Allow-Origin、Access-Control-Allow-Methods和Access-Control-Allow-Headers，以确保请求能够成功完成。

简单请求和复杂请求的区分对于理解跨域请求以及处理跨域问题非常重要，因为它们触发的浏览器行为和服务器端的处理方式有所不同。


【中等】说一下浏览器的缓存流程

浏览器缓存的流程涉及到资源的请求、存储和更新，缓存可以根据缓存策略和时间戳来管理。下面是浏览器的缓存流程，以及关于时间戳的说明：

  • 资源请求：当浏览器加载一个网页时，它会发起对网页中引用的各种资源（如HTML、CSS、JavaScript、图片等）的请求。
  • 缓存检查：浏览器首先检查是否已经有该资源的本地缓存副本。
  • 缓存命中：如果资源已经缓存并且仍然有效（未过期），浏览器会使用缓存的副本，而不会发起网络请求，从而提高加载速度。这通常适用于静态资源，如图片、CSS、JavaScript 文件。
  • 过期资源：如果资源已经缓存但过期了，浏览器会发起资源请求，并附带一个If-Modified-Since或If-None-Match头，以检查是否有更新的版本。
  • 服务器响应：服务器会检查请求中的条件头（If-Modified-Since或If-None-Match），如果资源没有发生变化，服务器会返回一个304 Not Modified响应，告诉浏览器继续使用缓存的资源。
  • 新资源获取：如果资源已经过期或有变化，服务器会返回新的资源数据，浏览器会存储并使用这个新的资源。

【中等】CDN 的原理是什么？

内容分发网络（CDN，Content Delivery Network）是一种通过将内容缓存到多个地理位置分散的服务器上来优化内容交付速度和可靠性的方法。CDN 的主要原理和工作机制如下：

1.分布式服务器

CDN 由多个地理上分布的服务器组成，这些服务器被称为节点。每个节点可以缓存内容的副本。通过将内容分布在不同的位置，CDN 可以确保用户请求得到快速响应，因为请求可以由距离最近的节点处理。

2.内容缓存

当用户请求某个内容时，CDN 会首先检查距离最近的节点是否已经缓存了该内容。如果缓存中存在，节点会直接将内容发送给用户，这减少了从原始服务器获取内容的延迟。缓存策略通常基于内容的类型和访问频率来确定缓存的存储时间。

3.负载均衡

CDN 使用负载均衡技术在多个服务器之间分配用户请求。这不仅提高了性能和响应速度，还减少了单个服务器的负载，防止服务器过载或宕机。

4.地理定位

CDN 利用地理定位技术确定用户的地理位置，并将用户的请求定向到离他们最近的服务器节点。这种方式可以显著减少网络延迟，提高内容传输速度。

5.动态内容加速

对于动态生成的内容，CDN 也能提供加速服务。虽然动态内容不能像静态内容那样直接缓存，但 CDN 可以通过优化路径、协议（如 HTTP/2）和压缩技术来减少传输时间。

6.安全性

CDN 提供额外的安全层，包括 DDoS 保护、WAF（Web 应用防火墙）和 SSL/TLS 加密。CDN 节点能够检测并拦截恶意流量，保护原始服务器不受攻击。

7.内容预取和更新

CDN 能够根据用户的访问模式预先获取和缓存内容。同时，当内容更新时，CDN 也会同步更新缓存，确保用户获取到最新的内容。


【中等】DNS 如何查询域名的

DNS（域名系统）是一个用于将域名（例如，www.example.com）转换为 IP 地址的分布式数据库系统。DNS 查询是通过一系列步骤来完成的，通常涉及到本地 DNS 服务器、根域名服务器、顶级域名服务器和授权域名服务器。以下是 DNS 查询的基本过程：

  • 本地 DNS 查询：当用户在浏览器中输入一个域名（例如，www.example.com）时，首先会进行本地 DNS 查询。本地计算机或网络路由器上保存着一个本地 DNS 缓存，它包含了最近查询过的域名和其对应的 IP 地址。如果所需的域名在本地缓存中已经存在，查询就会立即结束。
  • 根域名服务器查询：如果域名不在本地缓存中，本地 DNS 服务器会向根域名服务器发出查询请求。根域名服务器是全球 DNS 层次结构的最高级别，负责维护顶级域名服务器（如.com、.org、.net 等）的地址。根域名服务器不存储域名与 IP 地址的映射，但它会告诉本地 DNS 服务器如何找到顶级域名服务器。
  • 顶级域名服务器查询：本地 DNS 服务器根据根域名服务器的响应，向顶级域名服务器发送查询请求，以确定域名的顶级域名服务器。例如，对于 www.example.com，本地 DNS 服务器会查询.com 顶级域名服务器。顶级域名服务器会提供指向下一个级别的域名服务器（授权域名服务器）的信息。
  • 授权域名服务器查询：本地 DNS 服务器继续查询指向的授权域名服务器，以获取域名的实际 IP 地址。授权域名服务器通常由域名的拥有者或注册商维护。它包含了与特定域名相关的 IP 地址信息。
  • 解析响应：一旦本地 DNS 服务器获得了域名的实际 IP 地址，它将响应返回给用户的计算机或设备。用户的计算机将使用这个 IP 地址来建立连接并加载网站或资源。
  • 本地缓存更新：本地 DNS 服务器会将查询的结果缓存起来，以便下次查询相同域名时能够更快地获取响应。

这是 DNS 查询的基本过程。DNS 的分布式结构和缓存机制有助于减少对根域名服务器和顶级域名服务器的负载，并提高域名解析的效率。 DNS 是互联网的关键基础设施，它使我们能够使用易记的域名来访问互联网上的各种资源。


【中等】XSS、CSRF 攻击的原理以及防御手段

XSS（跨站脚本攻击）攻击原理：

XSS 攻击是一种常见的 Web 攻击，其原理是攻击者向网页中插入恶意脚本代码，使用户的浏览器执行这些脚本，从而达到攻击的目的。XSS 攻击通常分为以下几种类型：

  • 存储型 XSS：攻击者将恶意脚本存储在服务器上，当用户访问包含这些脚本的页面时，脚本被执行。
  • 反射型 XSS：攻击者将恶意脚本作为参数包含在URL中，当用户点击包含恶意脚本的链接时，脚本被执行。
  • DOM 型 XSS：攻击者利用浏览器的 DOM 操作，动态修改页面内容，插入恶意脚本，然后用户浏览器执行这些脚本。

XSS 防御手段：

  • 输入验证：对用户输入的数据进行验证和过滤，不信任的数据应该被拒绝或编码，以防止恶意脚本的注入。
  • 输出编码：在将用户输入的数据输出到页面时，对数据进行合适的编码，以确保浏览器将其视为纯文本而不是可执行的脚本。
  • CSP（内容安全策略）：CSP 是一种 HTTP 头，它允许网站管理员指定哪些资源可以加载和执行，以限制恶意脚本的来源。
  • HttpOnly 和 Secure Cookie 标志：使用HttpOnly标志将 cookie 标记为仅通过 HTTP 访问，使用Secure标志强制使用 HTTPS 连接，从而减少 XSS 攻击的风险。
  • X-XSS-Protection 头：浏览器可以通过X-XSS-ProtectionHTTP 头来启用内置的反射型和存储型 XSS 攻击检测和防御。

CSRF（跨站请求伪造）攻击原理：

CSRF 攻击是一种攻击方式，利用用户已登录的身份，以用户不知情的情况下，伪造用户的请求，执行一些非法操作。攻击者通常在受信任的网站上放置恶意链接或图像，当用户访问这些网站时，他们的登录凭证会被用于发起非授权的操作。

CSRF 防御手段：

  • CSRF Token：为每个用户生成一个唯一的 CSRF 令牌，并将该令牌与用户的会话关联。在每个请求中，包括表单提交，都需要包含这个令牌。服务器在接收请求时验证令牌，如果不匹配，则拒绝请求。
  • SameSite Cookie 属性：通过设置SameSite属性为"Strict"或"Lax"来减少 CSRF 攻击的风险。这可以限制跨站点 Cookie 的发送，从而减少攻击者的机会。
  • 检查 Referer 头：服务器可以检查请求中的Referer头来确保请求来自合法来源。
  • 双重提交 Cookie：将 CSRF 令牌存储在一个 cookie 中，同时也将它包含在请求的头部中。服务器会验证这两个值是否匹配，从而确定请求是否合法。
  • 限制敏感操作：对于敏感操作（如修改密码、更改邮箱地址等），需要用户进行额外的身份验证，例如输入密码或提供其他凭证。
  • 及时退出登录：用户应定期退出登录，以降低攻击者滥用用户凭证的机会。

以上是一些常见的防御手段，它们可以单独或组合使用，以减少 XSS 和 CSRF 攻击的风险。不同的应用可能需要采用不同的防御措施，具体的安全策略应根据应用的特定需求来制定。

【困难】对 Web Worker 有了解吗？什么是 Web Worker，它的应用场景是什么？

Web Worker是 HTML5 引入的一项技术，它允许在后台线程中执行 JavaScript 代码，而不会阻塞主线程（UI 线程）。Web Worker 可以在独立的线程中执行计算密集型任务，从而提高 web 应用的性能和响应速度。

Web Worker 的主要特点包括：

  • 并行计算：Web Worker 允许在后台线程中并行执行代码，不会阻塞主线程的执行，因此不会导致页面冻结或卡顿。
  • 独立的全局对象：每个 Web Worker 都有自己独立的全局对象，与主线程的全局对象分离。这意味着它们不共享相同的数据和变量。
  • 受限的访问：Web Worker 不能访问 DOM 元素，浏览器窗口对象，或执行同步 XMLHttpRequest 请求等。这是出于安全性和可靠性的考虑。
  • 通信机制：主线程和 Web Worker 之间可以通过消息传递进行通信。主线程可以向 Web Worker 发送消息，而 Web Worker 也可以向主线程发送消息。

应用场景：

  • 计算密集型任务：Web Worker 最常用的场景是执行计算密集型任务，如图像处理、数据计算、密码学等。通过将这些任务移至后台线程，可以提高网页性能，保持响应速度。
  • 长时间运行的任务：长时间运行的任务，如后台数据同步、文件解析、数据转换等，可以放在 Web Worker 中，以避免阻塞用户界面。
  • 多线程并行处理：当需要同时处理多个任务时，可以创建多个 Web Worker 实例以实现并行处理。
  • 网络请求：虽然 Web Worker 不能执行同步 XMLHttpRequest 请求，但可以用于执行异步网络请求，以避免阻塞主线程。
  • 游戏开发：在游戏开发中，Web Worker 可以用于执行游戏逻辑、物理引擎等任务，从而提高游戏性能。

总之，Web Worker 是一项强大的技术，用于处理多线程编程任务，可以提高 web 应用的性能和用户体验，特别是在执行计算密集型或长时间运行的任务时非常有用。

【简单】HTTP 请求常见状态码都有什么？

HTTP（Hypertext Transfer Protocol）是一种用于在网络上传输数据的协议，常见的状态码用于指示请求的结果。以下是HTTP常见的状态码及其含义：

1xx（信息性状态码）：请求正在处理

  • 100 Continue: 服务器已接收到请求的初始部分，客户端应该继续发送其余部分。
  • 101 Switching Protocols: 服务器已经理解客户端的请求，将通过升级协议的方式来响应。

2xx（成功状态码）：请求成功处理

  • 200 OK: 请求已成功，服务器返回了请求的数据。
  • 201 Created: 请求已成功，并且服务器创建了新的资源。
  • 204 No Content: 请求已成功，但服务器没有返回任何内容。

3xx（重定向状态码）：需要进行额外的操作以完成请求

  • 301 Moved Permanently: 请求的资源已经永久移动到新的位置。
  • 302 Found (或者307 Temporary Redirect): 请求的资源临时移动到新的位置。
  • 304 Not Modified: 客户端可以使用缓存的版本，因为资源未发生更改。

4xx（客户端错误状态码）：请求包含错误或无法完成

  • 400 Bad Request: 请求无效，服务器不理解或无法处理。
  • 401 Unauthorized: 请求需要认证或身份验证。
  • 403 Forbidden: 服务器拒绝请求，客户端没有权限访问资源。
  • 404 Not Found: 请求的资源未找到。

5xx（服务器错误状态码）：服务器无法完成有效请求

  • 500 Internal Server Error: 服务器遇到内部错误，无法完成请求。
  • 502 Bad Gateway: 服务器充当网关或代理，从上游服务器接收到无效响应。
  • 503 Service Unavailable: 服务器当前不可用，通常是临时性的。
  • 504 Gateway Timeout: 服务器充当网关或代理，但没有及时接收到来自上游服务器的响应。


这些是HTTP中最常见的状态码，还有其他一些少见的状态码用于特定的情况。不同的状态码提供了关于请求处理结果的信息，帮助客户端和开发人员更好地理解和处理HTTP请求和响应。


【中等】简述 http 的缓存机制

HTTP缓存是一种用于减少网络流量和提高网站性能的重要技术。HTTP缓存可以分为两种主要类型：强制缓存和协商缓存。它们的区别在于缓存策略和服务器与客户端之间的交互方式。

  • 强制缓存：
  • 强制缓存是通过设置响应头来控制的，通常使用Cache-Control和Expires头。
  • Cache-Control头中的max-age指定了资源在客户端缓存中的有效时间（秒），例如：Cache-Control: max-age=3600表示资源在客户端缓存中有效期为1小时。
  • Expires头是一个过时的方法，它指定了资源的过期日期，客户端将根据这个日期来判断资源是否过期。
  • 当资源被强制缓存，客户端在下次请求该资源时，会首先检查缓存中的有效期，如果资源尚未过期，客户端将直接从缓存中加载资源，而不向服务器发送请求。
  • 强制缓存通常用于不经常更改的静态资源，它可以减少服务器的负载并加快页面加载速度。
  • 协商缓存（304）：
  • 协商缓存是通过使用请求头来控制的，通常使用If-Modified-Since和If-None-Match等请求头。
  • 当客户端向服务器请求资源时，它会发送先前缓存的资源的元信息，如最后修改时间（If-Modified-Since）或资源的标识符（If-None-Match）。
  • 服务器会根据这些信息判断资源是否发生了变化，如果资源未发生变化，服务器会返回一个304 Not Modified响应，告诉客户端可以继续使用缓存中的资源。
  • 如果资源已经发生了变化，服务器会返回新的资源，并将其缓存，客户端会更新缓存并使用新的资源。
  • 协商缓存通常用于动态或频繁更改的内容，它减少了不必要的网络传输，但需要服务器进行额外的计算以判断资源是否已更改。

综上所述，强制缓存是通过设置响应头来告诉客户端在一定时间内直接使用缓存，而协商缓存是通过比较请求头中的信息来判断资源是否需要重新加载。通常，强制缓存优先级较高，如果资源被强制缓存，将不会进行协商缓存的检查。如果强制缓存失效，才会进行协商缓存的判断。这两种缓存策略可以结合使用，以提高性能并减少服务器负载。


【中等】HTTP 与 HTTPS 的区别？

HTTP（Hypertext Transfer Protocol）和HTTPS（Hypertext Transfer Protocol Secure）都是用于在客户端和服务器之间传输数据的协议，但它们之间存在重要的区别，主要涉及到安全性和数据传输的方式：

  • 安全性：
  • HTTP：HTTP是一种不安全的协议，数据在传输过程中以明文形式传递，容易受到中间人攻击，例如窃听或篡改数据。
  • HTTPS：HTTPS通过使用SSL/TLS协议来加密数据传输，使数据在传输过程中被加密，从而提供了更高的安全性。这有助于防止窃听、数据篡改和身份验证等攻击。
  • 端口号：
  • HTTP：默认使用端口80来传输数据。
  • HTTPS：默认使用端口443来传输数据。
  • URL前缀：
  • HTTP：URL以"http://"开头。
  • HTTPS：URL以"https://"开头。
  • 证书和身份验证：
  • HTTP：不提供身份验证机制，因此无法验证服务器的真实性。
  • HTTPS：通过使用数字证书，服务器可以验证其身份，从而确保客户端连接到正确的服务器。这有助于防止中间人攻击。
  • 加密：
  • HTTP：数据在传输过程中未加密，容易受到窃听。
  • HTTPS：数据在传输过程中被加密，只有客户端和服务器能够解密和理解数据。
  • SEO和排名：
  • 搜索引擎（如Google）通常更喜欢HTTPS网站，因此使用HTTPS有助于提高网站的搜索引擎排名。

总之，HTTPS是HTTP的安全版本，通过数据加密和身份验证提供了更高的安全性。在今天的互联网环境中，为了保护敏感信息和用户隐私，使用HTTPS是非常重要的，特别是对于涉及到用户数据的网站和在线交易的网站。因此，许多网站都已经切换到了HTTPS，以提供更安全的用户体验。

【中等】HTTP 1.1 、HTTP 2 、HTTP 3 的区别

请注意，HTTP/1.1、HTTP/2和HTTP/3是HTTP协议的不同版本，而HTTPS是HTTP协议的安全版本。HTTPS可以用于加密HTTP/1.1、HTTP/2和HTTP/3的通信。让我解释它们之间的关系和区别：

  • HTTP/1.1：
  • HTTP/1.1是HTTP协议的第一个广泛部署的版本，它于1999年发布。
  • 它采用明文传输，不提供数据加密和安全性。
  • 对于每个请求，通常需要建立一个新的TCP连接，这可能会导致性能问题，因为TCP连接的建立和拆除需要时间。
  • HTTP/1.1引入了持久连接（Keep-Alive）来减少建立新连接的开销，但仍然存在性能问题。
  • HTTP/2：
  • HTTP/2于2015年发布，是HTTP/1.1的后续版本。
  • HTTP/2引入了多路复用，允许多个请求和响应同时在单个连接上传输，减少了延迟。
  • 它采用二进制传输，以减少头部大小，从而提高效率。
  • HTTP/2支持服务器推送，服务器可以预加载资源，从而提高性能。
  • HTTP/2是一种相对较新的协议，但已经被广泛采用。
  • HTTP/3：
  • HTTP/3是HTTP协议的最新版本，于2020年发布。
  • HTTP/3采用了QUIC协议作为底层传输协议，取代了TCP，以减少延迟和提高性能。
  • 它继承了HTTP/2的多路复用和头部压缩等特性，同时具有更好的性能。
  • HTTP/3的一个重要特点是零RTT连接，允许客户端在第一次连接时发送请求，而无需等待握手完成，从而进一步减少延迟。

总结：HTTP/1.1、HTTP/2和HTTP/3是不同版本的HTTP协议，它们都可以使用HTTPS来加密通信。HTTP/2和HTTP/3引入了许多性能改进，特别是在多路复用和降低延迟方面。HTTPS协议用于在这些HTTP版本上提供数据加密和安全性。因此，对于保护数据和提高性能，推荐使用HTTPS，并使用最新的HTTP版本，如HTTP/2或HTTP/3。


【困难】HTTPS 的加密方式是什么？为什么会是安全的？

HTTPS使用了多种加密技术来保护数据的机密性和完整性，使通信过程更加安全。其中最常见的加密方式是基于公钥加密和对称密钥加密的结合，通常使用SSL（Secure Sockets Layer）或其继任者TLS（Transport Layer Security）协议来实现。

以下是HTTPS的主要加密方式及其工作原理：

  • 公钥加密（非对称加密）：
  • HTTPS通信的开始阶段，服务器会向客户端提供公钥，客户端使用公钥加密将其数据发送给服务器。
  • 公钥加密是一种非对称加密，意味着数据用公钥加密后，只有对应的私钥才能解密它。
  • 这意味着即使攻击者拦截了数据传输，也无法轻易解密数据。
  • 对称加密：
  • 一旦安全连接建立，客户端和服务器之间会协商生成一个对称密钥。
  • 对称加密使用相同的密钥来加密和解密数据，因此更快速而高效。
  • 对称加密用于实际的数据传输，因为非对称加密通常比较慢。
  • 完整性保护：
  • 数据在传输过程中还会通过哈希函数等机制来计算校验和（数字签名）以保证数据的完整性。如果数据在传输过程中被篡改，校验和将不匹配，从而警告双方。
  • 数字证书：
  • 服务器使用数字证书来证明其身份。数字证书包含服务器的公钥以及相关信息，由受信任的证书颁发机构（CA）签名，以确保服务器的真实性。
  • 客户端可以验证证书的签名，确保它是由受信任的CA颁发的。这有助于防止中间人攻击。

为什么HTTPS是安全的：

  • 数据加密：HTTPS使用非对称和对称加密来保护数据机密性。即使攻击者截获了数据，也无法轻松解密。
  • 完整性保护：HTTPS通过校验和等机制来确保数据的完整性。如果数据被篡改，双方会察觉到。
  • 身份验证：HTTPS使用数字证书来验证服务器的身份，确保用户连接到正确的服务器，而不是中间人。
  • 受信任的证书颁发机构：浏览器内置了受信任的CA根证书，以验证服务器的数字证书。这减少了中间人攻击的风险。

总之，HTTPS的安全性基于其加密和身份验证机制，确保数据在传输过程中保持机密、完整和可信。这使得HTTPS成为保护用户隐私和防止数据被篡改的重要工具，特别是在互联网上进行敏感交易和数据传输时。

【困难】对称加密和非对称加密的区别

对称加密和非对称加密是两种常见的加密技术，它们在数据加密和解密过程中的工作方式和原理有很大的区别。以下是它们的区别以及一些实现方案：

  • 对称加密：
  • 工作方式：在对称加密中，相同的密钥用于加密和解密数据。这意味着发送者和接收者都必须知道并使用相同的密钥来加密和解密通信中的数据。
  • 速度：对称加密通常比非对称加密快，因为加密和解密使用相同的密钥。
  • 安全性：对称加密相对来说更容易受到中间人攻击的威胁，因为发送方和接收方都必须共享相同的密钥，并且如果密钥被泄漏，通信将不再安全。
  • 实现方案：常见的对称加密算法包括DES（Data Encryption Standard）、3DES、AES（Advanced Encryption Standard）和RC4等。
  • 非对称加密：
  • 工作方式：在非对称加密中，有两个密钥，一个是公钥，另一个是私钥。公钥用于加密数据，私钥用于解密数据。发送方知道接收方的公钥，但不知道私钥。
  • 速度：非对称加密通常比对称加密慢，因为使用不同的密钥进行加密和解密。
  • 安全性：非对称加密提供更高的安全性，因为即使公钥泄漏，攻击者仍然需要私钥来解密数据。
  • 实现方案：常见的非对称加密算法包括RSA、DSA、ECC（椭圆曲线加密）等。

需要注意的是，通常非对称加密用于在安全通信的初始阶段建立对称密钥，然后双方继续使用对称加密来加密和解密实际的通信数据。这种方式兼顾了安全性和性能。

强缓存（Strong Caching）

强缓存允许浏览器直接从本地缓存中读取资源，而不向服务器发送请求。强缓存通过HTTP头中的Cache-Control和Expires字段来控制。

  • Cache-Control: 这是HTTP/1.1引入的头字段，支持多种指令，例如：
  • max-age: 指定资源被缓存的最长时间（以秒为单位）。
  • public: 指示响应可以被任何缓存存储，包括CDN缓存。
  • private: 指示响应只能被浏览器缓存。
  • no-cache: 强制浏览器每次都向服务器发送请求进行验证。
  • no-store: 禁止缓存，所有请求和响应都不会被缓存。
  • Expires: 这是HTTP/1.0使用的头字段，用于指定资源的过期时间点。这个时间点是一个具体的日期和时间。例如：Expires: Tue, 19 Jan 2021 20:00:00 GMT。如果Cache-Control头字段也存在，Expires将被忽略。

协商缓存（Negotiated Caching）

【中等】SSR 渲染、CSR渲染、SSG 渲染、同构渲染的区别是什么？

SSR（Server-Side Rendering）、CSR（Client-Side Rendering）、SSG（Static Site Generation）和同构渲染都是与前端Web开发相关的渲染方式，它们有不同的工作方式和应用场景。以下是它们之间的区别：

  • SSR（Server-Side Rendering）：
  • 服务端渲染。
  • 在服务器端生成HTML，并将完整的HTML页面发送到浏览器。
  • 通常用于传统的多页面应用（MPA）或需要搜索引擎优化（SEO）的单页面应用（SPA）。
  • 支持初次加载时更快的首屏渲染，有利于SEO，但可能会增加服务器负担。
  • CSR（Client-Side Rendering）：
  • 客户端渲染。
  • 仅在浏览器中使用JavaScript动态渲染内容。
  • 适用于单页面应用（SPA）和应用内路由，通过AJAX或Fetch请求从服务器获取数据。
  • 首次加载可能会较慢，不利于SEO，但可以提供更流畅的用户体验。
  • SSG（Static Site Generation）：
  • 静态站点生成。
  • 在构建时生成HTML文件，不需要服务器进行动态渲染。
  • 适用于内容不频繁更改的网站，如博客、企业站点等。
  • 静态文件可以通过CDN分发，提供高性能和低成本的解决方案。
  • 同构渲染（Isomorphic Rendering）：
  • 也称为Universal渲染。
  • 结合了SSR和CSR的优点，既在服务器端生成HTML，又在客户端进行交互。
  • 首次加载时有利于SEO和性能，同时提供SPA的交互性。
  • 通常需要专业的框架和库，如Next.js（React）或Nuxt.js（Vue）。

总结来说，SSR和SSG适用于需要SEO和性能优化的情况，CSR适用于交互性强的单页面应用，而同构渲染是一种折中的方法，结合了SSR和CSR的优点，适用于需要SEO和交互性的场景。选择合适的渲染方式取决于项目的需求和约束条件。


【中等】应该在什么场景下使用SSR渲染，什么场景下使用CSR渲染？

选择使用SSR（Server-Side Rendering）或CSR（Client-Side Rendering）应该根据项目的需求和目标来确定。以下是一些通常建议的使用场景：

使用SSR渲染的场景：

  • SEO重要的网站：如果你的网站需要在搜索引擎中获得良好的排名，SSR是一个不错的选择，因为搜索引擎可以轻松地抓取和索引服务器端渲染的HTML内容。
  • 首屏加载速度至关重要：对于需要快速加载首屏内容的应用程序，SSR通常比CSR更快，因为用户在首次加载时将获得完整的渲染页面。
  • 有限的客户端计算能力：如果你的客户端设备有限的计算能力，不适合进行复杂的客户端渲染，SSR可以将部分渲染工作转移到服务器上，降低客户端负担。
  • 多页面应用（MPA）：在传统的多页面应用中，每个页面都可以使用SSR，以确保每个页面都有良好的性能和SEO。

使用CSR渲染的场景：

  • 富交互性单页面应用（SPA）：对于需要大量客户端交互的应用程序，CSR是常见选择，因为它可以提供更流畅的用户体验，无需每次导航都从服务器加载整个页面。
  • 内容频繁变更的应用：如果你的应用的内容频繁变化，或者需要根据用户行为动态加载数据，CSR可以更灵活地处理这些情况。
  • 复杂的前端逻辑：如果你的前端逻辑非常复杂，包括大量的客户端路由、状态管理和组件交互，CSR可以更好地满足这些要求。
  • 单页应用（SPA）框架：使用流行的单页应用框架（如React、Angular、Vue.js）时，CSR是常见的选择，因为这些框架天生支持客户端渲染。

需要注意的是，有些项目也可以采用同构渲染的方法，结合了SSR和CSR的优点，以满足多种需求。最终的决策应该根据项目的特定需求、性能要求和SEO优化等因素来做出。


【中等】同构渲染应该如何实现？

同构渲染（Isomorphic Rendering）是一种结合了服务器端渲染（SSR）和客户端端渲染（CSR）的方法，旨在兼顾两者的优点。同构渲染旨在提供更好的性能、搜索引擎优化（SEO）、首次加载速度以及流畅的用户体验。以下是实现同构渲染的一般步骤：

  • 选择合适的框架：大多数现代JavaScript框架提供了支持同构渲染的能力。例如，React有Next.js，Vue.js有Nuxt.js，Angular也有Angular Universal等。
  • 服务器端渲染：在服务器上设置一个端点，当用户访问应用时，服务器会渲染初始HTML，并将其发送到浏览器。这有助于提供较快的首次加载速度和对搜索引擎友好的HTML。
  • 客户端端渲染：在浏览器中，客户端代码接管页面的控制，并继续处理后续的页面渲染和交互。这确保了应用的交互性和动态性。
  • 共享代码：同构渲染应用需要共享部分代码，以确保在服务器和客户端上都能正确渲染页面。这包括组件、路由配置、数据获取等。
  • 数据获取：在同构渲染中，数据获取可以在服务器端和客户端之间共享，以避免不必要的数据获取。通常，首次加载时数据可以在服务器端获取并附加到页面，后续的数据获取可以在客户端完成。
  • 路由处理：同构应用需要能够在服务器和客户端上处理路由，以确保用户导航时页面正确渲染。

实现同构渲染的主要目标是在首次加载时提供更快的渲染速度和更好的SEO，同时在客户端上提供交互性。同构渲染不一定需要纯粹的SSR或CSR，因为它结合了两者的优点。但在某些情况下，仍然可以考虑使用SSR或CSR来满足特定需求：

  • SSR用例：如果你有一些静态内容或需要SEO优化的页面，仍然可以使用SSR来渲染这些页面。同构渲染可以与SSR结合使用，以提供SEO友好的内容。
  • CSR用例：对于高度交互性的页面或部分，你仍然可以使用CSR，因为同构渲染中的CSR部分可以处理用户交互。

综上所述，同构渲染是一种强大的渲染方法，可以在多方面提高应用性能和用户体验，但根据具体需求，仍然可以结合使用SSR或CSR以满足不同的用例。


【中等】有了同构渲染之后，是不是意味着所有的项目都可以使用同构渲染而放弃单独的 SSR、CSR？

尽管同构渲染在许多情况下提供了SSR和CSR的优点，但并不意味着所有项目都应该使用同构渲染而放弃独立的SSR或CSR。项目的具体需求和考虑因素可能会导致你选择继续使用纯SSR或CSR，或者在同构渲染中结合使用它们。以下是一些情况，可能不适合完全采用同构渲染：

  • 项目复杂度：对于简单的项目，可能不需要同构渲染，因为纯CSR可能已经足够快速和有效。
  • 性能需求：对于高性能要求的应用，如大规模电子商务平台或在线游戏，可能需要更多的细粒度控制，此时纯SSR或CSR可能更合适。
  • 资源限制：在某些情况下，服务器端资源有限，或者希望减少服务器端负担的项目可能更倾向于纯CSR，以减轻服务器端的工作。
  • 交互需求：对于完全交互性和SPA功能的应用，CSR可能更适合，因为它可以实现更强大的客户端交互。
  • 开发团队能力：同构渲染可能需要更多的工程和开发成本，需要更复杂的配置和代码结构。如果开发团队没有相关经验或资源，可能会选择使用单一的SSR或CSR。

总之，决定采用同构渲染还是保持纯SSR或CSR取决于项目的具体需求、性能目标、开发团队能力和资源限制等因素。同构渲染是一个有用的渲染方法，但并不适用于每种情况。在选择时，建议仔细评估项目的要求和优势，以确保选取最适合的渲染策略。

当用户在 app1.a.com 系统中的 session 中查找不到时，因为之前登录会在 sso.a.com 中设置二级域名的 cookie（在这里是.a.com），所以 app1.a.com 的请求就可以拿到该 cookie，接着用该 cookie 请求 sso.a.com 的接口，sso.a.com 系统拿到 cookie 后会在自己的 session 中查找是否存在，存在则返回 session 给 app1.a.com 系统同步，否则让用户跳转到 sso.a.com 系统进行登录。


04：不同域下的单点登录

不同域下的单点登录参考 CAS 官网上的标准流程，具体流程如下：

  • 用户访问 app 系统，app 系统是需要登录的，但用户现在没有登录。
  • 跳转到 SSO 登录系统，SSO 系统也没有登录，弹出用户登录页。
  • 用户填写用户名、密码，SSO 系统进行认证后，将登录状态写入 SSO 的 session，浏览器（Browser）中写入 SSO 域下的 Cookie。
  • SSO 系统登录完成后会生成一个 ST（Service Ticket），然后跳转到 app 系统，同时将 ST 作为参数传递给 app 系统。
  • app 系统拿到 ST 后，从后台向 SSO 发送请求，验证 ST 是否有效。
  • 验证通过后，app 系统将登录状态写入 session 并设置 app 域下的 Cookie。

至此，跨域单点登录就完成了。以后我们再访问 app 系统时，app 就是登录的。接下来，我们再看看访问 app2 系统时的流程。用户访问 app2 系统，app2 系统没有登录，跳转到 SSO。

  • 由于 SSO 已经登录了，不需要重新登录认证。
  • SSO 生成 ST，浏览器跳转到 app2 系统，并将 ST 作为参数传递给 app2。
  • app2 拿到 ST，后台访问 SSO，验证 ST 是否有效。
  • 验证成功后，app2 将登录状态写入 session，并在 app2 域下写入 Cookie。

这样，app2 系统不需要走登录流程，就已经是登录了。SSO，app 和 app2 在不同的域，它们之间的 session 不共享也是没问题的。参考链接：

  • https://developer.aliyun.com/article/636281(opens new window)
  • 微服务登录流程分析(opens new window)

当然此方法涉及安全性问题，其实在cookie中保存token同样存在安全性问题。推荐使用sso框架CAS实现单点登录。

06：SSO原理（单点登录的过程）

以登录天猫为例进行说明：

  • 当⽤户第⼀次访问淘宝的时候，因为还没有登录，会被引导到认证中⼼进⾏登录。
  • 根据⽤户提供的登录信息，认证系统进⾏身份验证，如果通过，则登录成功，并返回给⽤户⼀个认证的凭据（JWT token）。
  • 当⽤户访问天猫时，就会将这个 JWT token 带上，作为⾃⼰认证的凭据。
  • 应⽤系统接收到请求后会把 JWT token 送到认证中⼼进⾏校验。
  • 如果通过校验，⽤户就可以在不⽤再次登录的情况下访问天猫了。

07：什么是CAS？

CAS框架：CAS（Central Authentication Service，即：统一认证服务）是实现SSO单点登录的框架。

CAS分为两部分

  • CAS Server
  • CAS Client。

CAS Server用来负责用户的认证工作，就像是把第一次登录用户的一个标识存在这里，以便此用户在其他系统登录时验证其需不需要再次登录。CAS Client就是我们自己开发的应用程序，需要接入CAS Server端。当用户访问我们的应用时，首先需要重定向到CAS Server端进行验证，要是原来登陆过，就免去登录，重定向到下游系统，否则进行用户名密码登陆操作。

08：CAS中3个术语?

Ticket Granting ticket (TGT) ：可以认为是CAS Server根据用户名密码生成的一张票，存在Server端Ticket-granting cookie (TGC) ：其实就是一个Cookie，存放用户身份信息，由Server发给Client端Service ticket (ST) ：由TGT生成的一次性票据，用于验证，只能用一次。相当于Server发给Client一张票，然后Client拿着这个票再来找Server验证，看看是不是Server签发的。

09：CAS处理流程?

  • 用户访问网站，第一次来，重定向到 CAS Server，发现没有cookie，所以再重定向到CAS Server端的登录页面，并且URL带有网站地址，便于认证成功后跳转，形如 http ?/cas-server:8100/login?service=http ?/localhost:8081注意：service后面这个地址就是登录成功后要重定向的下游系统URL。
  • 在登陆页面输入用户名密码认证，认证成功后cas-server生成TGT，再用TGT生成一个ST。 然后再第三次重定向并返回ST和cookie(TGC)到浏览器
  • 浏览器带着ST再访问想要访问的地址：http ?/localhost:8081/?ticket=ST-25939-sqbDVZcuSvrvBC6MQlg5注意：ticket后面那一串就是ST
  • 浏览器的服务器收到ST后再去cas-server验证一下是否为自己签发的，验证通过后就会显示页面信息，也就是重定向到第1步service后面的那个URL首次登陆完毕。
  • 再登陆另一个接入CAS的网站，重定向到CAS Server，server判断是第一次来（但是此时有TGC，也就是cookie，所以不用去登陆页面了），但此时没有ST，去cas-server申请一个于是重定向到cas-server，形如：http: //cas-server:8100/login?service=http ?/localhost:8082 && TGC(cookie) (传目标地址和cookie）
  • cas-server生成了ST后重定向给浏览器http ?/localhost:8082/?ticket=ST-25939-sqfsafgefesaedswqqw5-xxxx
  • 浏览器的服务器收到ST后再去cas-server验证一下是否为自己签发的，验证通过后就会显示页面信息（同第4步）

10：什么是Token?

Token的意思是“令牌”，是服务端生成的一串字符串，作为客户端进行请求的一个标识。当用户第一次登录后，服务器生成一个token并将此token返回给客户端，以后客户端只需带上这个token前来请求数据即可，无需再次带上用户名和密码。简单Token的组成；uid(用户唯一的身份标识)、time(当前时间的时间戳)、sign（签名，token的前几位以哈希算法压缩成的一定长度的十六进制字符串。为防止token泄露）。

11：OAuth是什么？

OAuth 是一个行业的标准授权协议，主要用来授权第三方应用获取有限的权限。实际上它就是一种授权机制，最终目的是为第三方应用颁发一个有时效性令牌 token，使得第三方应用能够通过该令牌获取相关的资源。OAuth 2.0 比较常用的场景就是第三方登录，当你的网站接入了第三方登录时一般就是使用的 OAuth 2.0 协议。现在OAuth 2.0也常见于支付场景（微信支付、支付宝支付）和开发平台（微信开放平台、阿里开放平台等等）。

12：介绍下Access Token ?

Access Token 是在 Oauth2.0 协议中，客户端访问资源服务器时需要带上的令牌（其实就是一段全局唯一的随机字符串）。拥有这个令牌代表着得到用户的授权。令牌里面包含哪个用户 在什么时候 授权给哪个app去做什么事情。当然这些信息是不能直接从Access Token 看出来的，而是存在平台方的数据库中，平台可以用Access Token 作为 key 去查询出这些信息，然后验证调用方是否有权限。

13：介绍下Refresh Token?

Refresh Token是专用于刷新 Access Token 的 token。如果没有Refresh Token，也可以刷新 Access Token，但每次刷新都要用户输入登录用户名与密码。有了 Refresh Token，客户端直接用Refresh Token 去更新Access Token，无需用户进行额外的操作。

14：介绍下JWT？

Json Web Token (JWT)是为了在网络应用环境间传递声明而执行的一种基于JSON的开放标准。该token被设计为紧凑且安全，特别适用于分布式站点单点登录场景。JWT由头部（header)、载荷（payload)、签证（signature) 三部分组成。
