【困难】Vue2 中的响应式实现方案有什么缺点？

Vue 2 中的响应式实现方案有一些缺点，这些缺点可能会在某些情况下对开发过程和性能产生影响。以下是一些Vue 2中响应式实现的主要缺点：

  • 初始化性能开销:
  • Vue 2的响应式系统在初始化时需要为每个数据属性创建getter和setter，这会导致初始化性能开销较大，特别是当数据对象非常大或嵌套层次很深时。
  • 不支持新增属性的响应:
  • 一旦一个对象被Vue实例化，它的属性就会被加入响应系统。这意味着如果你想要让一个新属性具有响应性，你必须在对象初始化时定义它，否则它将不会具有响应性。这对于动态添加属性时可能会导致问题。
  • 递归嵌套限制:
  • Vue 2的响应式系统在处理嵌套对象或数组时，会有一些限制。嵌套层次过深时，可能会导致性能问题或栈溢出错误。
  • 跟踪不可枚举属性的困难:
  • Vue 2无法直接监视对象的不可枚举属性，因为它们无法被getter和setter劫持。这可能导致需要通过Vue.set方法来设置响应式属性，增加了一些复杂性。
  • 无法跟踪数组索引和长度的变化:
  • Vue 2的响应式系统可以跟踪数组的变化，但对于数组索引和长度的变化并不敏感，因此需要使用特定的数组方法（如push、pop、shift、unshift、splice）来触发响应。
  • 性能优化挑战:
  • 在某些情况下，Vue 2的响应式系统可能导致不必要的重新渲染和更新。为了解决性能问题，需要手动使用shouldComponentUpdate（React 方法）或computed属性来进行性能优化。
  • 难以调试:
  • 当在开发过程中遇到响应式问题时，调试可能会有挑战。Vue 2的响应式系统在内部使用getter和setter来追踪变化，这可能使问题的定位和调试变得复杂。


需要注意的是，虽然Vue 2的响应式实现有一些缺点，但它仍然是一个功能强大的框架，并且在大多数应用中表现出色。随着Vue 3的发布，许多这些问题已经得到了解决，Vue 3引入了更快的响应式系统和更多的优化，以解决上述问题。因此，如果你在新项目中使用Vue，建议考虑使用Vue 3以获得更好的性能和开发体验。

【中等】Vue3 中为什么删除了过滤器？

在 Vue 2 中，过滤器是一种用于处理模板中文本格式化的功能。

然而，在 Vue 3 中，过滤器被删除，这是因为过滤器的功能可以用更强大的方式来实现，而且存在一些设计和性能方面的问题，因此它们被认为不再是 Vue 核心功能的一部分。

以下是一些关于为什么删除过滤器的原因：

  • 可复用性和维护性：Vue 2 的过滤器虽然能够格式化文本，但它们在组件之间的共享和复用性方面有一些限制。通过删除过滤器，Vue 3 鼓励开发者使用函数或计算属性等更灵活的方式来实现相同的功能，这有助于更好地组织和维护代码。
  • 性能问题：在 Vue 2 中，过滤器会在每次渲染时都被调用，这可能导致性能问题，尤其是当过滤器被频繁使用时。通过使用计算属性或函数，你可以更精确地控制何时触发计算，避免不必要的计算。
  • 可读性和维护性：过滤器的语法可能会导致模板变得复杂和难以理解，特别是在具有多个嵌套过滤器的情况下。通过将逻辑移到组件的计算属性或方法中，可以提高代码的可读性和维护性。
  • 更灵活的实现：通过使用函数或计算属性，你可以实现比过滤器更复杂的逻辑，包括异步操作和动态参数传递。这使得实现更灵活，能够满足更广泛的需求。

尽管 Vue 3 中删除了过滤器，但仍然可以在项目中使用类似的功能，只需使用计算属性、方法或函数来处理文本格式化的需求。这种做法不仅更强大和灵活，还有助于提高代码的可维护性和性能。如果你之前使用过过滤器，可以考虑将其替换为计算属性或方法，以适应 Vue 3 的新特性。


【中等】Vue3 使用了什么替换的过滤器？

Vue 3 中，过滤器被计算属性（Computed Properties）、方法（Methods）、函数、全局属性（重点）来替代。这些替代方案提供了更灵活的方式来处理文本格式化和数据转换的需求。以下是如何使用这些替代方案来实现与过滤器类似的功能：

PS：方法一般指的是对象的属性，而函数一般指的是一个单独的 function 函数。

例如:

const obj = { say: () => { console.log('hello') } }这就是方法

function say () { console.log('hello') }这就是函数


  • 计算属性（Computed Properties）：
  • 计算属性允许你定义在模板中可以直接使用的计算属性。它们会自动根据它们依赖的数据属性进行更新，类似于过滤器的自动更新。
  • 方法（Methods）（在 vue 中，这种操作一般被称之为方法，但是有些时候也会被叫做函数）：
  • 方法允许你在模板中调用函数来处理数据。方法与计算属性不同，它们不会缓存结果，每次调用都会执行函数。
  • 全局属性
  • 全局属性是一个特别好的替换过滤器的方式，他可以在一处注册，全局进行使用

这些替代方案提供了更灵活的方法来处理数据的格式化和转换需求，而不像过滤器那样受限于特定的语法和自动更新机制。

根据项目的需求，你可以选择使用计算属性、方法、函数、全局属性，以满足不同的场景。这样的灵活性有助于提高代码的可维护性和可读性。


【困难】vuex和pinia的区别，为什么要设计pinia

这是挺有意思的问题，很多同学可能只会用 pinia，对这个问题还真不太清楚。

所以大部分的回答就是：pinia 支持 compositionAPI 了，并且没有 mutation 了。

但是这样的回答是 完全八股文化，是很难得到大厂面试官认可的。

那么，想要回答好这个问题，就首先需要知道为什么 vuex 要设计 mutation，为什么 pinia 就不需要 mutation 了？

针对 vuex

针对于 vuex 很多同学都知道，它必须要通过 state 表示数据，通过 mutation 修改数据，通过 action 处理异步。

那么可以在 vuex 里面直接通过 state 修改数据吗？比如：store.state.count += 1。

答案是可以的！ 这样修改的数据依然具备响应式。只不过这样做 不推荐 而已。

那么为什么不推荐呢？其实，必须要使用 mutation 修改数据的原因是因为：所有的数据变化必须是有迹可循（在 VueDevTools 中可以被跟踪）的。

以此为目标，如果直接修改了 state 则 VueDevTools 无法跟踪对应的变化。

而 mutation 仅支持同步的操作，原因也与此有关。如果 mutation 的操作变成了异步的，则在 VueDevTools 中呈现 mutation 时，数据可能尚未发生变化。这就会导致 VueDevTools 呈现不准确。

因此，综合来说：之所以设计 mutation 的概念，本质上是为了 可以在VueDevTools中监听数据变化。

针对 pinia

那么，接下来咱们说pinia。

pinia 取消了 mutation 的概念，原因就是因为：pinia 直接修改 state，在 vueDevTools 中依然可以监听到变化。所以，mutation 的存在就没有必要了。

那么为什么直接修改也可以监听了呢？因为 pinia 的响应式依赖了 Vue3 的 reactive 和 ref 响应式系统，Proxy 直接实现对所有属性的深度监听。所以 直接修改 state（如 this.count++）也能被 Devtools 正确追踪 了


【中等】v-if 和 v-show 的区别？在什么情况下使用 v-if，什么情况使用 v-show

Vue官方中对v-if和v-show的使用场景区分是：当需要频繁显示和隐藏的时候，那么需要使用v-show否则需要使用v-if。

但是在实际的开发中我们是无法分辨一个场景（Dialog）是否是频繁切换的。

假如：有一个《添加用户》的按钮，点击这个按钮，弹出一个 dialog 完成添加用户的操作。

那么当前的操作是频繁的还是不频繁的呢？

这是无法进行判断的。

所以说，在实际开发中，判断v-if和v-show的使用需要根据它们的特性来进行判断。

v-show不会导致组件被销毁和渲染，但是v-if会导致组件被销毁和重新渲染。

也就是说：当使用v-show时，虽然用户看不到组件，但是当前组件的生命周期都已经执行了。如果在created中存在接口的调用，则【接口已经调用完成】。同时，当组件隐藏时，组件并不会销毁。相反：如果是v-if的每次点击时都会重新渲染组件，重新执行生命周期，隐藏则销毁组件。

所以说，判断v-if和v-show的使用条件应该是：当组件需要在指定时机创建，在指定时机销毁时，需要使用v-if。而 当组件仅需要创建一次时，则可以使用v-show。


【困难】v-if 和 v-show 的底层代码是如何实现的？

这个解释可以分为简单解释和复杂解释两种。

简单解释

要从 Vue.js 的源码层面分析v-if和v-show的实现，需要深入了解 Vue.js 的虚拟DOM和渲染过程。以下是对这两个指令的实现进行简要的源码层面的分析：

v-if的源码实现：

  • 当解析模板时，遇到v-if指令，Vue.js会在虚拟DOM中创建一个条件节点，这个条件节点包含了条件表达式和一个分支节点。
  • 当组件渲染时，Vue.js会评估v-if表达式的值。如果条件为true，则会渲染分支节点，否则忽略。
  • 如果条件为true，分支节点会被渲染为真实DOM并插入到页面中。如果条件为false，则分支节点不会被渲染，相应的DOM元素也不会出现在页面上。
  • 当条件从true切换到false或从false切换到true时，Vue.js会管理虚拟DOM的更新，销毁或创建相应的组件实例，以确保正确的DOM操作。

v-show的源码实现：

  • 当解析模板时，遇到v-show指令，Vue.js会在虚拟DOM中创建一个普通节点。
  • 在组件渲染时，无论v-show表达式的值是true还是false，普通节点都会被渲染为真实DOM。
  • 如果v-show的表达式为true，普通节点渲染为真实DOM后，Vue.js会将其显示，即将其CSS的display属性设置为合适的值（通常是block）。
  • 如果v-show的表达式为false，普通节点渲染为真实DOM后，Vue.js会将其隐藏，即将其CSS的display属性设置为none，从而使其不可见。

需要注意的是，v-show的元素在DOM中保持不变，只是通过CSS来。

【中等】v-for 循环为什么要绑定 key ，它的作用是什么？

在 Vue.js 中，使用v-for指令进行循环渲染时，通常建议绑定一个唯一的key属性到循环的元素上。

key的作用是帮助 Vue 识别每个循环项，以便在数据变化时更高效地更新虚拟 DOM 和实际 DOM。

下面是key的作用和原因：

  • 识别元素：key用于标识每个循环生成的元素。当数据发生变化时，Vue 使用key来判断哪些元素是新添加的、哪些是删除的，以便进行高效的更新。
  • 优化性能：如果没有指定key，Vue 可能会难以识别元素的唯一性，从而导致一些性能问题。有时，删除或更新元素可能会变得低效，因为 Vue 需要进行全面的比较。
  • 保持状态：如果没有key，在某些情况下可能会导致不正确的状态保持。特别是在表单元素等需要保持用户输入状态的情况下，key可以确保正确的状态被维护。
  • 处理动态数据：当数据中的元素顺序变化时，key可以帮助 Vue 正确地识别元素的位置和变化，从而避免渲染错误。

一般来说，key应该是具有唯一性的标识符，可以是元素的 ID、索引或其他唯一值。不推荐使用随机数或时间戳作为key，因为这可能会导致不稳定的行为，尤其在数据变化频繁的情况下。

【困难】VueLoader 是怎么处理 vue 文件的？

vue-loader是一个用于处理 Vue.js 单文件组件（.vue 文件）的 Webpack 加载器。它允许你在一个单个文件中编写 Vue 组件，包括模板、脚本和样式，然后将其转换为可在浏览器中运行的 JavaScript。

以下是vue-loader处理 Vue 单文件组件的主要步骤：

  • 解析单文件组件：vue-loader从入口文件中导入或引入 Vue 单文件组件。这个文件通常以.vue为扩展名，包含了模板、脚本和样式。
  • 分离模板、脚本和样式：vue-loader会分离单文件组件中的模板、脚本和样式部分。这允许你在一个文件中以更组织良好的方式编写 Vue 组件。
  • 处理模板：模板部分通常包括 Vue 的模板语法，vue-loader使用 Vue 模板编译器将其编译为渲染函数。这个渲染函数会在运行时将组件的虚拟 DOM 渲染到实际 DOM 上。
  • 处理脚本：脚本部分是 Vue 组件的逻辑，通常包括组件的数据、方法等。vue-loader可以将这部分代码用 Babel 等工具进行转译，以确保它在浏览器中可以执行。你也可以使用 TypeScript 来编写脚本。
  • 处理样式：vue-loader支持在单文件组件中使用各种样式语言，如 CSS、SCSS、Less 等。它可以将样式提取出来，同时允许你在组件中使用 CSS 模块化或预处理器特性。
  • 生成组件定义：一旦模板、脚本和样式都经过处理，vue-loader将它们合并为一个 Vue 组件定义对象，其中包含编译后的渲染函数、组件选项等。
  • 导出组件定义：最后，vue-loader导出 Vue 组件定义，以便在你的应用程序中使用。这个组件定义可以被导入到其他 Vue 文件或应用程序中，并通过 Vue 的组件系统注册和使用。

通过vue-loader的处理，你可以在一个文件中编写完整的 Vue 单文件组件，这提高了代码的可维护性和可读性，并使代码组织更加清晰。


【简单】reactive 为什么解构之后就会失去响应性

如果把 proxy 解构之后，就会得到一个基本数据（简单或者复杂数据类型），这样就无法监听 getter 和 setter 的行为。所以就会失去向影响。

【简单】ref 为什么必须要 .value？

因为 ref 方法返回的是一个 RefImpl 类型的实例对象。它在监听响应式的时候，主要依靠get value(){}和set value(){}这两个方法进行监听getter和setter的行为。

所以，当.value的时候，本质上触发的就是value方法。只有触发了value方法才可以监听到getter和setter。

【困难】结合源码完整的说一下 ref 的实现原理

在 Vue 3.2 中，ref的实现原理主要依赖于类封装、getter/setter、依赖追踪以及依赖触发更新。这些机制结合使用，使得ref的响应式系统在操作简单类型时更加轻量高效，并支持模板中的自动解包特性。以下是对ref实现原理的详细分析。

1.ref的创建入口

Vue 的ref函数用于将一个基本类型或对象包装成响应式数据。ref的核心实现逻辑在createRef函数中。

function ref(value) {
  return createRef(value, false)
}
  • ref函数接受一个value参数（可以是基本类型或对象）并调用createRef。
  • createRef会检查value是否已经是一个ref，如果是，则直接返回，否则创建新的RefImpl实例。

2.RefImpl类的构造

RefImpl是 Vue 中ref的核心实现类，它通过对value的getter和setter方法来实现响应式。

class RefImpl {
  private _value: any // 响应式值
  private _rawValue: any // 原始值（用于比较）
  public readonly __v_isRef = true // 标记为 ref 类型

  constructor(value, public readonly __v_isShallow: boolean) {
    this._rawValue = value
    // 如果是浅层（shallow）ref，则直接赋值，否则将值转换为响应式对象
    this._value = __v_isShallow ? value : toReactive(value)
  }

  get value() {
    // 依赖追踪：在访问 .value 时进行依赖收集
    trackRefValue(this)
    return this._value
  }

  set value(newVal) {
    // 判断新旧值是否不同，避免不必要的更新
    if (hasChanged(newVal, this._rawValue)) {
      this._rawValue = newVal
      this._value = this.__v_isShallow ? newVal : toReactive(newVal)
      // 触发依赖更新
      triggerRefValue(this)
    }
  }
}

细节分析

  • __v_isRef：这是一个内部标记，标识当前对象是ref类型，Vue 会利用这个标记进行解包处理。
  • _rawValue和_value：_rawValue存储的是传入的原始值，而_value是响应式后的值。Vue 通过_rawValue和_value的分离，能够在setter中判断新旧值是否发生变化。
  • 深浅响应：__v_isShallow控制了ref的深浅响应。如果为true，则ref为浅层响应式，不递归处理内部对象；否则，将value转为响应式对象，实现深层响应。

3. 依赖追踪与更新触发

在getter和setter中，Vue 利用依赖追踪（trackRefValue）和更新触发（triggerRefValue）来维护响应式关系。

依赖追踪：trackRefValue

当访问.value时，Vue 会调用trackRefValue来进行依赖收集。

function trackRefValue(ref) {
  if (shouldTrack && activeEffect) {
    trackEffects(ref.dep || (ref.dep = createDep()))
  }
}
  • 依赖收集：trackRefValue将当前活动的依赖（即使用ref.value的地方，如组件或计算属性）记录在ref.dep中。
  • dep集合：dep是一个依赖集合，存储所有依赖该ref的函数或组件，当ref更新时，dep中的依赖会被触发更新。

触发更新：triggerRefValue

在设置.value时，如果新值和旧值不同，Vue 会调用triggerRefValue，通知所有依赖项进行更新。

function triggerRefValue(ref) {
  if (ref.dep) {
    triggerEffects(ref.dep)
  }
}
  • 依赖更新触发：triggerRefValue会遍历dep集合，将所有依赖于该ref的副作用函数执行一次，从而实现依赖的自动更新。

4. 辅助函数

在RefImpl的实现中，Vue 使用了几个辅助函数来优化ref的响应式处理。

hasChanged：比较新旧值

Vue 在setter中通过hasChanged函数来比较新旧值，防止不必要的更新。

function hasChanged(value, oldValue) {
  return !Object.is(value, oldValue)
}
  • 值比较：hasChanged使用Object.is来判断新旧值是否相等，从而避免不必要的触发更新。

toReactive：深度响应式转换

当ref的值是一个对象时，toReactive会将其转为深层响应式对象。

function toReactive(value) {
  return isObject(value) ? reactive(value) : value
}
  • 深度代理：如果value是对象，Vue 会将其转为响应式对象，使用reactiveAPI 实现深层响应。
  • 浅层响应：如果ref是浅层响应式（即shallowRef），则直接返回对象，不做响应式转换。

5. 自动解包机制

Vue 在模板中访问ref时，会自动解包.value，不需要显式调用。这依赖于__v_isRef标记，Vue 渲染时检查这个标记，从而实现自动解包。

总结

Vue 3.2 中的ref实现依赖于RefImpl类的getter和setter，结合trackRefValue和triggerRefValue进行依赖追踪和触发更新。通过标记和自动解包，ref实现了对基本类型和复杂对象的高效响应式支持，使 Vue 的响应式系统更加灵活和简洁。

【困难】React 和 Vue 数据驱动的概念差异

React 和 Vue 都是主流的前端框架，它们都基于数据驱动的理念，即当数据发生变化时，视图自动更新。但两者在数据驱动的实现方式和概念上存在一些差异。下面详细介绍它们在数据驱动概念上的不同。

1. 数据驱动概念的核心思想

  • React：React 的数据驱动依赖于不可变数据和单向数据流的思想。React 组件的状态（state）是通过不可变的数据更新触发重新渲染。组件的更新是通过“重新渲染整个组件树”，而不是直接操作 DOM 元素。
  • Vue：Vue 则采用响应式数据系统，通过Proxy（Vue 3）或Object.defineProperty（Vue 2）来追踪数据的变化，自动更新视图。Vue 的数据驱动模型更接近于“声明式”编程，开发者只需要修改数据，Vue 的响应式系统会自动处理 DOM 更新。

2. 数据变化的追踪机制

React: 手动触发更新

  • 在 React 中，数据变化通过显式地更新 state来驱动视图更新。例如，使用this.setState或useState钩子来改变组件的状态，然后 React 会根据更新后的状态重新渲染组件。
  • 不可变数据：React 强调数据不可变，任何对数据的修改都需要返回一个新的对象，以便 React 识别状态是否发生变化。React 通过浅比较来判断组件是否需要更新。

React: 单向数据流

  • React 的数据流是单向的，即数据从父组件流向子组件，通过 props 进行传递。子组件不能直接修改父组件的数据，而是通过回调函数让父组件更新状态，这种设计简化了数据流的管理，特别是在复杂应用中，保证了数据流的可控性和可预测性。
  • 不可变性：React 提倡数据不可变，组件状态或 props 发生变化时，会创建新的对象，并通过setState或useState重新渲染组件。

Vue: 双向数据绑定和单向数据流

  • Vue 提供了类似 React 的单向数据流，通过 props 将数据从父组件传递到子组件。子组件可以发出事件（如$emit）通知父组件进行数据更新。

4. 数据更新与性能优化

React: 虚拟 DOM + Reconciliation

  • React 使用虚拟 DOM来管理视图的更新。当组件的状态或 props 发生变化时，React 会重新渲染虚拟 DOM 树，并将其与之前的虚拟 DOM 进行对比（即 diff 过程）。React 通过 diff 算法找出变化的部分，并将这些部分应用到真实 DOM 中。
  • 不可变数据：由于 React 使用不可变数据，React 可以通过浅比较来判断数据是否发生了变化，从而减少不必要的更新。

Vue: 响应式系统 + 模板编译优化

  • Vue 的响应式系统可以自动追踪数据变化，并只对发生变化的数据进行更新。通过响应式系统，Vue 可以做到在数据层发生变化时，自动将变化反映到视图层，而无需额外的 diff 比较。
  • 模板编译优化：Vue 在编译阶段对模板进行分析，提取出静态和动态部分，从而优化渲染时的性能。静态内容只渲染一次，而动态内容则通过响应式系统高效更新。

【困难】Vue 和 React 的区别（概述）

vue和react区别有以下 11 点：1、响应式原理不同；2、监听数据变化的实现原理不同；3、组件写法不同；4、Diff算法不同；5、核心思想不同；6、数据流不同；7、组合不同功能的方式不同；8、组件通信方法不同；9、模板渲染方式不同；10、渲染过程不同；11、框架本质不同。

1、响应式原理不同

vue：vue会遍历data数据对象，使用Object.definedProperty()将每个属性都转换为getter和setter，每个Vue组件实例都有一个对应的watcher实例，在组件初次渲染的时候会记录组件用到了那些数据，当数据发生改变的时候，会触发setter方法，并通知所有依赖这个数据的watcher实例调用update方法去触发组件的compile渲染方法，进行渲染数据。

react：React主要是通过setState()方法来更新状态，状态更新之后，组件也会重新渲染。

2、监听数据变化的实现原理不同

vue：Vue通过getter/setter以及一些函数的劫持，能精确知道数据变化。

react：React默认是通过比较引用的方式（diff）进行的，如果不优化可能导致大量不必要的VDOM的重新渲染。为什么React不精确监听数据变化呢？这是因为Vue和React设计理念上的区别，Vue使用的是可变数据，而React更强调数据的不可变，两者没有好坏之分。

3、组件写法不同

vue：Vue的组件写法是通过template的单文件组件格式。

react：React的组件写法是JSX+inline style，也就是吧HTML和CSS全部写进JavaScript中。

4、Diff算法不同

vue和react的diff算法都是进行同层次的比较，主要有以下两点不同：

  • vue对比节点，如果节点元素类型相同，但是className不同，认为是不同类型的元素，会进行删除重建，但是react则会认为是同类型的节点，只会修改节点属性。
  • vue的列表比对采用的是首尾指针法，而react采用的是从左到右依次比对的方式，当一个集合只是把最后一个节点移动到了第一个，react会把前面的节点依次移动，而vue只会把最后一个节点移动到最后一个，从这点上来说vue的对比方式更加高效。

5、核心思想不同

vue：Vue的核心思想是尽可能的降低前端开发的门槛，是一个灵活易用的渐进式双向绑定的MVVM框架。

react：React的核心思想是声明式渲染和组件化、单向数据流，React既不属于MVC也不属于MVVM架构。

6、数据流不同

vue：Vue1.0中可以实现两种双向绑定：父子组件之间，props可以双向绑定；组件与DOM之间可以通过v-model双向绑定。Vue2.x中去掉了第一种，也就是父子组件之间不能双向绑定了（但是提供了一个语法糖自动帮你通过事件的方式修改），并且Vue2.x已经不鼓励组件对自己的 props进行任何修改了。

react：React一直不支持双向绑定，提倡的是单向数据流，称之为onChange/setState()模式。不过由于我们一般都会用Vuex以及Redux等单向数据流的状态管理框架，因此很多时候我们感受不到这一点的区别了。

7、组合不同功能的方式不同

vue：Vue组合不同功能的方式是通过mixin，Vue中组件是一个被包装的函数，并不简单的就是我们定义组件的时候传入的对象或者函数。比如我们定义的模板怎么被编译的？比如声明的props怎么接收到的？这些都是vue创建组件实例的时候隐式干的事。由于vue默默帮我们做了这么多事，所以我们自己如果直接把组件的声明包装一下，返回一个HoC，那么这个被包装的组件就无法正常工作了。

react：React组合不同功能的方式是通过HoC(高阶组件）。React最早也是使用mixins的，不过后来他们觉得这种方式对组件侵入太强会导致很多问题，就弃用了mixinx转而使用HoC。高阶组件本质就是高阶函数，React的组件是一个纯粹的函数，所以高阶函数对React来说非常简单。

8、组件通信方法不同

vue：Vue中有三种方式可以实现组件通信：父组件通过props向子组件传递数据或者回调，虽然可以传递回调，但是我们一般只传数据；子组件通过事件向父组件发送消息；通过V2.2.0中新增的provide/inject来实现父组件向子组件注入数据，可以跨越多个层级。

react：React中也有对应的三种方式：父组件通过props可以向子组件传递数据或者回调；可以通过 context 进行跨层级的通信，这其实和 provide/inject 起到的作用差不多。React 本身并不支持自定义事件，而Vue中子组件向父组件传递消息有两种方式：事件和回调函数，但Vue更倾向于使用事件。在React中我们都是使用回调函数的，这可能是他们二者最大的区别。

9、模板渲染方式不同

vue：Vue是在和组件JS代码分离的单独的模板中，通过指令来实现的，比如条件语句就需要 v-if 来实现对这一点，这样的做法显得有些独特，会把HTML弄得很乱。

react：React是在组件JS代码中，通过原生JS实现模板中的常见语法，比如插值，条件，循环等，都是通过JS语法实现的，更加纯粹更加原生。

10、渲染过程不同

vue：Vue可以更快地计算出Virtual DOM的差异，这是由于它在渲染过程中，会跟踪每一个组件的依赖关系，不需要重新渲染整个组件树。

react：React在应用的状态被改变时，全部子组件都会重新渲染。通过shouldComponentUpdate这个生命周期方法可以进行控制，但Vue将此视为默认的优化。

11、框架本质不同

vue：Vue本质是MVVM框架，由MVC发展而来；

react：React是前端组件化框架，由后端组件化发展而来。
