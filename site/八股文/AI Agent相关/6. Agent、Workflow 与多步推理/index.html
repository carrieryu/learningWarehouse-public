【简单】什么是 Agent？它和普通聊天机器人有什么区别？

我的理解是，Agent 不是简单“会聊天”的机器人，而是一个能围绕目标主动完成任务的 AI 系统。

普通聊天机器人更像是“你问一句，它答一句”。它主要做的是文本生成，比如回答问题、解释概念、写文案、总结内容。用户不给下一步指令，它一般不会自己继续推进任务。

但 Agent 不一样。Agent 更强调“目标驱动”。比如用户说：“帮我查一下最近的订单异常，并生成一份处理建议。”普通聊天机器人可能只能给你一个分析思路；但 Agent 会进一步拆任务：先判断需要查哪些数据，再调用订单接口、日志系统或者知识库，然后根据结果做分析，最后输出结论，甚至还能触发工单、发消息通知相关人员。

所以我觉得两者最大的区别不是模型本身，而是系统能力边界不同。聊天机器人主要是“对话生成”；Agent 是“理解目标 + 拆解任务 + 调用工具 + 观察结果 + 继续决策”。

但 Agent 也不是越复杂越好。真正落地时我不会一上来就把所有功能都做成 Agent，因为它会带来更多不可控性，比如工具误调用、循环执行、权限越界、成本不可控、结果不可追踪。所以生产环境里一般要给 Agent 加边界：哪些工具能调、最多执行几步、失败怎么兜底、关键操作是否需要人工确认、每一步日志是否可追溯。

如果是面试里让我总结，我会说：普通聊天机器人解决的是“怎么回答用户”，Agent 解决的是“怎么帮用户完成任务”。它的核心价值在于把大模型从一个回答器，扩展成一个可以连接外部工具、执行真实业务流程的智能执行系统。

【中等】什么是 Workflow？它和 Agent 的边界在哪里？

我的理解是，Workflow 和 Agent 的核心区别不在于有没有用大模型，而在于“决策权”放在哪里。

Workflow 更像是我们提前设计好的一条执行链路。比如用户提问之后，先做意图识别，再走知识库检索，再重排，再拼 Prompt，再调用模型，最后做敏感词和格式校验。这里每一步基本是工程师预先定义好的，大模型可能参与某个节点，但整体流程是可控的、可观测的、可回滚的。

Agent 更像是把一部分决策权交给模型。比如它自己判断下一步要不要查资料、要不要调用工具、要不要继续追问、要不要拆任务。也就是说，Workflow 是“人设计流程，模型执行局部能力”；Agent 是“人给目标和边界，模型参与规划和决策”。

但真实项目里我不会把两者切得特别绝对。很多线上系统其实是 Workflow 里嵌了一点 Agent 能力。比如主链路还是固定的，但在某个复杂任务节点，让模型动态选择工具或者生成执行计划。这样比纯 Agent 更稳，也比纯固定流程更灵活。

这里的取舍很明显。Workflow 的优点是稳定、成本可控、排障容易，适合客服问答、审核、报告生成这类流程明确的场景。但它的问题是灵活性有限，遇到开放任务或者多步骤不确定任务，就容易写很多分支逻辑。Agent 的优点是能处理更开放的问题，比如数据分析、复杂工具调用、自动化排查，但代价是不可控性更高，延迟、成本、失败路径和安全风险都会放大。

所以我一般会按任务不确定性来划边界。如果任务路径清楚、步骤稳定、线上 SLA 要求高，我优先用 Workflow；如果任务目标明确但路径不确定，需要模型自己规划和选择工具，我才会引入 Agent。而且即使用 Agent，我也不会让它无限自由执行，会加工具白名单、权限控制、步数上限、日志追踪、结果校验和失败兜底。

我的原则是：能用 Workflow 稳定解决的，不要为了炫技上 Agent；只有当固定流程覆盖不了复杂决策时，才把有限决策权交给 Agent。

【中等】什么场景适合固定流程，什么场景适合 Agent 自主决策？

固定流程和 Agent 的分界，我会放在“执行路径是否确定”和“失败代价是否可控”这两个点上。

如果一个任务的步骤基本稳定，输入输出也比较明确，我更倾向于固定流程。比如知识库问答里的召回、重排、生成、审核，或者工单分类、合同字段抽取、内容安全审核，这些场景最重要的不是让模型自由发挥，而是结果稳定、问题可定位、成本可预期。固定流程的好处是每个节点都能监控，出了问题可以拆开看：是召回没命中，还是重排排错，还是生成阶段幻觉了。它的代价是灵活性差，复杂任务会写出很多分支。

Agent 自主决策更适合目标明确但路径不确定的任务。比如数据分析、线上问题排查、跨系统自动化处理，模型需要自己判断下一步是查日志、查数据库、调接口，还是继续追问用户。这类任务如果强行写成固定流程，规则会越来越复杂，而且覆盖不了长尾情况。Agent 的价值就在于让模型参与规划和工具选择。

但 Agent 不能等于“完全放权”。它一旦进入线上，就会带来循环调用、工具选错、延迟变长、成本失控、权限越界这些问题。所以我会把 Agent 放在受控边界里：工具白名单、最大步数、超时限制、权限校验、关键写操作人工确认、全链路日志和结果校验都要有。尤其涉及改数据、发消息、下单、审批这类动作，不能让它无确认执行。

所以我的判断是：路径清楚、风险高、要求一致性，就用固定流程；路径开放、规则覆盖成本高、需要动态规划，才引入 Agent。更稳的落地方式通常是固定 Workflow 作为主链路，Agent 只在局部复杂节点里做受控决策。


【中等】什么是 A2A ？ 单 Agent 和 A2A 架构分别适合什么业务？

这里我会把 A2A 理解成 Agent-to-Agent，也就是多个 Agent 之间通过协议或消息机制协作，而不是所有事情都塞给一个 Agent 做。

单 Agent 更适合目标相对集中、工具数量有限、任务链路不太长的业务。比如一个知识库助手、一个数据分析助手、一个客服质检助手，它可能会调用检索、数据库、报表这些工具，但整体还是围绕一个明确目标工作。单 Agent 的优势是链路简单，状态管理容易，日志也好追，出了问题比较容易判断是规划错了、工具错了，还是模型输出错了。对大多数业务来说，先把单 Agent 做稳，比一开始就拆成多 Agent 更现实。

A2A 更适合任务天然需要多角色协作的场景。比如一个企业级办公 Agent，要同时处理销售线索、合同审核、财务报销、日程协调，不同子任务背后的系统、权限、知识和规则都不一样。再比如复杂研发助手，可能需要需求分析 Agent、代码 Agent、测试 Agent、安全审查 Agent 协同。这里如果只用一个 Agent，会出现工具太多、上下文太长、权限边界混乱、决策责任不清的问题。拆成多个 Agent 后，每个 Agent 负责一个专业域，职责更清楚，也更容易做权限隔离和独立评估。

但 A2A 的代价也很明显。多个 Agent 一协作，就会有通信成本、状态同步、任务拆分错误、互相甩锅、循环调用这些问题。线上排障也会更复杂，因为一次失败可能不是某个 Agent 单点失败，而是任务交接、上下文传递或者协议约定出了问题。所以我不会为了“架构高级”直接上 A2A。只有当单 Agent 已经明显遇到边界，比如工具数量膨胀、权限隔离做不清、上下文塞不下、不同领域评估标准冲突，我才会考虑拆成 A2A。

我的判断原则是：业务目标单一、链路可控，用单 Agent；业务跨多个专业域、权限边界清楚、需要角色协作，才上 A2A。A2A 不是为了让系统看起来更智能，而是为了把复杂性拆到可管理的责任边界里。

【困难】A2A 系统里最常见的问题有哪些？

A2A 最常见的问题，不是“多个 Agent 能不能聊起来”，而是聊起来之后责任、状态和结果谁来保证。

  • 第一个问题是任务拆分不稳定。

一个复杂目标交给协调 Agent，它可能拆得太粗，也可能拆得太碎。拆得太粗，子 Agent 还是处理不了；拆得太碎，通信成本和延迟会被放大。这个问题在 Demo 里不明显，因为样例任务通常比较干净，但线上用户表达会很乱，任务边界经常不清楚。所以我会要求拆分结果可观测，至少要能看到任务树、每个子任务输入、输出和失败原因。

  • 第二个问题是上下文传递丢失。

A2A 不是把完整上下文复制给每个 Agent 就完了，那样成本高，也容易泄露权限。更常见的是只传摘要或结构化状态，但摘要一旦丢关键约束，后面的 Agent 就会做错。比如合同审核 Agent 忽略了用户所在地区，财务 Agent 忽略了金额上限，结果就会偏。所以这里要设计共享状态和上下文契约，哪些字段必须传、哪些字段不能传、哪些字段需要引用原始证据，都要明确。

  • 第三个问题是权限边界容易混乱。

A2A 里不同 Agent 能调的工具不一样，如果协调 Agent 可以随便把任务转给有高权限的 Agent，就可能形成变相越权。我的做法是权限跟任务和用户身份绑定，不是只跟 Agent 名字绑定。涉及写操作、审批、发消息、改数据，要有二次确认、审计日志和可回滚机制。

  • 还有一个很现实的问题是失败归因困难。

单 Agent 出错，通常还能看规划、工具调用、输出；A2A 出错，可能是拆分错、路由错、子 Agent 理解错、上下文传错，或者多个 Agent 互相强化了错误结论。所以必须有全链路 trace，把每次 Agent 间消息、工具调用、状态变更、最终输出都串起来，否则线上排障会很痛苦。

  • 最后是成本和延迟失控。

多个 Agent 互相调用，很容易出现循环、重复分析、过度验证。这里不能只靠 Prompt 约束，要有最大轮次、预算上限、超时、去重、缓存和降级策略。我的原则是：A2A 不是把复杂性消灭了，而是把复杂性分布到了多个角色之间，所以系统必须有清晰的责任边界、状态契约、权限控制和可观测性。

【中等】为什么 Agent 系统通常要引入状态管理？

Agent 需要状态管理，核心原因是它不是一次简单问答，而是在多轮决策里持续推进一个任务。

单次 LLM 调用可以只看当前 Prompt，但 Agent 往往要经历计划、工具调用、观察结果、再决策、再调用工具这样的循环。如果没有状态管理，每一步只能靠上下文里堆文本，时间一长就会出现几个问题：任务目标丢了、约束条件忘了、工具结果混在一起、失败重试不知道从哪里恢复，最后系统表现就会不稳定。

状态管理要解决的不是“记忆越多越好”，而是把任务推进过程中关键的信息结构化保存下来。比如当前目标是什么、已经执行到哪一步、调用过哪些工具、每次工具返回了什么、哪些假设已经被验证、哪些错误发生过、下一步候选动作是什么。这样做的好处是，Agent 的行为可以被追踪、可以恢复，也可以被评估。

真实项目里，这个点很重要。比如一个数据分析 Agent，如果已经查过某张表、跑过某段 SQL，就不能下一轮又重复查一遍；如果某个接口失败了，要知道是重试、降级，还是结束任务。再比如一个办公 Agent，用户前面说了预算上限、时间范围、审批人，这些约束不能在后面发起操作时丢掉。

但状态管理也有代价。状态太少，Agent 会失忆；状态太多，成本会上升，还会把无关信息带进上下文，影响判断。更麻烦的是状态如果写错，后面所有决策都会被带偏。所以我会把状态分层：短期状态放当前任务上下文，结构化任务状态放数据库或工作流引擎，长期偏好和历史信息单独管理，并且对关键状态做版本、审计和回滚。

所以我的判断是，Agent 引入状态管理不是为了显得复杂，而是为了让多步任务可持续、可恢复、可观测。没有状态，Agent 很容易变成每一步都重新猜；有了状态，系统才有机会稳定地完成长任务。

【困难】你会如何设计一个可中断、可恢复的 Agent 工作流？

可中断、可恢复的 Agent，重点不是让模型一次跑完整个任务，而是把它设计成一个“每一步都能停下来、再接着跑”的任务系统。

我会先给每个用户任务生成一个 task_id，也就是这个任务的唯一编号。后面所有计划、工具调用、错误、审批、恢复，都挂在这个 task_id 下面。这样哪怕服务重启，或者用户中途暂停，系统也能根据这个编号把任务找回来。

任务内部我会拆成一个个 step，也就是可独立执行的小步骤。每个 step 都要有状态，比如 pending 表示还没开始，running 表示正在执行，success 表示已经完成，failed 表示执行失败，waiting_approval 表示正在等人工确认，cancelled 表示任务被用户取消。这样恢复的时候，系统不是重新猜，而是先看上一次停在哪个状态：已经成功的步骤不重复跑，失败的步骤看是否重试，等确认的步骤就继续等人确认。

工具调用这里要特别谨慎。查日志、查数据库这类读操作，失败后通常可以重试；但改订单、发消息、创建工单这种写操作，恢复时不能重复执行。所以我会给写操作加 idempotency key，也就是幂等键。它的作用是让外部系统知道“这其实是同一次操作”，即使恢复时又请求了一次，也不会重复创建两条记录。对于不可逆或者高风险动作，还要先进入 waiting_approval，让用户或审批人确认后再执行。

上下文恢复也不能简单把历史消息全塞回模型。更合理的做法是从状态里重建一个精简上下文：任务目标是什么，关键约束有哪些，已经完成了哪些步骤，工具返回了哪些关键结果，现在卡在哪里，下一步候选动作是什么。原始日志要保留，但主要用于排障回放，不应该每轮都塞进模型上下文，否则成本会上升，也会干扰模型判断。

可观测性也要从一开始设计进去。每个 step 的输入、输出、工具参数、模型决策、耗时、错误码、token 成本都要串到同一个 trace 里。trace 可以理解成一次任务的全链路记录，方便我们回放它每一步为什么这么做。否则 Agent 一旦恢复失败，很难判断是模型规划错了、工具超时了、权限不足，还是状态写坏了。

最后我会加运行边界，比如最大执行步数、最大重试次数、超时时间、预算上限，以及失败后的降级策略。因为可恢复不等于无限重试，更不是让 Agent 一直跑。我的原则是：Agent 工作流要靠持久化状态、清晰状态机、幂等工具调用和全链路 trace 来保证恢复，而不是靠模型临场记忆。

【中等】什么是 Planner、Executor、Critic 这类角色划分？有什么价值？

Planner、Executor、Critic 这类划分，我更愿意把它理解成把 Agent 的“想、做、检查”拆开，而不是让一个模型在同一段上下文里把所有事情都包了。

  • Planner 是规划者，负责把用户目标拆成步骤，比如先查资料、再调用工具、最后汇总答案。它重点解决“下一步做什么”的问题。
  • Executor 是执行者，负责把某个具体步骤做完，比如调用搜索、查数据库、跑代码、生成报告。它重点解决“这一步怎么落地”的问题。
  • Critic 是检查者，负责看结果有没有问题，比如有没有漏掉约束、有没有工具调用错误、有没有幻觉、输出是否符合要求。它重点解决“做完之后靠不靠谱”的问题。

这种拆分的价值在复杂任务里会比较明显。一个 Agent 如果既负责规划，又负责执行，还负责自我检查，很容易出现两个问题：

  • 一是上下文越来越乱
  • 二是自己很难发现自己的错误

拆成不同角色后，每个角色的目标更单一，Prompt 更容易写，日志也更清楚。

出了问题可以定位：是 Planner 拆错了，Executor 执行错了，还是 Critic 没拦住。

但这个架构不是越拆越好。

角色一多，就会增加模型调用次数，延迟和成本都会上升，还会带来角色之间信息传递的问题。

比如 Planner 的意图没有传清楚，Executor 就可能做偏；Critic 太保守，又可能让任务反复重做。

所以我会看任务复杂度来决定。如果只是普通问答或者简单工具调用，一个 Agent 内部完成就够了；如果是多步骤、高风险、需要校验的任务，比如代码修改、数据分析、自动化运维、合同审核，我才会考虑拆 Planner、Executor、Critic。

真正落地时，我会给三类角色清楚的输入输出约束。Planner 输出结构化计划，Executor 只能执行被授权的步骤，Critic 不直接改结果，而是给出问题、证据和是否通过。

这样系统更容易观测、评估和回滚。我的原则是：角色划分不是为了让架构看起来复杂，而是为了把复杂任务里的规划、执行和校验拆成可控的责任边界。

【困难】什么是 Human-in-the-loop？它适合插入在什么节点？

Human-in-the-loop 我会理解成“人在关键决策点参与闭环”，不是简单地把所有结果都丢给人审核。

它的价值主要是在 Agent 或大模型系统不够确定、但动作后果又比较重的时候，让人来做确认、修正或者兜底。比如模型生成一段内容，普通草稿可以直接给用户；但如果要发到公司官方账号，或者要触发支付、改订单、发合同、审批报销，这种动作就不能只靠模型自己判断。因为一旦错了，影响的是钱、权限、客户关系或者合规风险。

我会把人插在几个关键节点。第一个是高风险写操作之前。写操作就是会改变外部系统状态的动作，比如发消息、改数据库、提交审批、创建订单。这里适合让人确认“是否执行”，而不是事后才发现错了。

第二个是低置信度判断之后。比如检索证据不足、模型自己判断冲突、分类结果置信度低，或者 Critic 检查出答案可能有问题。Critic 可以理解成检查者，它负责看输出有没有漏约束、有没有幻觉、有没有不符合规则。这个时候让人介入，比让 Agent 继续猜更可靠。

第三个是权限升级或越权风险出现时。比如用户原本只允许查数据，但 Agent 发现要完成任务需要改数据、访问敏感字段、调用更高权限工具，这时必须让人授权。权限不能因为任务看起来合理就自动放大。

第四个是失败重试到达边界时。比如 Agent 已经重试几次仍然失败，或者任务状态进入 waiting_approval，也就是等人工确认，系统就应该停下来，把当前上下文、已做步骤、失败原因、推荐动作展示给人，而不是继续无限试错。

但 Human-in-the-loop 也有代价。人一介入，链路会变慢，自动化率会下降，用户体验也可能被打断。所以我不会把它放在每一步，而是放在失败代价高、模型不确定、权限敏感、动作不可逆的节点。低风险、可回滚、可自动验证的动作，尽量让系统自己完成。

真正落地时，人介入的界面也很重要。不能只弹一句“是否确认”，而要展示证据、风险、模型建议和可选动作，比如通过、拒绝、修改后通过、转人工处理。这样人不是替模型从头做一遍，而是在关键点做有效判断。

我的原则是：Human-in-the-loop 不是降低智能化，而是把人的判断放在最值钱、最能降低风险的位置。
