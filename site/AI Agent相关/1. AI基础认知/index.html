【简单】简述以下名词/概念的理解：MCP、A2A、Claude Code、Codex、Manus、AutoGLM、豆包手机、Cowork、OpenClaw/Clawdbot、Vibe Coding、Skills、WorkFlow

我用 Gemini 生成了一张说明图，大家可以作为参考（先看图，再看解释，更好理解）：



先把它们分成四类来看：

  • 协议
  • Agent 产品
  • AI 编程方式
  • Agent 的能力组织方式。

这样讲会更像你真的理解了它们之间的关系，而不是在背名词。

  • MCP，我的理解就是大模型接外部工具和数据的标准协议。你可以把它理解成 AI 世界里的通用插座。模型通过 MCP 去接文件、数据库、搜索、内部系统，不用每家都搞一套私有接法。
  • A2A，我会理解成Agent 和 Agent 之间通信协作的协议。MCP 解决的是“Agent 怎么连工具”，A2A 解决的是“Agent 怎么跟另一个 Agent 协作”。这个区分面试里挺关键。
  • Claude Code，本质上就是Anthropic 做的 coding agent。它不是单纯代码补全，而是能读整个仓库、改多个文件、跑命令、跑测试，偏“帮你完成开发任务”。
  • Codex，现在我会把它理解成OpenAI 的云端软件工程 Agent。它和 Claude Code 很像，都是 coding agent，但 Codex 更强调云端 sandbox、多任务并行、围绕代码仓库做实际交付。
  • Manus，我一般会把它看成结果交付型的通用 Agent。它重点不是陪聊，而是直接帮你产出 PPT、网页、报告、应用这些成品，所以它更像一个“替你做事”的 agent。
  • AutoGLM，我的理解是智谱在做的自主执行 Agent 能力和产品体系。一层是浏览器/手机操作这类 phone use、browser use 能力，另一层是把这些能力产品化，去做自动上网、自动执行、自动生成报告这类事情。
  • 豆包手机，它更像AI 原生手机助手的产品形态。核心点是把 AI 助手直接做进手机交互里，不只是 App，而是更接近“手机级助手”。
  • Cowork，我会理解成Anthropic 面向知识工作的桌面 Agent。Claude Code 偏研发，Cowork 偏非研发办公场景，比如整理资料、操作应用、处理文件、交付成品。
  • OpenClaw / Clawdbot，如果在面试里让我一句话说，我会说它是聊天入口驱动的个人 Agent / AI 助手形态。它强调通过 WhatsApp、Telegram、Slack 这类你原本就在用的入口，让 AI 不只是回答问题，而是真的去执行任务。现在官方公开名字更常见的是 OpenClaw。
  • Vibe Coding，这个词我会直接说是AI 时代的一种编程方式：人更多描述目标、感觉和结果，AI 负责大量生成和改代码。这个词最早是 Karpathy 提出来的。
  • Skills，我的理解是给 Agent 封装好的可复用能力包。里面可以有说明、模板、脚本、资源文件。它本质上是在沉淀“某件事怎么稳定做好”，让 Agent 下次能自动调用，而不是每次重新 prompt 一遍。
  • Workflow，我一般会说它是预先编排好的执行链路。比如先分类，再路由，再检索，再生成，再校验。它最大的特点是路径基本是提前定好的，不像 Agent 那样会动态决定下一步。


对照咱们在一开始的时候说的四类：

  • MCP、A2A 属于协议层
  • Claude Code、Codex、Manus、AutoGLM、Cowork、OpenClaw 属于 Agent/Product 层
  • Vibe Coding 是 AI 编程范式
  • Skills 和 Workflow 属于 Agent 能力组织与执行方式



【简单】什么是生成式 AI？它和传统机器学习系统最大的区别是什么？

生成式 AI，就是一类能够基于已有数据分布，生成新内容的 AI。它生成的内容可以是文本、图片、代码、音频、视频等。

比如：

  • ChatGPT 生成一段回答
  • 文生图模型生成一张图片
  • 代码模型生成一段函数代码

它和传统机器学习最大的区别在于：

传统机器学习更偏“判断”和“预测”，核心是把输入映射到一个确定结果上。比如：垃圾邮件分类、用户流失预测、推荐排序、房价预测，这类任务本质上是在做分类、回归或打分。

而生成式 AI 更偏“生成”，它不是只给一个标签或分数，而是能够输出一段新的内容。比如：用户问一个问题，它不是返回“是/否”，而是直接组织语言生成完整答案。

传统机器学习的重点是学会“怎么判断”，生成式 AI 的重点是学会“怎么生成”。

【简单】为什么推送大部分使用 SSE，而不是 WebSocket？

我会先说结论：如果业务主要是服务端单向推送，比如 AI 对话流式输出、消息通知、任务进度、日志流，这类场景我一般会优先选 SSE，而不是 WebSocket。原因不是 WebSocket 不行，而是SSE 在这种单向场景下更轻、更稳、接入成本更低。

我实际会从几个维度判断。

第一，通信方向不一样。

SSE 本质上就是服务端往客户端持续推数据，天然适合“我发起一次请求，后端不断吐结果给我”这种模式。比如大模型逐 token 返回、工作流执行日志、报表生成进度，这些本来就不需要前端持续反向发消息。但 WebSocket 是全双工，它更适合聊天、协同编辑、多人对战、实时控制这种“前后端都要频繁互发消息”的场景。很多推送业务其实根本用不到这个能力。

第二，工程接入更简单。

SSE 走的是标准 HTTP 连接，和现有的网关、鉴权、Cookie、Nginx、监控体系更容易融合。前端用EventSource或者基于fetch的流式读取就能接，后端实现也相对直接。WebSocket 虽然功能更强，但协议升级、连接管理、心跳保活、断线重连、会话清理这些都要多做一层，整体运维和排障成本会更高。

第三，稳定性和可观测性通常更好做。

SSE 因为还是 HTTP 语义，所以在日志追踪、请求链路、鉴权、超时控制这些方面更贴近常规 Web 服务。出了问题也比较容易定位，是网关断了、后端 flush（把当前已经写到缓冲区里的内容，立刻发出去）不及时，还是客户端重连了。WebSocket 一旦连接生命周期很长，排查问题会更偏“连接态管理”，复杂度会明显上去。

第四，对 AI 应用特别合适。

像 AI 聊天这种场景，本质就是：用户发一次问题，服务端持续返回中间 token、状态、引用片段、最终答案。这时候 SSE 非常贴合业务模型，而且天然支持“边生成边展示”，用户体感会更好。所以现在很多 AI 系统，尤其是 LLM Chat、Agent 执行日志、工作流状态流，默认都会先用 SSE。

当然，SSE 也不是万能的。

它的限制很明确：只能服务端推客户端，不能真正双向通信；对二进制支持也一般；如果是高频双向交互，WebSocket 更合适。

所以我的原则一般是：单向流式推送优先 SSE，双向实时互动优先 WebSocket。

一个注意点：

openai （chatGPT）现在支持Responses AP的 websocket 模式，open ai 的 文档

所以，如果在面试状态可以这么说：

“不过这个结论我会补一个边界。像 OpenAI 现在在 Responses API 里也提供了 WebSocket mode，这说明如果场景不是普通的单向流式输出，而是那种长链路、多轮 tool call、需要持续 continuation 的 Agent workflow，那 WebSocket 反而更合适。因为它优化的不是单纯的 token 推送，而是连续续跑的开销。所以更准确地说，普通推送场景我优先选 SSE，但复杂 Agent 编排链路我会重点评估 WebSocket。”

这个补充有两个好处：第一，它不会把你前面的回答推翻，而是把边界补完整。第二，面试官会觉得你不是在死记“推送=SSE”，而是真的理解了场景决定协议。、


【简单】SSE 如何实现稳定推送？会话中断如何恢复？

先说结论：SSE 的稳定性，核心不是“连上就行”，而是要同时做好两层设计：传输层稳定推送，业务层可恢复。

如果我来做，我一般会这么设计。

先说稳定推送。

  • 第一，连接本身要保活。SSE 是长连接，如果中间一直没数据，代理层、网关、浏览器都可能把它断掉，所以我一般会加 heartbeat（插入一个表示当前链接还是正常的消息），比如定时发注释行或者轻量事件，告诉链路“这个连接还活着”。
  • 第二，服务端要支持及时 flush（把当前已经写到缓冲区里的内容，立刻发出去）。不能等内容攒一大坨再一次性返回，不然用户看到的就不是流式了。我会在服务端按 chunk（chunk 的意思就是每次都是一小段内容） 或按 token 写入，然后立刻 推送过去，同时把 Nginx（这是一个代理服务器） 这类代理的 buffering（响应缓存） 关掉，不然上游在流，下游看起来还是卡住的。
  • 第三，客户端要有自动重连机制。SSE 原生就支持重连，但生产里我不会只依赖默认能力。我一般会加重连退避、重连次数控制，以及网络切换后的重新订阅，避免频繁抖动把服务打挂。
  • 第四，每条消息都要有可追踪的序号或者 event id（可以理解为每次请求的一个 id）。这个很关键。因为没有这个，你断了以后根本不知道自己收到哪了。我的做法通常是每个 chunk （chunk 的意思就是每次都是一小段内容）都带一个递增序列编号，或者直接用 SSE 的id字段。

然后说会话中断恢复。

可以分成两个层面去讲：

  • 第一个层面是传输恢复。客户端断线重连时，会把上次收到的Last-Event-ID或者本地保存的游标带回来，服务端根据这个游标，从未消费的那一段继续补发。这个适合日志流、任务进度流、消息通知流这种“事件可重放”的场景。
  • 第二个层面是业务恢复。像 AI 对话这种流式生成场景，很多时候不是简单补发几条消息就够了，因为模型生成可能已经结束了，或者底层流已经断了，未必能从某个 token 精准接着吐。所以我会把“恢复”设计成：恢复到最近一个稳定状态，而不是强行从中间 token 恢复。

比如我会持久化这些东西：

  • 会话 ID
  • 用户最后一次问题
  • 已经生成的文本片段
  • 当前任务状态，比如 running / done / failed（自定义的几种状态：进行中、结束、失败）
  • 如果有工具调用，还会存工具执行到哪一步

这样前端刷新或者断线后，可以先把已生成内容秒级恢复出来，用户不会看到一片空白；如果后台任务还在跑，就重新订阅继续接；如果后台任务已经丢了，就从最近一次稳定上下文重新发起生成，最多做到“从段落级恢复”，而不是“从 token 级恢复”。

我觉得这里面最重要的一点是：不要把“连接恢复”和“业务恢复”混为一谈。连接恢复解决的是流不要断了；业务恢复解决的是断了以后用户还能不能无感继续。

总结一句话就是：

SSE 稳定推送靠 heartbeat（插入一个表示当前链接还是正常的消息）、flush（把当前已经写到缓冲区里的内容，立刻发出去）、重连和消息序号；会话中断恢复要分传输层补发和业务层恢复，尤其 AI 场景里更现实的做法是恢复到最近稳定状态，而不是强求从某个 token 精准续传。

【简单】大语言模型为什么能“像人在说话一样”输出内容？

大语言模型之所以能“像人在说话一样”输出内容，本质上是因为它在训练过程中，学习了大量人类语言里的统计规律、语义关系和上下文关联。

最核心的一点是：它本质上是在做“根据前文预测下一个最可能出现的 token”。

比如一句话已经有了前半句，模型会根据上下文去判断，后面最合理、最自然、最符合语境的内容应该是什么。因为它见过海量文本，所以学到了很多语言表达模式、语法结构、知识关联和问答方式。

你可以把它理解成三层能力：

  • 第一层是语言模式学习。它学会了什么样的词经常一起出现，什么句式更自然，什么回答方式更像人类表达。
  • 第二层是上下文理解。它不是只看一个词，而是会结合前面的整段内容，判断当前语境下什么回答更合适。
  • 第三层是对齐训练。在预训练之后，模型通常还会经过指令微调、人工反馈对齐这些过程，所以它不只是会“续写文本”，还会更像一个助手一样去回答问题、解释概念、遵循要求。

所以面试里可以总结成一句话：

大语言模型看起来像人在说话，不是因为它真的理解了人类思维，而是因为它通过海量训练，学会了人类语言中“什么情况下该怎么说”这套规律。

它的核心机制是“基于上下文预测下一个 token”，而像人一样说话的感觉，其实是这种预测能力在大规模数据和对齐训练下表现出来的结果。

【中等】Token（词元） 是什么？为什么做 AI 应用时要重点关注 Token 而不是字符数？

Token 可以理解成模型处理文本时的最小计量单位。它不完全等于“一个字”或者“一个单词”，而是模型分词器切分之后得到的一段文本片段。

比如：

  • 一个英文单词可能是 1 个 token，也可能被拆成多个
  • 一个中文汉字有时是 1 个 token，有时几个字会一起组成 token
  • 标点、空格、数字，甚至代码符号，也都会占用 token

所以，模型真正看到和计算的不是字符数，而是 token 序列。

之所以做 AI 应用时要重点关注 token，而不是字符数，主要有 4 个原因：

第一，模型的上下文限制是按 token 算的。模型能处理多长的输入、能记住多少历史消息，看的是 token 数，不是字符数。同样 1000 个字符，中英文、代码、表格，它们对应的 token 数可能差很多。

第二，费用通常也是按 token 计费的。不管是输入成本还是输出成本，大多数模型厂商都是按 token 收费。所以你要控制成本，核心就是控制 token 消耗。

第三，性能和延迟也和 token 强相关。输入 token 越多，模型处理越慢；输出 token 越多，响应时间也越长。所以首字延迟、总耗时、吞吐量，本质上都和 token 有关系。

第四，很多工程问题其实都是 token 问题。比如上下文过长、RAG 召回内容塞太多、Prompt 写得太啰嗦、历史消息越带越多，这些最后都会转化成 token 爆炸。

你可以用一句话总结：

字符数是给人看的，token 数是给模型算的。做 AI 应用时，模型的容量、成本、速度，基本都直接和 token 挂钩。

所以在 AI 系统里，Prompt 压缩、上下文裁剪、历史摘要、检索结果限流，本质上都是在做 token 管理。

【中等】上下文窗口（Context Window）是什么？它会影响哪些能力？

上下文窗口（Context Window），就是模型在一次对话或一次请求里，能够同时看到和处理的最大 token 范围。

你可以把它理解成模型当前这次“思考时能放进脑子里的内容上限”。这里面通常包括：

  • system prompt
  • 用户当前问题
  • 历史对话
  • 检索出来的知识库内容
  • 工具返回结果
  • 以及模型正在生成的内容

这些内容加起来，不能超过它的上下文窗口限制。

它会影响的能力，主要有这几类：

第一，影响模型的“记忆长度”。上下文窗口越大，模型一次能看到的历史内容越多，多轮对话里就越不容易“忘前文”。如果窗口太小，前面的关键信息可能会被挤掉。

第二，影响长文本处理能力。比如长文总结、长代码分析、长合同问答、长对话续聊，本质上都依赖上下文窗口。窗口不够大，模型就没法一次读完全部内容。

第三，影响 RAG 和知识注入能力。做知识库问答时，检索出来的片段、引用信息、用户问题都要塞进上下文。如果窗口太小，就放不下足够的有效信息；如果硬塞太多，又可能导致噪声变大。

第四，影响多工具、多步骤 Agent 的表现。Agent 场景下，除了用户输入，还可能有工具调用记录、执行中间结果、计划步骤。上下文窗口不够时，模型在多步任务里就容易丢状态、逻辑断裂。

第五，影响成本和延迟。上下文窗口越大，不代表就一定越好。因为你塞进去的 token 越多，通常成本越高、响应越慢，而且注意力也可能被稀释。

你可以总结成一句话：

上下文窗口决定了模型一次最多能“看见多少信息”，它直接影响多轮记忆、长文本处理、RAG 效果、Agent 执行能力，以及成本和延迟。

所以工程上不是窗口越大越好，而是要尽量把真正有用的信息放进去，把无关内容裁掉。这也是为什么 AI 系统里会做历史摘要、上下文裁剪、检索限流这些优化。

【简单】什么是幻觉（Hallucination）？为什么大模型会产生幻觉？

幻觉（Hallucination），指的是大模型输出了看起来很合理、很像真的，但实际上不准确、没有依据，甚至是错误的信息。

比如：用户问一个不存在的论文，模型却一本正经地编出作者、发布时间和结论；或者让它总结一份它其实没读到的文件，它也可能“脑补”出内容。这种情况，本质上就是幻觉。

大模型会产生幻觉，核心原因是：

它本质上不是在“查事实”，而是在“根据上下文生成最可能出现的下一个 token”。也就是说，它的第一目标是把话接得自然、通顺、像样，而不是天然保证每一句都真实。

再往下讲，一般有几个常见原因：

第一，训练目标决定了它更擅长生成，不擅长事实校验。它在预训练阶段主要学的是语言模式和知识分布，而不是像数据库那样做精确查询。

第二，问题超出了它的知识边界。当用户问到它没学好、记不清，或者训练数据里本来就少的内容时，它也可能硬着头皮继续答。

第三，上下文不充分或者有误导。如果用户给的信息不完整，或者检索出来的资料质量不好，模型就容易基于残缺信息去“补全”。

第四，Prompt 或工具链设计不合理。比如明明应该走知识库检索或工具查询，却直接让模型裸答，这时候幻觉概率就会上升。

总结来说就是：

幻觉就是模型“编得很像真的，但其实不对”。它会出现，是因为大模型本质上是概率生成模型，擅长语言续写，不天然等于事实引擎。

如果要降低幻觉，通常要从几个方向入手：明确约束 Prompt、接入 RAG、增加工具调用、做结果校验，以及在答不上来时允许模型拒答，而不是硬答。

【中等】什么是预训练、监督微调和对齐训练？它们分别在解决什么问题？

这三个阶段可以理解成：先让模型学会“看懂和续写语言”，再让它学会“按要求回答问题”，最后再让它学会“更符合人类偏好和安全要求”。

1. 预训练（Pretraining）

预训练就是先拿海量文本数据去训练模型，让它学习语言本身的规律。最核心的任务通常是：根据前文预测下一个 token。

这个阶段主要解决的是：

  • 让模型学会语言模式、语法结构、常识知识
  • 学会词和词之间、句子和句子之间的关联
  • 具备基础的理解、续写、归纳和泛化能力

所以你可以把预训练理解成：

先把模型训练成一个“会读书、会接话”的通用语言模型。

但这个时候，它虽然会续写文本，不代表它已经是一个好用的助手。

2. 监督微调（SFT，Supervised Fine-Tuning）

监督微调，就是在预训练模型的基础上，再给它喂一批“高质量问答样本”或者“指令-回答样本”，让它学会按人类期望的方式回复。

这个阶段主要解决的是：

  • 让模型从“会续写”变成“会听指令”
  • 让回答格式更稳定
  • 让模型更像助手，而不是单纯文本补全器

比如：用户让它“总结一段话”“输出 JSON”“扮演面试官”，这些都是通过监督微调去强化的。

所以这一阶段本质上是在解决：

让模型从通用语言能力，变成可用的指令跟随能力。

3. 对齐训练（Alignment）

对齐训练，就是进一步让模型的输出更符合人类偏好、价值观和安全要求。常见做法包括基于人工反馈的优化，比如 RLHF，或者其他偏好对齐方法。

这个阶段主要解决的是：

  • 让回答更符合人类期待
  • 减少有害、违规、攻击性输出
  • 提高回答的礼貌性、帮助性和安全性
  • 在多个候选回答里，更倾向选择“更像人类认可的那个”

所以它解决的不是“会不会回答”，而是：

回答得是不是更合理、更稳、更安全、更符合人类使用习惯。

所以最后总结来说

预训练解决“会不会语言和知识”，监督微调解决“会不会按要求回答”，对齐训练解决“回答得是不是更符合人类偏好和安全要求”。

这三个阶段并不是互相替代，而是层层往上叠。预训练决定基础上限，SFT 决定指令跟随能力，对齐训练决定最终使用体验。

【中等】Temperature、Top-p、Top-k 分别控制什么？

这三个参数本质上都在控制一件事：模型在生成下一个 token 时，到底要“保守一点”还是“更发散一点”。

它们控制的角度不一样。

1. Temperature

Temperature 控制的是概率分布的“平滑程度”。

你可以理解成：它会影响模型在候选 token 之间，到底是更倾向选“最稳妥的那个”，还是愿意尝试一些没那么高概率、但更有变化的词。

  • Temperature 越低：输出越稳定、越保守、越一致
  • Temperature 越高：输出越发散、越随机、越有创造性

比如做：

  • 代码生成
  • 结构化输出
  • 精准问答

一般会把 Temperature 调低一点。

如果是：

  • 创意写作
  • 文案生成
  • 头脑风暴

通常可以适当调高。

2. Top-k

Top-k 控制的是：每一步生成时，只从概率最高的前 k 个候选 token 里选。

比如：

  • Top-k = 1，基本就只看最可能的那个
  • Top-k = 10，就只在前 10 个高概率候选里挑
  • k 越大，可选范围越大，随机性也越强

所以它控制的是：候选集合的数量上限。

3. Top-p

Top-p 控制的是：只保留累计概率达到 p 的那一批候选 token。

比如：

  • Top-p = 0.9表示从高到低累加概率，只保留总概率达到 90% 的那些 token，剩下低概率的直接丢掉。

所以它不是固定保留几个，而是动态决定保留范围。

它控制的是：候选集合的“概率覆盖范围”。

三者的区别

你可以这样总结：

  • Temperature：调节整体随机性强弱
  • Top-k：限制只看前 k 个候选
  • Top-p：限制只看累计概率前 p 的候选

实际工程里，Top-p 比 Top-k 更常用，因为它更灵活；而 Temperature 通常是最常调的参数。

【简单】为什么同一个问题问模型两次，返回结果可能不一样？

同一个问题问模型两次，结果可能不一样，核心原因是：大语言模型本质上不是“查固定答案”，而是在每一步生成时，按概率去选下一个 token。

也就是说，它不是像传统程序那样：

  • 输入一样
  • 输出就一定完全一样

而是会在多个“都合理”的候选里，选一个继续往下生成。只要采样过程里有随机性，两次结果就可能不同。

通常有几个原因：

1. 采样本身有随机性

如果模型不是强制选概率最高的那个词，而是从一批候选里按概率抽样，那么两次生成路径就可能不同。

这和前面提到的参数有关，比如：

  • Temperature
  • Top-p
  • Top-k

这些参数只要不是特别保守，输出就会有波动。

2. 一个问题本来就不止一种合理表达

很多问题并没有唯一标准答案。比如“解释一下什么是 RAG”，模型完全可以：

  • 用定义来回答
  • 用例子来回答
  • 先讲原理再讲应用

它们都对，只是表达路径不一样。

3. 上下文的细微差异也会影响输出

哪怕你问的是同一句话，只要前面的对话历史、系统提示词、工具返回结果稍微有变化，最终回答就可能不同。

面试里可以总结成一句话

因为大模型本质上是一个概率生成模型，不是在背固定答案，而是在根据上下文逐步采样生成内容，所以同一个问题问两次，结果可能不完全一样。

如果把随机性调低，比如把 Temperature 设得更低，输出通常会更稳定，但也可能少一些灵活性。

【中等】什么是推理模型？什么是非推理模型？它们适合哪些场景？

推理模型，一般指那种会在正式输出前，投入更多“测试时计算量”去做复杂分析、分步求解和多步规划的模型；

  • OpenAI 的官方文档里把它和普通 GPT 模型分开说明，强调这类模型更擅长复杂任务和多步推理，但通常也更慢、更贵。
  • Anthropic 这边对应的思路叫extended thinking，本质上也是让模型在复杂问题上花更多计算资源来思考。

非推理模型，可以理解成更偏“直接响应”的通用模型。它们通常也很强，但更强调速度、成本和高吞吐，适合清晰任务、短链路任务和大规模线上请求。OpenAI 的文档里就明确提到，GPT 模型通常更快、更省，但在复杂问题上往往更依赖你把指令写清楚。

两者最核心的区别，不是“一个聪明、一个不聪明”，而是算力分配方式不同：

  • 推理模型更像“先多想几步再回答”，适合难题
  • 非推理模型更像“尽快给出高质量响应”，适合高频日常任务

适用场景上，你可以这样说：

推理模型更适合：

  • 复杂逻辑分析
  • 数学、算法、证明题
  • 多步 Agent 规划
  • 复杂代码生成与排错
  • 需要权衡多个约束的系统设计题因为这类任务不是“马上答”最重要，而是“答对、答稳、链路完整”更重要。

非推理模型更适合：

  • 日常问答
  • 文本改写、总结、翻译
  • 常规客服回复
  • 低延迟聊天
  • 成本敏感的大规模在线调用因为这些场景更看重响应速度、吞吐量和单位成本。

简单来说就是：

推理模型强在“复杂问题求解”，非推理模型强在“快速稳定交付”；前者偏解难题，后者偏跑业务。真正落地时，通常不是二选一，而是按任务做模型路由。

【困难】你怎么理解“模型能力边界”和“AI 应用能力边界”的区别？

  • “模型能力边界”说的是模型本身能做到什么
  • “AI 应用能力边界”说的是把模型接进真实业务后，整个系统最终能做到什么。

先说模型能力边界

模型能力边界，核心看的是模型自身的上限，比如：

  • 能不能理解复杂指令
  • 能不能做多步推理
  • 能不能写代码、读图、做结构化输出
  • 长上下文下会不会掉信息
  • 工具调用是否稳定
  • 幻觉率、泛化能力、遵循指令能力怎么样

这部分本质上是：脱离具体业务，单看这个模型本身聪不聪明、稳不稳定、支持哪些模态和能力。

比如一个模型数学强、代码强、长文本总结强，那是它的模型能力。但这不代表你拿它做出来的产品就一定好用。

再说 AI 应用能力边界

AI 应用能力边界，看的就不是模型单点能力了，而是整个系统上线后，能不能稳定解决真实问题。

它通常取决于这些东西：

  • Prompt 设计是否合理
  • RAG 质量怎么样
  • 工具链是否完整
  • 数据是不是干净
  • 权限、安全、风控是否到位
  • 前后端交互是否顺滑
  • 成本、延迟、并发是否能扛住
  • 评测、监控、回滚机制是否完善

所以应用能力边界，本质上是：模型能力 × 系统工程 × 业务约束 × 数据质量 的综合结果。

两者最大的区别

模型能力边界，回答的是“这个模型理论上能干什么”；应用能力边界，回答的是“这个产品在线上到底能不能把这件事做好”。

模型强，不代表应用就强。

因为真实业务里，问题往往不出在“模型不会答”，而出在：

  • 没有给它足够对的上下文
  • 检索召回错了
  • 工具没接全
  • 输出没校验
  • 前端状态没处理好
  • 安全边界没收住
  • 成本太高不敢放量

举个很典型的例子

比如你做一个 AI 客服系统。

从模型能力看，它可能已经具备：

  • 理解用户问题
  • 生成自然回复
  • 总结上下文
  • 调用工具查订单

但真正上线时，应用能力边界还要看：

  • 能不能接企业知识库
  • 能不能区分该 AI 回还是该转人工
  • 能不能避免瞎编退款政策
  • 能不能处理高并发会话
  • 能不能在 2 秒内先回第一段内容
  • 能不能记录日志、追溯决策链路
  • 能不能控制每轮对话成本

所以最后用户感知到的，不是“模型有多强”，而是“这个系统到底靠不靠谱”。

因为很多人做 AI 项目时，容易把问题全归因到模型：

  • 答得不好，怪模型
  • 成本高，怪模型
  • 幻觉多，怪模型

但实际上，很多时候模型只是占一部分。

模型能力决定天花板，应用能力决定最终交付。

我的理解是：模型能力边界更偏底层能力上限，AI 应用能力边界更偏业务落地上限。前者决定你“有没有这个武器”，后者决定你“能不能把这把武器真正打准、打稳、打出结果”。

所以很多 AI 项目的核心竞争力，最后不在模型选型，而在系统设计、数据治理、工具链、评测和工程落地。

【困难】如果用户说“AI 最近变笨了”，你会如何排查？

“AI 变笨了”通常不是一个单点问题，而是模型、Prompt、上下文、RAG、工具链、数据、前端展示、评测口径这几层里，至少有一层发生了变化。

所以我的排查思路一般是：先确认现象，再做分层定位，最后做回归验证。

1. 先确认“变笨”到底指什么

因为用户说变笨，可能指的是完全不同的问题，比如：

  • 回答不准了
  • 幻觉变多了
  • 响应变慢了
  • 风格变了
  • 不会调用工具了
  • 知识库命中率下降了
  • 多轮对话更容易忘上下文了

所以第一步是先拿真实 case，确认到底是：效果问题、稳定性问题、性能问题，还是交互感知问题。

2. 先看最近有没有“变更”

如果以前正常、最近突然不行，我会优先查最近有没有上线变更。重点看这几类：

  • 模型版本有没有切换
  • Prompt 有没有改
  • RAG 检索策略有没有变
  • rerank 有没有下掉
  • 工具描述、参数 Schema 有没有改
  • 上下文裁剪规则有没有改
  • 前端渲染或消息拼接逻辑有没有改
  • 是否做了降本，比如缩短上下文、换便宜模型、减少召回条数

因为很多“变笨”，其实不是模型本身退化，而是为了降本或提速，系统偷偷做了能力压缩。

3. 按链路拆分排查

我一般会按这条链路看：用户输入 → Prompt 组装 → 上下文注入 → 检索召回 → 重排 → 工具调用 → 模型生成 → 前端展示

第一层：输入和 Prompt 层

先看 system prompt、业务 prompt、few-shot 有没有变化。比如：

  • 约束是不是变弱了
  • 输出格式要求是不是变复杂了
  • Prompt 里有没有互相冲突的指令
  • 历史 Prompt 版本回退时有没有丢规则

有时候模型不是变笨，而是你给它的任务表达变差了。

第二层：上下文层

看上下文是不是被截断了，或者被噪声污染了。常见问题有：

  • 历史对话带太多，关键内容被挤掉
  • 检索片段塞太多，模型注意力被稀释
  • 摘要质量下降，丢了关键约束
  • 工具返回结果太长，把核心信息淹没了

很多“答非所问”，本质上不是模型不会，而是它看到的信息已经不对了。

第三层：RAG 层

如果这是知识库类产品，我会重点查：

  • 召回结果是不是变差了
  • chunk 策略有没有改
  • embedding 模型有没有切
  • 向量库数据有没有漏更新
  • metadata 过滤条件是不是错了
  • reranker 是否失效
  • 引用内容和最终答案是否一致

这类问题很常见。用户觉得 AI 变笨了，实际上是知识没喂准。

第四层：工具调用层

如果是 Agent 或 Tool Calling 场景，我会看：

  • 工具调用成功率有没有下降
  • 参数是否经常解析失败
  • Tool 描述是否让模型误解
  • 是否出现误调用、漏调用、循环调用
  • 工具超时后系统有没有兜底

因为用户只会看到“AI 没答对”，但实际上可能是工具没调起来。

第五层：模型生成层

这一步才是看模型本身：

  • 模型是不是换版本了
  • temperature、top-p 之类参数有没有改
  • 最大输出长度有没有缩短
  • 是否触发了更严格的安全策略
  • 是否被做了路由降级，复杂问题落到了便宜模型上

也就是说，要先排除系统问题，再怀疑模型问题。不能一上来就说“模型退化了”。

第六层：前端展示层

这个经常被忽略。有时候后端返回其实没问题，但前端让用户感觉“变笨了”，比如：

  • 流式拼接丢字
  • Markdown 渲染异常
  • 工具结果卡片没展示全
  • 会话恢复时拿错历史消息
  • 多条消息状态错乱，导致用户以为回答不完整

所以我也会对比：原始模型返回内容和用户最终看到的内容是否一致。

4. 看监控和日志，不靠感觉排查

我会重点拉这些指标：

  • 命中知识库的比例
  • 工具调用成功率
  • 平均输入 / 输出 token
  • 首字延迟、总响应时长
  • 拒答率、失败率、重试率
  • 线上满意度或人工标注分
  • 不同模型路由占比
  • 样本级 trace

如果是复杂系统，我会直接看单次请求的完整 trace：

  • 用户问了什么
  • 实际拼出来的 prompt 是什么
  • 检索召回了哪些内容
  • 有没有 rerank
  • 调了哪些工具
  • 每一步返回了什么
  • 最终展示给用户的是什么

这样能快速判断问题落在哪一层。

5. 做对照实验

排查时我不会只看一个 case，而是会做 AB 对照：

  • 老 Prompt vs 新 Prompt
  • 老模型 vs 新模型
  • 带 RAG vs 不带 RAG
  • 带 rerank vs 不带 rerank
  • 全量上下文 vs 裁剪上下文
  • 原始工具链 vs 当前工具链

这样能快速判断，到底是：

  • 模型能力变了
  • 系统组装变了
  • 还是评测口径变了

6. 最后做回归验证

定位出来之后，我会用一批固定测试集做回归。不能只修一个 case，就说问题解决了。

一般至少覆盖：

  • 高频真实用户问题
  • 边界问题
  • 多轮问题
  • 知识库问题
  • 工具调用问题
  • 长上下文问题

面试里可以这样总结

如果用户说 AI 最近变笨了，我不会直接归因到模型，而是会把问题拆成现象确认、变更检查、链路分层排查、日志指标分析和回归验证五步。因为 AI 是否“聪明”，最终取决于模型、Prompt、上下文、RAG、工具链和前端交付的整体结果，而不是某一个点。

我的经验是，线上 AI 变笨，很多时候不是模型退化，而是系统某一层的信息质量下降了。真正有效的排查方式，不是换模型试运气，而是把整条链路拉直看。

【困难】如何防止模型把系统提示词、内部知识或其他用户数据泄露出去？

这个要靠分层防护。

因为系统提示词、内部知识、其他用户数据的泄露，本质上属于prompt injection + 数据越权 + 工具外传的组合问题

OpenAI 和 OWASP 都把这类风险明确列为重点，尤其是在模型能访问网页、文件、向量库或 MCP/工具时，攻击者可以把恶意指令夹带进外部内容里，诱导模型把私有数据带出去。

我一般会从六层来防。

第一层是 不要把真正敏感的信息直接塞进 prompt。

能不放进去的就不放。Anthropic 的文档明确建议，只有模型完成任务必须用到的专有信息才放进提示里，额外的 proprietary 信息只会增加泄露面，而且过度复杂的“防泄露 prompt”本身还可能拖累效果。也就是说，系统提示词里不要放密钥、数据库连接串、完整内部策略、其他用户原始数据这些高价值内容。

第二层是 把“指令”和“数据”严格分开，所有外部内容一律按不可信输入处理。

OpenAI 明确建议不要把不可信变量直接放进 developer message，因为这类消息优先级高，攻击者一旦注入，控制力最强；OWASP 也强调，prompt injection 的根源之一就是系统把自然语言指令和普通数据混在一起处理。因此我会把用户输入、网页内容、文档内容、检索结果、邮件、API 返回，全都当作不可信数据，用清晰边界包起来，只作为“被分析对象”，而不是“可执行指令”。

第三层是 做最小权限和数据隔离。

OWASP 的建议很明确：工具权限要最小化、按工具分 scope、不同信任级别用不同工具集，敏感操作要额外授权。落地上就是：读和写分离，内部知识查询和外部联网分离，A 用户会话和 B 用户会话内存隔离，跨用户数据默认不可见。真正高风险的工具，比如发邮件、执行命令、写数据库、删除文件，都不能让模型自由调用。这样即便模型被诱导，它也没有足够权限把东西真正带出去。

第四层是 限制数据流，尤其是工具参数和模型输出必须校验。

OpenAI 建议用结构化输出约束节点之间的数据流，并对工具参数做 schema 或 regex 校验，避免模型把任意 payload 偷渡到下游；Anthropic 也建议先做 output screening 和 post-processing，检查输出里有没有提示词片段、内部标识符、敏感关键词或异常长文本。我的做法一般是：模型永远不直接决定最终外发内容，所有要发到第三方接口、网页、邮箱、日志系统里的内容，先走白名单字段提取和脱敏过滤，再决定是否发送。

第五层是 把高风险流程拆阶段执行，不让“外部不可信内容”和“内部敏感数据”同场出现。

OpenAI 在 deep research 的安全文档里给了很典型的建议：涉及敏感数据时，把工作流分阶段，比如先做公开网页检索，再开第二次调用访问私有 MCP 或内部数据，而且这第二步不再给它 web access。这个思路非常重要，本质上就是降低混合上下文导致的数据外传风险。简单说：能分两跳就别一锅炖，能先公网后内网，就不要让一个 agent 同时看到两边。

第六层是 监控、审计和红队测试。

OpenAI 和 OWASP 都强调要记录并审查工具调用、模型消息和敏感链路；Anthropic 也建议定期审计 prompt 与输出。因为这类风险没有“100% 防住”的说法，只能靠持续监控和对抗测试去发现问题。我会重点监控：是否有人频繁尝试索要 system prompt、是否出现异常长输出、是否有跨用户 ID 命中、是否把内部字段传给第三方域名、是否有 memory 被污染后跨会话复现。上线前也会做 prompt injection、间接注入、越权查询、隐蔽编码、文档藏指令这些红队用例。

防止系统提示词、内部知识和其他用户数据泄露，核心不是“让模型自己保密”，而是把它当成一个不完全可信的执行体来设计系统：少给数据、分开上下文、最小化权限、约束输出、分阶段执行、全链路审计。模型可以参与决策，但不能直接拥有无限读取和无限外发能力。
