【简单】什么是 RAG？它主要解决了什么问题？

我的理解是，RAG 本质上就是一件事：别让大模型闭着眼睛答，先给它找资料，再让它回答。

因为大模型虽然看起来什么都懂，但真正到线上你会发现两个问题特别明显。

  • 一个是它知道的东西不一定新
  • 另一个是它很容易把“像真的”说成“是真的”。

所以 RAG 的核心思路其实很朴素，就是用户提问之后，系统先去知识库里把相关内容找出来，再把这些内容和问题一起喂给模型，让它基于资料回答。

所以它本质上不是单纯生成，而是“检索 + 生成”的组合。

它主要解决的，我一般会讲三类问题：

  • 第一类是知识更新问题。比如产品规则、客服话术、公司政策、接口文档，这些内容经常变。你不可能每改一次文档就去重训一次模型，所以更现实的做法是把知识放到外部库里，查完再答。
  • 第二类是幻觉问题。当然 RAG 不能保证百分之百不幻觉，但至少它能让模型尽量围绕检索到的材料说，而不是完全自由发挥。
  • 第三类是私有知识接入问题。企业内部很多东西，通用模型本来就不知道。这个时候不是模型不聪明，而是它压根没见过。RAG 就是在补这个信息入口。

所以这题我不会把 RAG 说得很玄，我会直接说：

RAG 解决的不是模型推理能力本身，而是模型回答时“没有最新知识、没有私有知识、没有依据”这几个问题。

【中等】创建知识库的流程是什么？使用 rag 切分检索的主要思路是什么？

真正可用的知识库，一般要分成两段看：

  • 第一段是入库流程怎么做
  • 第二段是检索链路怎么设计。

先说创建知识库的流程。我一般会这么做：

  • 第一步不是直接切分，而是先做数据接入和清洗。因为知识源通常很杂，可能有 PDF、Word、Excel、网页、FAQ、内部文档，甚至截图 OCR。这里先要解决的是格式统一、乱码、页眉页脚、重复段落、脏数据这些问题。否则后面切得再好，检索质量也会被脏数据拖垮。
  • 第二步是做文档解析和结构化。比如标题、章节、表格、列表、问答对、时间、产品名、版本号这些信息，我一般都会尽量提出来，变成 metadata。因为真实检索里，光靠向量相似度不够，后面往往还要结合 metadata 过滤，比如只搜某个产品线、某个时间范围、某个知识库版本。
  • 第三步才是分块。这里我一般不会一刀切按固定字数硬切。更常见的做法是“语义边界优先，长度控制兜底”。比如优先按标题、段落、问答对、表格块去切，如果某一段太长，再做二次切分。然后再加一点 overlap，保留前后 1 到 3 句，避免上下文断裂。这里的核心取舍是：chunk 太大，召回可能不准，噪音多；chunk 太小，信息不完整，模型拿到后也答不明白。所以我不会拍脑袋定 chunk 大小，一般会拿真实 query 去测召回效果。
  • 第四步是向量化和建索引。把 chunk 用 embedding 模型转成向量，写入向量库，同时把 chunk 原文、metadata、文档版本这些一起存进去。很多人只关注 embedding 模型，但我觉得索引设计、字段设计、版本管理同样重要，不然后面做增量更新、灰度发布会很痛苦。
  • 第五步是评测和迭代。知识库不是建完就结束了，要拿一批真实问题做离线评测，看召回率、命中率、TopK 命中位置、最终回答准确率。如果效果差，再回头看是解析有问题、切分有问题，还是检索策略有问题。

再说 RAG 切分检索的主要思路。

它本质上是在做一件事：先尽量找到对的上下文，再让模型基于这些上下文去回答。

所以切分的目标，不是机械拆文本，而是让每个 chunk 都尽量成为一个“可被独立召回、又能表达完整意思”的信息单元。

比如 FAQ 场景我更倾向按问答对切；产品文档更适合按标题和小节切；表格类内容可能要单独转成结构化文本，不然检索很容易失真。

检索这块，一般也不是只做单路向量召回。更稳一点的做法通常是：

  • 先做 query 理解，看要不要改写、补全、做意图识别；
  • 然后做多路召回，比如向量召回 + 关键词召回 + metadata 过滤；
  • 把候选结果捞出来之后，再做rerank 重排；
  • 最后把前几条最相关的 chunk 拼进 prompt，让模型生成答案。

有些场景还会要求引用来源、返回片段位置，方便用户校验。

这里面最关键的工程判断是：RAG 的效果，不是某一个模型单点决定的，而是“切分质量 + 召回策略 + 重排能力 + Prompt 约束”共同决定的。

我自己会特别注意几个坑：

  • 一个是chunk 切得看起来很工整，但不符合用户提问方式。比如用户问的是一个完整流程，但你切出来都是碎片句子，这种就很难答好。
  • 一个是只做向量召回，不做关键词兜底。像产品名、错误码、版本号这种精确词，纯语义检索不一定稳。
  • 还有一个是知识库更新后没有版本管理，结果新旧内容混着召回，线上会很乱。

所以如果让我总结，我会说：知识库建设的重点，不是“存进去”，而是“能不能被稳定、准确、可控地找出来”；RAG 的重点，也不是“检索到了”，而是“检索到的内容能不能刚好支持模型回答这个问题”。

最后总结来说：知识库不是把文档塞进向量库，而是先清洗解析、再按语义合理切分、再做多路召回和重排，最终让模型拿到“刚好够用的正确上下文”去回答。

【中等】为什么引入了 RAG 之后，模型幻觉并不会自动消失？

我不会认为上了 RAG，幻觉就自然没了。

RAG 本质上只是给模型多塞了一层“外部参考信息”，它解决的是知识更新和事实依据不足的问题，但它并没有从根上消灭模型“会猜、会补、会编”的生成特性。

我一般会把这个问题拆成两层看。

第一层，检索阶段本身就可能出错。比如你查到的 chunk 不相关，召回不全，排序不准，或者知识库里本身就是过期的、冲突的、脏的数据。那模型拿到的“参考资料”就已经不靠谱了。很多人会以为 RAG 的重点只是“接上向量库”，但真正到线上，问题往往出在检索链路：切分不合理、embedding 不适合这个领域、query 改写失败、topK 太小或太大、rerank 没做好，这些都会让模型看到错误上下文。

第二层，就算检索到了对的内容，模型也不一定老实照着答。因为生成模型不是数据库查询，它会做语言补全和推断。真实场景里很常见的情况是：资料里只说了一半，模型为了让回答更完整，就把后半段“脑补”出来了；或者上下文里有多个片段互相矛盾，模型会自己做一个看起来合理、但其实不严格正确的融合。

所以 RAG 只是“给依据”，不是“强约束”。

再往下说，RAG 还有一个常见误区：检索命中 ≠ 最终答案正确。

因为最终答案是“检索结果 + prompt 约束 + 模型推理 + 输出策略”共同决定的。只要后面这几层没控住，幻觉还是会出来。

所以如果放到真实项目里，我不会把“接了 RAG”当成幻觉治理完成，而是会继续做几件事：

  • 一是提高检索质量，包括 chunk 策略、召回 、重写、知识去重和更新机制。
  • 二是让模型学会引用和拒答，比如要求“只能基于已检索内容回答，找不到就明确说不知道”。
  • 三是做答案约束，像结构化输出、引用来源、置信度判断、低置信度走澄清或转人工。
  • 四是单独评测，不要只看最终回答，要拆开看召回质量、命中率、答案忠实度、引用一致性。

所以我的理解是：RAG 不是消灭幻觉，而是把幻觉从“纯瞎编”变成“有机会被约束、被发现、被降低”。

它能显著降低幻觉概率，但前提是检索链路、上下文治理和输出约束都得跟上。否则很多项目只是“把幻觉从闭卷编，升级成开卷乱编”。

【中等】RAG 的完整链路一般包括哪些步骤（用户输入查询、向量数据库、大模型，之间的完整配合方案）？

如果放到真实项目里，我会把它拆成两条链路：离线建库链路和在线问答链路。真正能不能做好，核心不是只看中间那次检索，而是前后两段都要通。

先说 离线建库。

  • 第一步是知识采集，把 PDF、Word、网页、FAQ、数据库记录这些原始资料拿进来。
  • 第二步是清洗和结构化处理，因为原始资料通常很脏，可能有页眉页脚、乱码、重复段落、表格断裂、过期内容，不处理干净，后面检索质量会很差。
  • 第三步是切分 chunk（分块，之前有多次提到过这个概念，这里不重复说了哈），也就是把长文档拆成适合检索的小片段。这个不是随便按字数切，通常要结合标题、段落、语义边界来切，不然上下文会断。
  • 第四步是生成 Embedding，把每个 chunk 转成向量。
  • 第五步是把原文、元信息、向量一起写进向量数据库，比如 chunk 内容、来源、标题、更新时间、权限信息这些。到这里，知识库才算真正建好。

然后是 在线问答链路。

用户先输入问题后，我一般不会直接拿原问题去搜，而是先做一层查询预处理。比如： query rewrite、补全上下文、识别用户意图、判断是不是需要多轮历史、是不是要走指定知识域。因为很多用户提问其实很口语、很模糊，直接检索效果不一定好。

query rewrite，可以理解成：查询改写。把用户原始问题改写成更适合检索的搜索语句。

接着进入召回阶段。

系统会把用户 query 也向量化，然后去向量数据库里找相似 chunk，通常会拿到 topK 结果。但这里一般还不够，因为只靠向量召回容易“差不多相关，但不够准”，所以很多线上系统会再加一层 重排，把真正最相关的内容排到前面。有些场景还会做混合检索，把关键词检索和向量检索一起用，避免只靠语义检索漏掉专有名词、编号、产品名。

拿到候选片段之后，不是直接全塞给大模型，而是要做上下文组装。也就是把最相关的 chunk、来源信息、系统提示词、回答约束一起拼成 prompt。这里通常会控制 token 长度，做去重、合并、排序，不然上下文太长反而会影响模型判断。

然后才进入生成阶段。

大模型根据“用户问题 + 检索到的资料”来生成答案。如果要求更严一点，我一般还会加规则：只能基于检索内容回答，答不上来就明确说不知道；尽量附带引用来源；不要超出资料做过度发挥。

最后还有一层很多人容易漏掉，就是后处理和评估闭环。

比如答案格式化、引用高亮、敏感信息过滤、低置信度转人工、日志埋点、用户反馈回收。再往后会基于这些数据持续优化 chunk 策略、embedding 模型、topK、rerank 和 prompt。

所以如果让我用一句工程化的话来概括，我会说：

RAG 的完整链路不是“检索增强生成”这六个字，而是“知识入库 → 查询改写 → 召回 → 重排 → 上下文组装 → 大模型生成 → 引用与评测闭环”这一整套系统。

很多项目做不好，不是因为大模型不行，而是前面的切分、召回、重排，或者后面的约束和评测没做好。

【困难】RAG 系统常见的评测指标有哪些？

真实项目里，RAG 评测一定要拆层看，不然很容易出现一种假象，那就是：最终答案看着还行，但其实检索已经歪了；或者检索没问题，最后是生成阶段在胡说。

所以我一般会把 RAG 的评测拆成三层：

  • 检索层
  • 生成层
  • 系统层。

另外再补一个判断：有标注集和没标注集，评测方法也不一样。像 Databricks 这类方案里，就明确区分了 deterministic / ground-truth 型评测，和 LLM-as-a-judge 这类 judge 型评测。前者更适合有标准答案的离线集，后者更适合大规模自动化巡检。

先说检索层指标。

这一层我最关心的是：有没有把对的内容找回来，以及排得对不对。

最常见的是：

  • Precision@K / chunk relevance：看 topK 里返回的内容有多少是真的相关。这个指标高，说明召回结果“杂质少”。
  • Recall@K / document recall / context recall：看该找回来的相关文档有没有被找回来。这个指标低，说明系统容易漏信息，最终答案就可能不完整。
  • MRR：看第一个相关结果出现得够不够靠前，适合问答场景，因为很多时候 top1、top3 的质量最关键。
  • MAP@K：看整体排序质量，不只是第一个相关结果。
  • DCG@K / NDCG@K：这类指标会同时看“相关性强弱”和“排位靠前程度”。Databricks 甚至把 DCG@10 作为 retrieval quality 的主指标，因为它既看排序，也看每条结果到底有多有用。
  • Average relevance / relevance distribution：这个更像诊断指标，能看出结果是不是“两极分化”，比如一半特别准，一半完全不相关。

然后是生成层指标。

这一层的重点不是“文采好不好”，而是回答有没有忠于检索内容、有没有真的回答问题。

我实际最常看的有几个：

  • Faithfulness / Groundedness（忠实度 / 依据性）：这是 RAG 里非常核心的指标。意思是模型生成的结论，能不能从检索到的上下文里找到证据支撑。答得再流畅，只要有编造成分，这项就该扣分。RAGAS、TruLens、LangSmith 都把这类指标放在核心位置。
  • Answer Relevance（答案相关性）：回答是不是在正面回应用户问题，有没有答偏、答空、答了一堆不相干的内容。
  • Answer Correctness（答案正确性）：这个通常要对比 ground truth，看最终回答是否正确。它和 groundedness 不完全一样：有时候答案是基于上下文生成的，但上下文本身就不够，最后仍然可能答错。
  • Context Sufficiency（上下文充分性）：这个指标我觉得很有工程价值。它不只是问“检索结果相关不相关”，而是问“这些检索结果够不够支撑出标准答案”。相关但不充分，照样会让模型半对半错。
  • Safety（安全性）：很多人会漏掉这一层，但线上系统必须看，尤其是医疗、金融、客服类场景。

再说系统层指标。

线上项目里，我不会只盯质量，因为 RAG 最后是个系统，不是论文实验。

这一层至少要看：

  • Latency：整体延迟，尤其是检索 + rerank + 大模型生成串起来之后的总耗时。
  • Token / Cost：输入 token、输出 token、总 token，以及单次请求成本。因为很多 RAG 优化，最后都会落到“质量提升值不值得这笔钱”。
  • 稳定性和可重复性：RAGAS 论文里也专门提到，LLM 评测本身就有波动，所以离线评测要关注复现性，不然你以为自己优化了，其实只是 judge 飘了。

如果面试官继续追问“那你真实项目里到底怎么评”，我会补一句：我不会只看最终 answer score，而是做拆层评测。先看检索召回和排序，再看 groundedness / correctness，最后再看延迟和成本。因为只有这样，你才知道问题到底出在 embedding、召回、rerank、prompt，还是模型生成本身。LangSmith、TruLens、RAGAS 这些方案，本质上也是沿着这个思路在做。

再落地一点说，如果是通用问答知识库，我最常盯的其实就是这几个：Recall@K、NDCG@K、Groundedness、Answer Correctness、Latency、Cost。因为这几个基本能把“找没找对、排没排对、答没答对、贵不贵、慢不慢”都覆盖掉。

最后我会收一个原则：RAG 评测不是单指标游戏。检索指标决定你喂给模型什么，生成指标决定模型有没有乱说，系统指标决定这个方案能不能上线。

【一句话版本】RAG 常见评测指标本质上分三层：检索看 Recall/Precision/MRR/NDCG，生成看 Groundedness/Faithfulness、Answer Relevance、Correctness，系统看 Latency 和 Cost；真正有工程价值的是拆层评测，不是只看最终答案分。

【中等】分块的策略是什么？如何进行分块？

我一般不会把 RAG 的分块理解成“随便切一段文本”这么简单。

它本质上是在做一件事：把原始知识，切成既方便检索、又不容易丢上下文的最小可用单元。切得太大，召回的时候会带很多噪音，模型上下文也容易被无关内容占满；切得太小，又会把一句完整的话、一个完整规则、一个完整步骤拆碎，最后召回来也答不准。

所以真实项目里，我不会直接拍脑袋定一个固定长度，而是会先看文档类型，再定分块策略。

比如最基础的一类是固定长度分块，按字数或者 token 数去切，这种实现最简单，适合 Demo 或者文本结构很差的场景，但问题也很明显，就是容易把语义切断。线上更常见的是按结构分块，比如按标题、段落、列表、表格、代码块、FAQ 问答对去切。因为很多知识本身就是有层级的，像“一级标题 + 二级标题 + 正文”这种结构其实很有信息量，保留下来对召回很有帮助。再往上一层，会做语义分块，也就是不是看到长度到了就切，而是看话题是不是变了、段落语义是不是断了，这样切出来的块通常更自然，但代价是实现更复杂，预处理成本也更高。

还有一些比较常用的增强手段，比如overlap 重叠分块，前后保留一点重叠内容，避免关键信息刚好被切在边界上；再比如父子块或者层级块，检索时先召回小块，真正喂给模型时再带上它的父级上下文，这样准确率和上下文完整性会更平衡。

如果问我项目里怎么做，我通常会按这个流程走：

  • 先做文档解析和清洗，把 PDF、Word、网页、FAQ、表格这些内容先转成结构化文本，同时保留标题层级、段落关系、表格标题、字段名这些元数据。因为很多时候，不是 chunk 切得不好，而是解析阶段就把结构搞丢了。
  • 然后做分块策略设计。如果是 FAQ、帮助中心、政策说明，我更倾向于按问答对或者按小节切，因为这类内容天然就是一个问题对应一个答案。如果是长文档、产品手册、研究报告，我一般按“标题层级 + 段落”去切，再加一点 overlap。如果是表格、代码、配置说明，我不会跟正文混着切，而是单独处理，因为这种内容对格式依赖很强，切坏了基本就没法用了。

块大小上，我一般不会追求统一数值，而是看 Embedding 模型和业务 query 的粒度。比如用户的问题都比较短、比较明确，那 chunk 可以偏小一点，提升定位能力；如果用户经常问流程型、规则型、多条件问题，那 chunk 就不能太碎，不然模型拿不到完整上下文。经验上会先做一个基线，比如几百个 token 一个块，再用真实 query 去测 Recall、命中率、最终答案准确率，然后再调。

还有一个我比较看重的点是：chunk 不只是文本本身，还要带 metadata。比如文档标题、章节路径、来源、发布时间、产品线、适用地区、是否生效，这些信息在线上非常重要。很多时候提升准确率，不是靠把 chunk 切得更花，而是靠 metadata 先过滤，再做向量召回。

最后我会用评测来定方案，而不是靠感觉。我通常会看几类指标：

  • 一类是召回指标，比如 Recall@K、命中率；
  • 一类是生成侧指标，比如最终回答正确率、引用是否对、幻觉率；
  • 还有工程指标，比如索引体积、检索延迟、重建成本。

因为分块不是越细越好，也不是越长越好，它本质上是准确率、延迟、成本、可维护性之间的 取舍。

所以总结一下，我对 RAG 分块的理解不是“怎么切文本”，而是“怎么把知识组织成对检索最友好的单位”。

真实项目里，优先级通常是：先保结构，再保语义，再用 overlap 和 metadata 做补强，最后拿真实 query 和标注集去验证。

【中等】为什么“分块（Chunking）策略”会直接影响检索质量？

分块是实现 RAG 的非常核心的一层，因为它直接决定了“被检索的最小知识单元”长什么样。

说白了，Embedding、向量库、召回算法这些东西再强，前提也是你喂进去的 chunk 本身要合理。chunk 切得不好，后面基本就是在垃圾输入上做高质量检索，效果上不去。

我自己的理解是，分块策略主要会从三个层面影响检索质量。

第一，它会影响语义完整性。

如果切得太碎，一段话刚好把定义切在前面，把约束条件切在后面，模型检索到其中一块时，信息其实是不完整的。这样召回结果表面上“语义有点像”，但真正拿去回答就容易偏，尤其在 FAQ、制度文档、技术文档这种强上下文场景里很明显。

第二，它会影响召回精度。

如果 chunk 太大，又会把很多无关内容一起塞进去，向量表达会被稀释。最后用户问一个很具体的问题，检索出来的是一大段“相关但不够准”的内容。这个时候你会发现 topK 好像命中了，但真正有效片段不突出，后面的生成也容易答偏或者绕。

第三，它会影响后续生成阶段的上下文利用率。

因为召回不是终点，召回出来的 chunk 还要塞进 prompt。如果 chunk 普遍偏大，一次检索回来几个 chunk，很快就把上下文窗口吃掉了，还会带来噪音，影响模型注意力。反过来如果 chunk 太小，虽然单块很短，但你可能要召回更多块才能把问题拼完整，延迟和复杂度也会上来。所以它不是单纯越大越好，也不是越小越好。

所以真实项目里，我一般不会拍脑袋定 chunk 大小，而是结合文档类型来定。比如 FAQ、问答对、字段说明这种本来边界就很清楚的内容，我更倾向于按语义单元切；如果是长文档、产品手册、政策说明，我会优先按标题、段落、章节这种结构切，再配一点 overlap，避免上下文断裂；如果是代码、日志、表格这类特殊内容，分块方式还要再单独设计，不能直接套通用规则。

另外一个很容易被忽略的点是，分块策略其实和 query 类型强相关。用户如果问的是“定义类”“事实类”问题，偏小一点、边界清晰的 chunk 往往更好；但如果问的是“流程类”“对比类”“原因类”问题，往往需要更完整的上下文，这时候 chunk 不能太碎，不然召回回来都是残片，模型很难组织出完整答案。

所以我一般会把 chunking 当成一个需要评测的参数，而不是固定写死。真实落地里会拿标注集去测，比如看 Recall@K、命中率、最终回答准确率，甚至看 badcase：到底是没召回、召回不准，还是召回了但 chunk 信息不完整。很多时候问题不在模型，也不在向量库，而是在 chunk 从一开始就切坏了。

我的原则一直是：分块的目标不是“切开文本”，而是“切出可检索、可理解、可拼装的知识单元”。

这件事做不好，后面的检索质量很难真正稳定。

【中等】固定长度分块和按语义分块，各有什么优缺点？

我的理解是，这两个方案没有绝对谁更好，核心还是看你的数据形态、检索目标，以及你更在意实现成本还是最终效果。

如果先给结论，我一般不会只选一种。

  • 固定长度分块更像一个工程上稳定、简单、好落地的默认方案；
  • 按语义分块更像一个效果导向的优化方案，通常准确率更好，但实现复杂度也更高。

先说固定长度分块。

它最大的优点就是简单、稳定、可控。你比如按 500 字、800 token 这种方式切，处理链路很好做，速度快，规则统一，不太依赖文档结构，对各种脏数据也比较稳。像 PDF 转出来格式很乱、标题层级不清、内容质量参差不齐的时候，固定长度反而不容易翻车。

但它的问题也很明显，就是它不理解内容边界。它可能正好把一个完整定义切成两半，也可能把上一段的结论和下一段的无关内容拼在一起。这样做出来的 chunk，在 embedding 的时候语义往往是不自然的。线上就容易出现一种情况：召回看着命中了，但拿出来一看，信息要么断了，要么混了，最终生成阶段不太好用。

再说按语义分块。

它的核心优势是更符合人类阅读和知识组织方式。比如按标题、段落、章节、问答对、步骤块去切，这样每个 chunk 本身往往更完整，语义边界更清晰，所以检索出来的内容通常更“像一个能直接拿来回答的知识单元”。尤其像产品文档、制度文档、帮助中心、技术说明这种结构化文本，按语义切通常效果会比固定长度好。

但它的问题也不是没有。

  • 第一，复杂度高。你得先把文档结构识别出来，很多 PDF、扫描件、表格、乱格式文档，这一步本身就不稳定。
  • 第二，结果不够均匀。有的 chunk 很短，有的 chunk 很长，后面 embedding、召回、上下文拼接时不太好控制。
  • 第三，过度依赖上游解析质量。只要标题识别错了、段落合并错了，后面的 chunk 质量就会被连带影响。

所以真实项目里，我一般这样看：

如果是冷启动阶段，或者数据很杂，我通常先用固定长度分块打底，因为它便宜、快、稳定，能先把系统跑起来。如果后面发现 badcase 很多都集中在“上下文断裂”“答案不完整”“召回到的片段像半截话”，我才会逐步往语义分块升级。再进一步，很多线上系统其实会做混合方案，比如“先按语义结构切，再对超长段落做固定长度二次切分”，或者加 overlap，把效果和稳定性折中一下。

固定长度更偏工程效率和一致性；语义分块更偏检索效果和内容完整性。

如果文档质量高、结构清晰，我更愿意用语义分块；如果文档来源杂、解析不稳定，我会更谨慎，宁愿先用固定长度保底。

最后我一般不会拍脑袋定方案，而是拿真实请求去测。比如看 Recall@K、命中率、最终回答准确率，再看失败的情况是“没召回到”，还是“召回到了但 chunk 不完整”。很多时候不是模型不行，而是 chunk 本身组织得不对。

Recall@K：表示正确答案有没有出现在前 K 个召回结果里。

比如用户问一个问题，系统去知识库里召回 top5 个 chunk。如果真正相关的那段内容，出现在这 5 个里面，不管是第 1 个还是第 5 个，这条样本通常就算“召回成功”。那如果 100 条测试问题里，有 80 条都能在 top5 里找到正确 chunk，那Recall@5 = 80%。


所以我自己的原则是：先保证 chunk 稳定可用，再追求 chunk 足够聪明。线上系统里，稳定性和可维护性往往比一开始就追最优更重要。

【困难】Chunk 太大和 Chunk 太小分别会带来什么问题？

如果先说结论：

  • Chunk 太大，问题通常是语义被稀释、噪音变多、检索不够准；
  • Chunk 太小，问题通常是上下文不完整、信息断裂、召回回来也不够用。

这两个方向都会影响最终回答质量，只是坏的方式不一样。

先说 Chunk 太大。

我一般不会把一个 chunk 切得特别长，因为一旦太长，它会把很多不相关的信息揉在一起。Embedding 做向量化的时候，表达出来的就不是某一个明确知识点，而是一坨“主题上相关、细节上发散”的混合语义。这样一来，用户问一个很具体的问题时，召回结果可能看起来相关，但不够精准。

而且 chunk 太大还有几个很现实的问题。

第一，检索精度会下降。因为一个大 chunk 里可能同时有定义、例子、限制条件、扩展说明，用户只问其中一个点，但向量表示已经被整段内容平均掉了，最后容易召回到“大概相关”的内容，而不是“最该命中的那一小段”。

第二，上下文窗口浪费会很明显。召回出来的 chunk 还要塞给大模型。如果每个 chunk 都很大，top3、top5 一塞，prompt 很快就膨胀了。真正有用的信息可能只占其中 20%，剩下都是噪音。这个事在 Demo 阶段不一定明显，但线上 query 一复杂，模型注意力就会被稀释，回答容易绕、容易偏。

第三，rerank 和生成阶段的利用效率会变差。因为后面的重排模型或者大模型，需要从一个很长的 chunk 里再去找重点，成本更高，延迟也更高。说白了，你把“找针”的工作留给了后面链路，前面没有把知识颗粒切干净。

再说 Chunk 太小。

很多人会以为 chunk 越小越精准，其实这也不对。因为检索不是只看“像不像”，还要看“够不够回答”。

如果 chunk 太小，最直接的问题就是上下文断裂。

比如定义在这一块，适用条件在下一块，例外说明在再下一块。单独看每一块都像那么回事，但都不完整。最后即使召回到了，模型拿到的也是半截信息，很容易答得片面。

第一，召回链路会变得更脆弱。因为一个完整知识点被拆成很多碎片之后，你得指望系统刚好把这些碎片一起捞上来，而且顺序还得比较合理。只要漏掉其中一块，答案就可能失真。所以 chunk 太小，表面上颗粒度细了，实际上对 topK、排序、拼接策略要求更高。

第二，召回数量和系统成本可能会上升。因为单块信息承载量太低，为了拼出完整上下文，你可能需要召回更多 chunk。这样不仅向量检索结果更多，rerank 成本更高，prompt 拼装也更复杂，整体延迟和费用都会上来。

第三，噪音不一定真的减少，反而可能增加碎片噪音。因为你会拿到一堆局部相关但彼此孤立的小片段，模型要自己去拼。很多时候它不是没有知识，而是拿到的知识像散装零件，组装不起来。

所以真实项目里，我一般不会拍脑袋去定“500 token”还是“1000 token”，而是会看三件事：

第一，看文档类型。FAQ、问答对、字段说明，本身边界清晰，可以切小一点；流程文档、制度文档、技术手册，通常需要更完整语义，chunk 就不能太碎。

第二，看问题类型。如果用户问的是定义类、事实类问题，小 chunk 往往够用；如果问的是流程类、对比类、原因类问题，就更依赖完整上下文，chunk 太小容易翻车。

第三，看评测结果和 badcase。如果 badcase 主要是“召回很泛、不够准”，我会怀疑 chunk 太大；如果 badcase 主要是“召回到了但信息是半截”，我会怀疑 chunk 太小。然后结合 Recall@K、命中率、最终回答准确率一起看，而不是只盯一个参数。

所以我自己的原则一直是：

Chunk 太大，问题在于不够准；Chunk 太小，问题在于不够全。

RAG 真正要找的，不是一个固定长度，而是一个既能表达完整语义、又不会把无关信息裹进去的“合适知识颗粒度”。

【中等】什么是Overlap？为什么要做 Overlap？Overlap 应该怎么设计？

我的理解是，Overlap 本质上就是相邻 chunk 之间保留一部分重复内容。

我一般不会把它看成一个可有可无的小优化，它其实是在给分块做兜底。因为真实文档不是天然按检索边界写的，你硬切 chunk 的时候，很容易把一句完整的话、一个定义和它的约束条件、或者一个流程的前后步骤切开。这个时候如果完全不重叠，检索到的 chunk 就可能是半截信息。

所以为什么要做 Overlap，我觉得核心就两个字：补裂缝。

  • 第一，它是为了减少上下文断裂。比如这一段最后一句提到“但前提是用户已经完成实名认证”，下一块才开始详细讲实名认证条件。那如果没有 overlap，前一个 chunk 和后一个 chunk 都是不完整的。加一点 overlap 后，至少边界附近的信息不会直接断掉。
  • 第二，它是为了提高召回稳定性。因为用户的问题不一定刚好命中你切块的中间部分，很多 query 命中的其实就是边界区域。如果边界内容没有重叠，相关信息可能刚好被切散；有 overlap 后，边界附近的重要内容会同时存在于两个 chunk 里，召回命中的概率会更高一些。
  • 第三，它还能提升生成阶段的可用性。很多时候不是没召回，而是召回回来的 chunk 差半句话。尤其在制度文档、操作流程、技术说明这种场景里，模型最怕拿到“半个条件”或者“半个结论”。Overlap 的价值，就是尽量让每个 chunk 在边界处也保持基本可理解。

但这个东西也不是越多越好。很多人会觉得 overlap 加大一点更保险，但问题是它会带来重复、冗余和成本上升。

如果 overlap 太大，常见问题有几个：

  • 一个是索引冗余会变重。大量 chunk 内容高度相似，向量库里其实塞了很多重复信息，存储和检索成本都会上去。
  • 第二个是召回结果容易重复。topK 里看起来召回了 5 条，结果 3 条本质上是同一段内容，只是边界略有不同。这样会浪费上下文窗口，也影响后面的 rerank 和生成。
  • 第三个是噪音变多。因为你以为自己是在补信息，实际上可能是在不断重复同一段，最后模型拿到的是很多相似片段，信息密度反而下降。

所以 Overlap 应该怎么设计，我一般不会拍脑袋定一个固定值，而是按这几个原则来做。

  • 第一，Overlap 不是独立参数，要跟 chunk 大小一起看。chunk 本身很小，那 overlap 就不能太大，不然重复比例太高；chunk 本身比较大，那可以适当给一点 overlap，主要保护边界语义。
  • 第二，看文档类型。如果是 FAQ、问答对、字段解释这种天然边界清晰的内容，overlap 可以很小，甚至不需要太多；如果是长段落文档、政策说明、产品手册、流程文档，我一般会保留一定 overlap，因为这类内容跨句、跨段依赖更强。
  • 第三，看 badcase 来调。如果你发现很多问题都是“召回到了，但就差一句条件”或者“定义和限制条件被切开了”，那通常 overlap 偏小；如果你发现 topK 结果高度重复、上下文塞满相似 chunk，那通常 overlap 偏大。
  • 第四，更推荐按语义单位去设计，而不是只按字符数死切。比如不是机械地 overlap 100 个字符，而是尽量保留 1 到 2 句，或者保留一个完整句子边界。因为 overlap 的目的不是重复字数，而是重复有价值的语义。

如果让我给一个工程上的原则，我会这么说：Overlap 的作用不是让 chunk 重复更多，而是让边界信息不丢。所以它应该足够小，小到不会制造太多冗余；但也要足够大，大到能把边界处的关键语义接住。

真实项目里，我一般会先给一个保守值跑评测，然后重点看两类 badcase：

  • 一类是信息断裂型，说明 overlap 不够；
  • 一类是重复召回型，说明 overlap 太多。

最后还是靠标注集和真实 query 去调，而不是靠经验拍一个万能数字。

【中等】什么是召回（Recall）？什么是重排（Rerank）？

如果先给结论：

  • 召回解决的是“先尽可能别漏”；
  • 重排解决的是“再尽可能排准”。

先说召回。

召回就是用户来了一个问题以后，系统先从知识库里捞出一批“可能相关”的候选内容。这里的目标不是一上来就做到最精准，而是先把可能有用的东西尽量找全一点，别把正确答案漏掉。

因为真实线上场景里，用户问法很多样，有时候是口语化表达，有时候是简称、别名、上下文省略。如果你第一步就卡得特别死，只追求特别高的精度，很容易把真正相关的内容直接漏掉。那后面模型就没材料可用了。

所以召回更像一个“粗筛”阶段。常见做法可能有向量召回、关键词召回、混合召回、多路召回这些，本质都是先拿到一批候选 chunk。这个阶段我更关注的是覆盖率，也就是别漏召。

然后说重排。

重排是在召回之后，对这一批候选结果再做一次更细的排序。因为召回阶段拿回来的一堆内容里，通常会有“相关但不够相关”的、“沾边但不是最优”的，还有多个结果都差不多像的情况。这时候就需要一个更精细的模型或者策略，把最该排前面的内容顶上来。

所以重排本质是在解决“召回回来了，但顺序不对”的问题。比如正确 chunk 本来在 top20 里面，但排在第 14 个，那如果你只取前 5 个喂给大模型，它其实等于没用。重排就是尽量把真正最相关的内容提到前面，让有限的上下文窗口用在刀刃上。

如果用一个很直白的比喻：

召回像是先去图书馆把一堆可能相关的书从书架上搬下来；重排是再从这堆书里，按你这个问题的相关性重新排顺序，优先把最该看的那几本放到最前面。

在工程上，这两个阶段的 取舍 也不一样。

  • 召回阶段通常更看重速度和覆盖。因为它面对的是全库搜索，数据量很大，所以一般不会用特别重的模型，不然延迟扛不住。
  • 重排阶段通常更看重精度。因为它只处理召回出来的几十条、几百条候选，范围小很多。

所以我一般会把它理解成一个两阶段检索架构：

  • 第一阶段召回，解决“有没有找到”；
  • 第二阶段重排，解决“找到了之后哪个最该排前面”。

如果线上效果不好，我也不会笼统地说“检索不行”，而是先拆开看：如果正确答案压根不在 topK 候选里，那问题更可能在召回；如果正确答案明明召回到了，但排位很靠后，没进最终 prompt，那问题更可能在重排。这两个阶段定位思路不一样，优化手段也不一样。

所以我自己的理解一直是：

召回决定下限，重排决定上限。召回做不好，正确内容都进不来；重排做不好，正确内容进来了也用不上。

【困难】如何判断 RAG 问题到底出在召回、重排还是生成阶段？

真实项目里，RAG 至少要拆成三段看：召回有没有捞到、重排有没有排对、生成有没有用好。这三个阶段任何一个出问题，最后用户看到的都是“答案不准”，但定位方法完全不一样。

我的做法通常是，先把一条 badcase 拆成一条链路来查。

第一步先看：正确知识有没有出现在召回结果里。

比如我会把这条 query 的 topK 召回结果拉出来，人工或者结合标注集去看，标准答案对应的 chunk 到底在不在里面。

如果topK 里根本没有正确 chunk，那这事基本就不是生成的问题了，优先怀疑召回。这时候常见原因一般有几个：

  • query 本身和文档表述差异太大，Embedding 没对上
  • chunk 切得有问题，关键信息被切碎了
  • 召回路数太单一，只做了向量检索，没补关键词检索
  • 索引参数、topK、query rewrite 这些设置不合适

这个阶段我最关注的是：有没有捞上来。因为没捞上来，后面排得再好、模型再强都没用。

第二步，如果正确 chunk 已经在召回结果里了，我就继续看：它排得靠不靠前。

比如正确 chunk 明明在 top20 里面，但只排在第 17 条，而系统最后只取前 5 条进 prompt，那本质上还是等于没用到。这时候问题更可能在重排，不在召回。

重排有问题时，典型特征就是：

  • 召回结果里“有对的”，但前几条不是最对的
  • 很多“沾边但不关键”的 chunk 被顶到前面
  • 真正该进 prompt 的内容被挤下去了

这种场景我一般会重点查：

  • rerank 模型本身够不够适合当前领域
  • 重排特征是不是太单一
  • 候选集太脏，前面召回带了太多噪音
  • 是否需要做 query 改写或者意图识别，帮助重排更准

所以召回和重排的分界点很简单：召回看有没有，重排看顺序对不对。

第三步，如果正确 chunk 不但召回到了，而且也排进了最终 prompt，这时候答案还是不对，我才会重点怀疑生成阶段。

生成阶段出问题，通常长这样：

  • 检索材料其实是对的，但模型总结错了
  • 材料里明明有约束条件，模型没遵守
  • 多个 chunk 之间有冲突，模型自己瞎补了
  • Prompt 约束不够，模型自由发挥太多
  • 上下文太长，关键信息被淹没了

这个阶段我一般会去看两类东西：

  • 第一类是输入给模型的最终上下文。不是只看召回结果，而是看最终塞进 prompt 的那份内容到底长什么样。很多时候不是“没检索到”，而是“检索到了但拼装错了”，比如顺序乱了、噪音太多、截断了。
  • 第二类是模型输出过程和 Prompt 设计。比如有没有明确要求“只能基于检索内容回答”、有没有要求“找不到就说不知道”、有没有做引用约束、结构化输出、冲突信息优先级这些限制。如果这些约束不够，模型就容易进入自由生成状态。

所以我自己在项目里定位这类问题，一般会按一个很实用的判断顺序来：

先看召回结果，再看重排顺序，最后看生成输入和输出。

更具体一点，我会用三问去判断：

  • 第一问：正确 chunk 在不在召回 topK 里？不在，就是召回问题。
  • 第二问：正确 chunk 有没有进入最终 prompt？在 topK 里但没进 prompt，多半是重排问题，或者候选裁剪策略有问题。
  • 第三问：正确 chunk 已经进 prompt 了，模型为什么还答错？这时候重点查生成，包括 Prompt、上下文拼接、模型遵循性、幻觉控制。

如果再往工程化走一步，我一般不会只靠人工看几条样本，而是会把评测链路拆成分段指标：

  • 召回阶段看 Recall@K
  • 重排阶段看 MRR、NDCG、命中位置变化

MRR全称是Mean Reciprocal Rank，中文你可以理解成：首个正确结果的平均倒数排名。

它特别关注一件事：第一个正确答案排在第几位。

比如：

  • 正确结果排第 1，那得分就是1/1 = 1
  • 排第 2，那得分就是1/2 = 0.5
  • 排第 5，那得分就是1/5 = 0.2

然后把很多测试样本的这个值取平均，就是 MRR。

所以它有个很明显的特点：

  • 越早出现第一个正确结果，MRR 越高
  • 它特别适合看“用户最先看到的结果对不对”

这个指标很适合那种场景：用户通常只看前 1 条、前 3 条，你特别在意“第一个命中的结果是不是对的”。

你可以这么记：

MRR 主要看“第一个对的，排得有多靠前”。

-----------------------------------------------------

NDCG全称是Normalized Discounted Cumulative Gain，中文一般叫：归一化折损累计增益。

这个名字听着唬人，但你不用死背定义，你就记住它主要干的事：

它不只看有没有正确结果，还看多个结果的相关性高低，以及它们排的位置合不合理。

它比 MRR 更细一点，因为它不是只盯着“第一个正确结果”，而是会看整个前 K 个排序质量。

它有两个核心思想：

第一，排得越靠前，价值越大

同样一个高相关结果：

  • 排第 1 很值钱
  • 排第 5 也有价值，但价值会打折
  • 排得越靠后，分数折损越明显

所以它会对靠后的位置做“折损”。

第二，不是所有相关结果都一样重要

有些结果是：

  • 完全正确，强相关
  • 部分相关
  • 只是沾边

NDCG 能处理这种“相关性有等级”的情况。比如你可以给文档打标签：

  • 3 分：非常相关
  • 2 分：比较相关
  • 1 分：略相关
  • 0 分：不相关

然后它会看：你是不是把高分结果排在前面了。

最后再跟“理想排序”做一个归一化，所以分数通常在0 到 1之间，越接近 1 越好。

你可以这么记：

NDCG 主要看“整个排序列表排得合不合理，高相关内容有没有尽量排前面”。

  • 生成阶段看最终答案准确率、引用一致性、人工评分

这样好处是，你不会把所有问题都甩给“大模型不行”。很多团队一看到答案差，就直接改 Prompt 或换模型，最后越改越乱。但真实做下来，RAG 问题很大一部分其实出在前面，尤其是 chunk 组织、召回策略和重排质量。

如果让我用一句更像面试现场的话收口，我会这么说：

我判断 RAG 问题，不会直接看最终答案，而是按链路反推。先看正确知识有没有被召回，再看有没有被排到前面并进入 prompt，最后才看模型有没有正确利用这些上下文。因为召回解决的是“找没找到”，重排解决的是“排得对不对”，生成解决的是“用得好不好”。这三层分开看，定位才会准。

【困难】如果用户提问非常模糊，RAG 应该怎么处理？

我的理解是，用户提问很模糊的时候，RAG 不能直接“硬检索 + 硬生成”，不然很容易出现两种问题：

  • 一种是召回一堆沾边但不准的内容
  • 另一种是模型自己脑补，把一个本来没说清的问题答得很自信。

所以我一般会把这种场景先当成一个查询理解问题，而不是直接当成普通检索问题。

我自己的处理思路通常是先分层判断这个“模糊”到底是哪种模糊。

  • 有的模糊是指代不清，比如用户说“这个怎么改”“那个规则是啥”，这里其实缺主语；
  • 有的模糊是范围太大，比如“介绍一下报销”“这个产品怎么用”，问题太宽；
  • 还有一种是表达太短，比如就丢两个词，“续费失败”“登录异常”，这种信息量不够。

这几种模糊，处理方式其实不一样。

真实项目里，我一般不会上来就让模型直接答，而是先做一个轻量的意图识别和歧义检测。如果系统判断这个 query 信息不足、检索置信度低，第一选择通常不是强答，而是先做澄清。比如让系统追问一句：“你想了解的是报销流程、报销条件，还是报销失败原因？”因为很多时候，多问这一句，比后面堆多复杂的 rerank 都更有效。

但问题是，线上又不能什么都追问，不然交互会很笨。所以我一般会做一个分流：

如果是高风险、高成本场景，比如客服政策、金融、医疗、合规类问题，我更倾向于先澄清，再检索。因为答错代价太高。如果是低风险、泛知识类场景，我会先尝试“保守检索 + 显式说明范围”。比如用户问得很泛，我不会装作已经理解得很准，而是会先给一个范围化回答，比如“这个问题可能涉及 A、B、C 三部分，我先说最常见的一种”，这样能减少幻觉和误答。

从检索链路上讲，模糊 query 一般要做几件事。

第一是利用上下文补全 query。

很多模糊提问，其实放回会话历史里并不模糊。比如上一轮在聊退款规则，下一轮问“那多久到账”，这时候不能拿“多久到账”单独去检索，而是要把历史主题拼进去，改写成“退款多久到账”。所以我一般会先做 query rewrite，把省略信息补全之后再查知识库。

第二是做多路召回，而不是单路硬查。

因为模糊 query 的表达空间很大，单一路径很容易漏。我会考虑用 query expansion 或 multi-query，把一个模糊问题拆成几个可能的明确问法，再分别检索，然后合并结果做重排。比如“会员怎么开不了”可以扩成“会员购买失败”“会员无法激活”“会员权限未生效”这几种子查询。这样做的目的不是扩大噪音，而是提高在不确定表达下的覆盖率。

第三是看检索置信度，不够就别硬答。

这点我觉得很关键。很多系统最大的问题不是“答不上来”，而是“明明没找到，还硬给一个像样的答案”。所以我会给 RAG 加一层低置信度兜底，比如：召回结果相似度整体偏低、topK 结果彼此分散、rerank 分数不集中，这种情况下我会认为系统其实没理解清楚。那这时候更合理的做法是：要么追问澄清，要么明确告诉用户“当前信息不足，需要你补充一下具体场景”。

第四是生成阶段要收紧约束。

模糊 query 最大的坑，就是前面检索已经不确定了，后面生成还特别自由，那最后就很容易一本正经地胡说。所以这种场景下，我一般会在 Prompt 里明确约束：只能基于检索结果回答；如果信息不足，要先说明不确定点；不要擅自补全用户未提供的前提。我的经验是，模糊 query 场景下，生成阶段的“克制”比“聪明”更重要。

如果从工程角度再往前走一步，我会把它做成一个完整策略，而不是单点优化。比如一条模糊 query 进来，先做 query 分类：明确 / 轻度模糊 / 严重模糊。明确的直接进正常 RAG；轻度模糊的先做 query rewrite + 多路召回 + 重排；严重模糊的直接触发澄清或兜底话术。这样系统会稳定很多，不会把所有模糊问题都压给大模型自由发挥。

我自己会特别关注一个误区：> 很多人觉得“用户问题模糊，那就多召回一点、多塞一点上下文给模型自己理解”。但我实际做下来，这通常不是最优解。因为问题一模糊，召回范围一放大，噪音会急剧上升，最后模型更容易被干扰。所以这类场景真正重要的，不是单纯扩大检索，而是先把问题变清楚，再做检索。

所以如果面试官问我，RAG 怎么处理模糊提问，我会给一个很明确的结论：不要急着答，先判断它到底模糊在哪；能补全就补全，能澄清就澄清，实在不确定就降级，而不是让模型带着不确定输入去强行生成一个看起来很完整的答案。

【中等】你会如何设计一个支持 PDF、Word、Excel、网页的统一知识入库流程？

我的理解是，这个题我不会按“支持四种文件格式”去答，我会把它看成一个异构数据统一入库的问题。

真正难的点不在上传，而在于：不同格式怎么被稳定地解析成统一的知识表达，再进入后面的切分、索引和检索链路。

如果是我来设计，我一般不会让 PDF、Word、Excel、网页直接各走各的野路子，最后分别产出一堆不一致的 chunk。那样短期能跑，后面维护、调优、排错都会很痛。

我通常会先做一层统一中间表示层，也可以理解成 Canonical Document Schema。

也就是说，不管前面是 PDF、Word、Excel 还是网页，最后都先转成统一结构，比如：

  • 文档基础信息：标题、来源、作者、更新时间、语言、权限
  • 文档结构信息：章节、段落、列表、表格、图片、链接
  • 文本块信息：块类型、块内容、上下级关系、页码/sheet/DOM 路径
  • 附加元信息：来源文件 ID、版本号、租户 ID、标签、业务域

我觉得这一步特别关键，因为后面 chunking、embedding、召回、引用展示，最好都只面向这一套统一结构做，而不是每种格式各写一套逻辑。

如果往下拆，我一般会把整个入库流程设计成 7 个阶段。

第一层是接入层。支持用户上传文件或者提交 URL，先做文件类型识别、大小校验、去重、权限校验，然后把原始文件放到对象存储里，比如 OSS、S3 这种，数据库里只存任务记录和元信息。这里我一般会做成异步任务，不会让用户在上传接口里同步等完整解析，不然大文件和 OCR 场景很容易超时。

第二层是解析适配层。这里我会按格式做 adapter，但输出统一结构。

  • PDF：先区分是文本型 PDF 还是扫描型 PDF。文本型直接提取文本和版面结构；扫描型要走 OCR。表格型 PDF 我会单独识别，不然很容易把列关系打散。
  • Word：重点保留标题层级、段落、列表、表格、图片说明这些结构，不然文档语义会塌掉。
  • Excel：我不会简单把整个 sheet 拼成一大段文本，那样检索效果很差。通常会按 sheet、表头、行记录去结构化，必要时把每一行转成半结构化记录。
  • 网页：重点是正文抽取、去导航栏、去广告、去页脚、去重复 DOM，然后保留标题层级、链接、发布时间这些元信息。

我一般会把这层理解成：格式解析不是为了拿到纯文本，而是为了尽量保留知识结构。

第三层是清洗和标准化层。这个阶段主要做：

  • 去噪，比如页眉页脚、水印、重复导航、版权信息
  • 编码统一、换行修复、空白字符清理
  • 标题归一、表格规范化、图片 alt 或说明抽取
  • 文档语言检测、时间格式统一
  • 重复内容检测

因为真实项目里，脏数据往往比解析本身更影响效果。很多 badcase 不是模型不行，是你把一堆页眉页脚和正文一起向量化了。

第四层是结构化切分层。这里我一般不会直接固定长度硬切。我更倾向于“先按结构切，再按长度兜底”。

比如：

  • PDF/Word/网页：优先按标题、段落、列表、表格块去切
  • Excel：优先按 sheet、表头语义、记录粒度去切
  • 超长块再做二次切分，并加 overlap
  • 表格内容可以转成“字段名: 值”的可检索文本，必要时保留原表结构 JSON

这里的核心不是把文本切开，而是切出可检索、可引用、可回答的知识单元。

第五层是索引构建层。我一般不会只建向量索引，而是会做混合索引：

  • 稠密向量索引：做语义召回
  • 稀疏索引 / 关键词索引：做精确匹配
  • 元数据索引：按来源、时间、业务域、权限过滤

同时每个 chunk 都会带完整 metadata，比如：

  • source_type：pdf / docx / xlsx / web
  • source_id / document_id / tenant_id
  • page_no / section / sheet_name / url
  • version / updated_at / access_scope

因为你后面不仅要检索，还要做权限控制、引用回溯、增量更新和删除，这些都靠 metadata。

第六层是质量校验和版本管理层。这个阶段我会重点做两件事。

第一件事是校验。比如：

  • 解析成功率
  • OCR 置信度
  • chunk 数量异常告警
  • 空文本率、重复率
  • embedding 成功率
  • 样本抽检可读性

第二件事是版本管理。我不会每次全量重建，通常会给文档做 content hash，判断是新增、更新还是无变化。更新场景下尽量做增量重建，只替换变更 chunk，旧版本保留可回滚能力。因为知识库一大，全量重建成本很高，而且会影响线上稳定性。

第七层是任务编排和可观测性。真实项目里，这种入库链路一定是异步的，所以我会加消息队列或者任务队列，把“上传、解析、清洗、切分、向量化、入库”拆成可重试的阶段。每一步都要有状态流转，比如：uploaded -> parsing -> normalized -> chunked -> embedded -> indexed -> done / failed。失败要支持重试和局部补偿，而不是整条链路从头跑。

同时我会加监控：

  • 各格式解析成功率
  • 平均入库时长
  • OCR 耗时和失败率
  • 单文档 chunk 数量分布
  • 入库后召回命中率
  • 各来源 badcase 分布

因为这个系统最后不是“能不能传进去”，而是“传进去之后能不能被稳定检索出来”。

如果面试官再往下追问，我会补一句：

我设计这套统一入库流程，核心思路不是“一个 parser 打天下”，而是前面按格式适配，后面按统一结构治理。这样前端是多入口，后端是单治理模型，后续不管你再接 PPT、Markdown、Notion、Confluence，本质上都只是多写一个 adapter，不需要把后面的 chunking、embedding、索引链路全重做一遍。


【中等】什么是 Metadata？为什么 RAG 场景下它很重要？

Metadata 就是文档内容之外的描述信息。比如这段文本来自哪个文件、哪个业务模块、哪一页、创建时间、更新时间、作者、文档类型、标签、权限范围，甚至它属于哪个客户、哪个产品线。这些信息本身不一定参与向量语义匹配，但会直接影响检索结果是否可控、可信、可追溯。

如果只做最简单的 RAG，可能就是把文档切块、向量化、存到向量库，然后用户提问时按相似度召回。

但真实项目里这样很容易出问题。比如用户问“退款规则是什么”，如果知识库里同时有旧版规则、新版规则、国内规则、海外规则，那光靠向量相似度，模型可能召回一条语义很像但已经过期的内容。这个时候 Metadata 就很重要，可以通过更新时间、地区、业务线、文档状态先做过滤，避免把错误知识喂给模型。

我一般会把 Metadata 用在几个关键地方。

  • 一个是检索前过滤，比如只查某个租户、某个产品、某个语言、某个权限范围内的知识。这个在 SaaS 或企业知识库里尤其重要，因为不能让 A 客户的问题召回 B 客户的资料，也不能让普通员工查到高权限文档。
  • 另一个是排序和增强召回。比如同样相似度的两段内容，我会更优先选择更新时间更近、权威等级更高、命中文档标题更相关的内容，而不是只看向量分数。因为 RAG 最后不是为了“找相似文本”，而是为了“找能回答问题的可信依据”。
  • 还有一个是答案溯源和排查问题。线上如果用户反馈回答错了，我需要知道模型到底引用了哪几个 chunk，这些 chunk 来自哪个文件、哪一页、哪个版本。如果没有 Metadata，排查就会非常痛苦，只能猜是召回错了、切分错了，还是生成阶段编错了。

当然 Metadata 也不是越多越好。字段太乱、标签不统一、维护成本高，反而会让过滤条件失效。

所以我一般会先设计一套稳定的核心字段，比如 source、docId、chunkId、tenantId、permission、category、updatedAt、version、pageNumber。后面再根据业务需要补充标签，而不是一开始就把所有信息都塞进去。

所以我对 Metadata 的判断是：它不是单纯的文档备注，而是 RAG 系统做权限控制、精准召回、结果溯源、版本治理和线上可维护性的基础。

如果没有 Metadata，RAG 很容易停留在 Demo；有了合理的 Metadata，才更接近一个可上线、可治理的知识系统。

【困难】知识库内容存在时效性时，你会如何做增量更新？

如果知识库内容有时效性，我一般不会简单地“新文档来了就重新全量向量化”。真实项目里，知识库一大，全量重建成本会很高，而且重建期间还可能影响线上检索稳定性。我更倾向于做一套增量更新机制，把“变更检测、重新切分、重新向量化、版本控制、灰度生效”串起来。

我的思路是，先要识别哪些内容变了。比如文档有新增、修改、删除，或者只是标题、标签、权限这些 Metadata 变了，这几类处理方式是不一样的。新增内容比较简单，解析、切分、生成 Embedding，然后写入向量库就可以。修改内容要更谨慎，我不会只更新原文，因为 chunk 可能会因为内容变化导致边界变化，所以通常会以文档版本为单位重新切分这份文档，再把旧 chunk 标记为失效，新 chunk 写入新版本。删除内容也不会物理删除为主，而是先做软删除，打上 deleted 或 inactive 状态，避免误删后无法回滚。

这里最关键的是版本控制。比如同一份退款规则，旧版和新版语义可能很接近，如果不控制版本，检索时很容易召回旧内容，模型就会给出过期答案。所以我会在 Metadata 里维护 docId、chunkId、version、updatedAt、effectiveTime、status 这些字段。检索时先根据状态和生效时间过滤，只让当前有效版本参与召回。对于有明确生效时间的文档，比如“5 月 1 日起执行新规则”，就不能简单按更新时间判断，而要按 effectiveTime 判断。

真正上线时，我还会避免直接覆盖线上索引。比较稳的做法是双写或者影子索引：新内容先进入待发布版本，跑一轮离线评测，看核心问题的 Recall@K、命中位置、答案准确率有没有下降；如果没有问题，再把检索入口切到新版本。这样出了问题也可以快速回滚到旧索引。

还有一个很容易被忽略的点是缓存和生成结果。

如果 RAG 上层做了 query cache 或答案缓存，知识库更新后必须让相关缓存失效，否则底层知识已经更新了，用户看到的还是旧答案。所以我通常会根据 docId、version 或知识库更新时间戳做缓存失效策略。

当然，增量更新也有代价。它比全量更新复杂，需要维护文档指纹、版本、状态、索引一致性和回滚能力。如果知识库规模很小，或者更新频率很低，全量重建可能反而更简单。但只要是企业知识库、客服知识库、政策类知识库这种时效性强的场景，我会优先做增量更新，因为它更可控，也更适合线上长期维护。

我的原则是：内容变化不能只看“有没有重新生成 Embedding”，而要保证线上检索永远只命中当前有效、权限正确、可追溯、可回滚的知识。

【困难】如果知识库里有互相冲突的信息，你会怎么处理？

因为在 RAG 里，模型不是权威来源，知识库才是权威来源。一旦把两段冲突内容都塞给模型，模型很可能会综合出一个看起来合理、但实际上不可靠的答案。

我会先把这个问题拆成两层：

  • 一层是入库阶段尽量发现冲突
  • 另一层是检索和生成阶段要有兜底

在入库阶段，我会重点做版本、来源和优先级管理。

比如同样是“退款规则”，可能一个来自旧版 FAQ，一个来自最新政策文档，一个来自客服手册。

这个时候不能只看文本相似度，而要看 Metadata（上面有解释），比如文档更新时间、生效时间、版本号、业务线、适用地区、来源权威等级。

通常我会把官方制度、最新生效版本、人工审核过的文档优先级设得更高，旧版本不一定马上删掉，但要标记为 inactive 或 historical，默认不参与线上检索。

如果是新增文档和已有知识发生冲突，不要直接覆盖。比较稳的做法是先进入待审核状态，通过规则或者模型辅助识别相似主题下的矛盾点，比如金额、时间、条件、流程不一致，然后推给运营或业务负责人确认。因为很多冲突不是技术能判断的，比如“7 天可退”和“15 天可退”，系统只能发现它们冲突，但谁是对的，必须由业务规则来定。

在检索阶段，我会尽量避免把冲突内容同时召回给模型。

检索前先根据租户、权限、版本、生效时间、状态做过滤；召回后再根据权威等级、更新时间、文档类型做重排。如果还是发现 topK 里存在明显冲突，比如两个高相关 chunk 给出了不同结论，我不会让模型强行回答，而是让它走保守策略：提示“知识库中存在不一致信息”，并展示来源，或者转人工确认。

在生成阶段，Prompt 里也要明确约束模型：只能基于最高优先级、当前生效的资料回答；如果资料冲突或证据不足，不要编结论。这个对客服、金融、法务、政策类场景尤其重要，因为答错的成本很高。

线上还要有反馈闭环。用户点了“答案不准确”，或者人工客服纠正了一次，这个 badcase 要回流到知识库治理流程里，更新冲突记录、修正文档状态，必要时重新生成 Embedding 和索引。否则同样的问题下次还会错。

所以我的处理原则是：冲突信息不能只靠 Prompt 解决，核心要靠知识治理。

入库时识别冲突，存储时管理版本和权威性，检索时过滤和排序，生成时保守回答，线上再通过反馈持续修正。

这样 RAG 才不会从“知识增强”变成“冲突增强”。

【中等】什么是 Query Rewrite？为什么在检索前要改写用户问题？

真实项目里，用户的问题经常不是标准检索语句。比如用户会问“这个怎么弄”“上次那个规则还生效吗”“我想退一下”，这种表达对人来说可能能理解，但对向量检索或者关键词检索来说信息太少，很容易召回不到真正相关的知识。

所以我一般会在检索前做一层问题改写，把用户的口语表达、上下文指代、模糊意图，转换成更清晰、更完整、更适合召回的查询语句。这就是Query Rewrite。

比如客服场景里，用户先说“我买了会员”，下一轮问“能退吗”。如果直接拿“能退吗”去检索，召回质量肯定不稳定。更合理的是结合上下文改写成“会员购买后是否支持退款，退款规则是什么”。这样检索系统才能更准确地命中会员退款政策，而不是召回一堆泛泛的退款内容。

我一般会把 Query Rewrite 用在几个场景。一个是补全上下文，解决“它、这个、刚才那个”这种指代问题；一个是意图标准化，把用户口语转成业务里的标准术语；还有一个是扩展检索表达，比如用户说“打不开”，可能要补成“登录失败、页面无法访问、系统异常”等候选表达，提升召回率。

但这里有个风险，改写不是越激进越好。如果模型把用户原意改偏了，后面 RAG 整条链路都会错。比如用户问“能不能取消订单”，结果改写成“如何申请退款”，这两个在业务上就不完全一样。所以我会要求改写尽量保守，最好保留原始 query，同时把改写 query 作为增强检索，而不是完全替换原问题。

工程上我更倾向于做多路检索：原始问题走一路，改写后的问题走一路，必要时再加关键词检索。最后把多路召回结果合并、去重、重排。这样既能利用改写提升召回，也能避免改写失败导致结果全偏。

评估的时候也不能只看改写文本顺不顺，而是要看检索结果有没有变好。比如 Recall@K 有没有提升，正确 chunk 的命中位置有没有提前，生成答案的准确率有没有提升，同时延迟和 token 成本能不能接受。如果收益不明显，我不会在主链路里加很重的改写模型。

所以我对 Query Rewrite 的理解是：它是 RAG 检索前的一层“查询对齐”，把用户自然表达对齐到知识库的表达方式。但它必须可控、可评估、可回退，不能让模型自由发挥。

【中等】什么是引用溯源（Citation）？为什么在企业场景里很重要？

在企业场景里，Citation 更核心的作用是让 AI 的回答可验证、可追责、可审计。

因为企业内部的 AI 系统，很多回答不是闲聊，而是会影响业务决策的。比如：客服回答用户退款政策、法务查询合同条款、销售查询产品报价、员工查询公司制度，这些场景里，AI 不能只说“我认为是这样”，它必须告诉用户：这个结论来自哪份文档、哪一段内容、哪个版本。

我一般会把 Citation 放在 RAG 链路里一起设计，而不是最后临时补。

文档入库时，就要保留元信息，比如文档 ID、标题、来源系统、版本号、更新时间、段落位置、页码、权限范围。

检索出来的 chunk 进入生成阶段时，也要带着这些元信息一起传给模型。

最后生成答案时，要求模型把关键结论和对应来源绑定起来，而不是统一在末尾扔一堆参考资料。

这里面有个很重要的取舍：Citation 不是越多越好。引用太多，用户反而不知道该看哪里；引用太少，又无法支撑关键结论。所以我更倾向于只给关键结论加引用，尤其是涉及金额、政策、流程、权限、合规这类内容，一定要能追到原文。

真正落地时，我还会做一层校验。比如答案里出现了某个结论，但引用文档里根本没有对应依据，这就属于“伪引用”，风险比没有引用还大。

因为它会让用户误以为这个答案有依据。

所以企业级系统里，Citation 不能只是展示层能力，还要结合检索结果、权限控制、版本管理和答案校验一起做。

它的重要性主要在三个方面。

  • 第一是降低幻觉风险，用户可以回到原文验证；
  • 第二是满足企业审计和合规要求，出了问题能定位是哪份知识导致的；
  • 第三是提升用户信任，尤其在内部知识库、客服、金融、医疗、法务这些场景里，没有出处的答案很难被真正采用。

所以我的理解是，Citation 的本质不是“引用格式”，而是企业 AI 系统里的可信证据链。

它解决的是：AI 为什么这么答、依据是什么、出了问题能不能回溯。

【困难】如何设计一个“答不上来就明确拒答”的 RAG 系统？

在真实线上系统里，模型很容易嘴硬，它可能明明没有依据，也会根据常识补一个看起来很合理的答案。所以这个能力一定要从检索、重排、生成和校验几层一起做。

我的思路是先定义什么叫“答不上来”。

不是模型觉得不会，而是系统判断当前知识库没有足够证据支撑答案。

比如召回结果为空、召回分数太低、TopK 文档和问题不相关、多个来源互相冲突，或者生成答案里的关键结论找不到引用来源，这些都应该进入拒答逻辑。

在检索阶段，我会设置一个最低相关性阈值。

比如：向量召回后，不是直接把 TopK 全塞给模型，而是先看相似度分数、关键词命中、元数据过滤和重排分数。

如果最高分都很低，或者重排后没有任何文档能覆盖用户问题，那就不要进入正常回答链路，而是直接返回类似“当前知识库中没有找到可靠依据”的结果。

但这里不能只看一个分数，因为向量相似度不是绝对可信的。

有些问题相似度高，但文档只是语义接近，并不包含答案。

所以我通常会再做一层证据判断，让模型或一个轻量分类器判断：这些检索片段能不能直接回答用户问题。它输出的不是答案，而是answerable / unanswerable / conflict这类结构化结果，并且要求给出对应依据。

到了生成阶段，我会强约束模型：只能基于给定上下文回答，不能使用外部常识补全；如果上下文不足，就明确说不知道。

同时要求答案里的关键结论必须绑定引用来源。这样做的好处是，后面可以校验答案是否真的被检索内容支撑，而不是模型自己编出来的。

真正上线我还会加一个后置校验。比如：模型生成完之后，再检查答案中的关键事实是否能在引用 chunk 里找到依据；如果找不到，就降级成拒答，或者转人工。尤其是客服、金融、法务、医疗这种场景，我宁愿拒答，也不能让模型编一个看似专业但不可验证的答案。

这里面有个取舍：拒答阈值太高，会导致系统保守，很多本来能回答的问题也被拒掉；阈值太低，又容易幻觉。所以我不会拍脑袋定阈值，而是拿真实问题集做评测，看两个指标：一个是该拒答时有没有拒，也就是幻觉率；另一个是本来能答却被拒的比例，也就是误拒率。上线后还要持续看用户追问、人工接管率、投诉率和 badcase。

所以我的原则是：拒答不是一句“不会就说不会”的 Prompt，而是一套证据门控机制。只有当检索证据足够、来源可靠、内容不冲突、答案能被引用支撑时，系统才回答；否则就明确拒答，并告诉用户当前知识库没有可靠依据。
